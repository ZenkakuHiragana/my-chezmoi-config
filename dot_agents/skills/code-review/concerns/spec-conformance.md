# 仕様照合レビュー

実装が固定済みの仕様、`Requirement contract`、設計書、公開 API 契約、互換契約と一致するか確認する。

## Triggers

- レビュー対象の正否を `Requirement contract`、設計仕様、仕様書、公開 API 文書、標準、互換契約、生成仕様のいずれかで判定する。
- 仕様が identifier、parameter、設定項目、プロトコル項目、ファイル形式、単位、値域、戻り値、制約、副作用、順序、割当を定義する。
- 実装が外部 tool、実行環境、公開 API、生成物、永続形式へ接続する。
- 仕様未確認のまま、局所実装や compile / test 成功だけで正しさを判断している。

## Checks

- 照合に使う仕様を明示する。仕様が複数ある場合は、どの仕様がどの実装面を決めるか分ける。
- 仕様が定義する各項目を、実装、test、docs、生成物の対応箇所へ対応づける。
- 仕様が表形式でなくても、要求、禁止、値域、単位、順序、戻り値、制約、副作用を項目として切り出して照合する。
- identifier、parameter、設定項目、プロトコル項目、ファイル形式内の項目などの名前や割当は、仕様側の定義と実装側の出現を突き合わせる。
- 仕様に無い割当、仕様と食い違う割当、二重割当、未定義 identifier、値域外、単位違い、順序違い、戻り値または副作用の取り違えを指摘する。
- 仕様が未確認で判断できない項目は `Uncertain` とし、`investigation` または `public-research` で確認すべき根拠を示す。

## Severity

- 明示仕様、公開契約、`Requirement contract`、互換契約に反する実装は `Blocker` とする。
- 仕様との対応が欠け、取り込み後に契約違反を見逃す可能性が高い場合は `Major` とする。
- 仕様根拠が足りず危険度を確定できない場合は `Uncertain` とする。

## Output

- 照合した仕様
- 仕様項目と実装箇所の対応
- 未照合の仕様項目
- 仕様違反または未確認事項
