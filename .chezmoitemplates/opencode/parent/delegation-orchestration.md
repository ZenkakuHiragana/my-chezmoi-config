## サブエージェントへの委譲規則

## 1. 委譲してはならない条件

- いちばん大きい不確実性がユーザー意図であるときは委譲してはならない。
- 順番依存が強く、状態の持ち主を 1 人に保つ必要があるときは委譲してはならない。
- 親エージェントが検証のために大半をやり直す必要があるときは委譲してはならない。
- 同じ file、shared API、shared config、schema、prompt hierarchy、lockfile、global rule を複数のサブエージェントが触るときは委譲してはならない。
- サブエージェントに渡す文脈伝達の方がタスク本体より重いときは委譲してはならない。

## 2. 委譲の判定

サブエージェントへ渡す前に、次をすべて満たすことを確認する。

- 目的、対象、入力、制約、読むべき文脈、必要な根拠、出力形式、停止条件 をすべて明示できる。
  - 詳細は `assignment packet` の節に従う
- 親エージェントが結果を検証または突き合わせできる。
- サブエージェントが、未解決の親判断や他タスクの未完了結果を必要としない。
- ずれや手戻りのコストを小さく抑えられる。

- 独立した作業単位が 2 つ以上あり、1 節の禁止条件がどれも当てはまらないなら、並列に委譲する。
- 直列にするなら、1 節のどの禁止条件が当てはまるかを明記する。
- 既存 task file が `write_set` 非重複の `Parallel group` を定義しているなら、blocker が無い限り、その group を並列起動する。
- `Parallel group` が無い task file でも、作業単位、`write_set`、依存関係から安全に分割できるなら、並列委譲を組む。

## 3. 委譲計画

委譲を実行する前に、短い委譲計画を出力する。

- 独立単位: 分割した作業単位を書く。
- 並列群: 同時に起動する群を `Parallel group` として書く。
- `Parallel basis`: domain / surface / review / none のどれで分割したかを書く。
- `write_set` 非重複: 競合しないことを確認する。
- 直列化: 並列にしない作業は、1 節のどの禁止条件に該当したかを書く。

## 4. 委譲の仕方

- `assignment packet` の各項目を具体的に書き、サブエージェントがその作業票 **のみ** を参照して委譲したタスクを遂行できるように指示する。
- 並列に複数のタスクを委譲する時は、エージェント間で書き込むファイルが競合してはならない。

## 5. 結果を受け取った後にやること

- 結果の統合、競合する主張の解決、検証は親エージェントが担当する。
- サブエージェントの結果は、`read_set`、`evidence_required`、指定した検証条件を満たすか確認してから取り込む。
- 委譲したこと自体は、分割がユーザーに実際に影響するときに説明する。

## 6. サブエージェントの選び方

- `general-fast`: 範囲が限定され、根拠集め中心で、read-only、1 skill で済み、止めどころが明確なとき
- `general-strong`: 調査範囲が広い、仮説が競合する、設計比較が要る、または複数結果の突き合わせが要るとき

## 7. review の fan-out

review で `work_class` が `broad-or-unclear` のとき、観点別に fan-out する。

- 発動条件: review 対象の `work_class` が `broad-or-unclear`。
- 分割単位: review skill が提供する観点（concern / profile）ごと。
- 各サブエージェントは割り当てられた観点のみを検査し、他の観点へ踏み込まない。
- 各サブエージェントの `mode_constraint` は `read_only` とする。
- 結果の統合は親が担当する。各サブエージェントの結果を事前に相互に見せない。
- 1節の禁止条件（同じ file を複数サブエージェントが書き込む等）に該当する場合は fan-out しない。ただし review は読み込みのみであり、同じ成果物を複数観点で読み比べる操作は禁止条件に該当しない。
- fan-out しない場合は、その理由を明示する。

## 8. review と fan-out の接続

`review-orchestration` が起動した場合、7節の fan-out 規則は `review-orchestration` の固定済み検査集合に対してだけ実行する。fan-out の発動条件と手順は 7節に従う。`review-orchestration` の発動条件と運営手順は、AGENTS.md の「文章と成果物」と `review-orchestration` skill の正本に従う。

## 9. 終了判定の外部化

`review-orchestration` が `ready_for_exit_check` を返した場合、親は次の条件を全て満たす証拠を集めてから review loop を終了する。親は条件を満たす証拠を集める主体であって、条件を緩める主体ではない。

- 割り当てた全 review unit が完了している。
- 新規指摘の受付を閉鎖している。
- 全候補を裁定済みである。
- 未裁定の指摘（`needs-investigation`）が 0 件である。
- `accepted` の全件に修正前の失敗と修正後の成功を記録している。
- task contract の tests が成功している。
- 全ての変更が受理済み指摘または契約条項へ追跡可能である。
- 判定不能領域と再開条件を記録している。

親が `rejected`、`out-of-scope`、契約解釈の変更、または `判定不能` の受容を判断した場合は、その判断を fresh な別実行者の独立確認へ渡す。親の自己承認だけでこれらを確定してはならない。
