# Derive Fixture Variants

観測済みの失敗事例またはセッション採掘結果から、頑健性確認専用の派生固定試験事例草稿を作る。
草稿は観測事例を増やさない。事例族の適用域を確認する候補として扱う。
この agent は、調査結果と草稿をローカル報告へ保存する。生成した fixture の実行者は `read_only` とし、保存を含む副作用を行わせない。

## 失敗ログの保存先

失敗ログの保存先は次の順序で決める。

1. 現在位置が `chezmoi source-path` 配下なら、`$(chezmoi source-path)/.opencode/local-failure-logs/`
2. それ以外で `chezmoi source-path` があれば、`$(chezmoi source-path)/.opencode/local-failure-logs/`
3. それ以外は `~/.local/share/chezmoi/.opencode/local-failure-logs/`

## 入力

- 失敗記録またはセッション採掘報告の指定
- 対象とする事例族、失敗記録、または規則（任意）。規則を指定する場合、対応する観測済み失敗記録を特定して引用できなければならない。

対象指定がない場合、上の失敗ログの保存先にある `session-mining/` と直下の失敗記録を読む。観測事実への引用、最小の決定点、観測可能な成果物判定、`read_only` への縮約方法を示せる記録だけを適格候補として一覧化する。この実行では草稿を作らず、候補一覧だけをローカル報告へ保存し、ユーザーが次回指定できる失敗記録または事例族を返す。適格候補がない場合、その理由を報告する。

## 実行経路のゲート

草稿化の前に `target_selection_status` と `execution_route` を確定し、報告へ記録する。

- `target_selection_status: present`: ユーザーが失敗記録の path、事例族の識別子、または規則と対応する観測済み失敗記録を指定した場合。
- `target_selection_status: absent`: 「候補を探す」「派生できる事例を調べる」などの探索依頼、または規則だけが指定され、対応する観測済み失敗記録が指定されていない場合。
- `target_selection_status: present` なら `execution_route: derive` とする。
- `target_selection_status: absent` なら `execution_route: survey_only` とする。`survey_only` では派生固定試験事例草稿、実行者向け本文、採点者向け判定基準、捏造検査を作ってはならない。草稿数は0件として報告する。

対象を推測して `survey_only` から `derive` へ変更してはならない。

## 事例族の条件

同じ事例族にまとめるには、次をすべて満たす。

- 決定点が同じである。正しい行動の分岐と、避ける失敗が同じである。
- 根拠の役割関係が同じである。正本、矛盾、消費者経路、確認対象の関係を保つ。
- 成果物の判定条件が同じである。pass / fail を分ける観測可能な性質を保つ。
- 元事例の配送条件を特定できる。assignment packet、作業モード、資料の提示順を記録し、派生時の安全な縮約との差分を説明できる。

いずれかを保てない候補は別の事例族として扱う。元事例が1件だけでも草稿候補にしてよいが、独立した観測の反復とは報告してはならない。

## 派生で変えるもの

初回に許可する操作は、表層の変更と別ドメインへの移植だけとする。

- 題材、識別子、依頼文の表現、言語、ファイル名
- 同じ根拠の役割関係を持つ別ドメインの素材

次を変えてはならない。

- 決定点、期待する成果物の性質、採点条件
- 正本、矛盾、消費者経路の役割関係
- assignment packet の判断材料と資料の提示順
- 元事例にない失敗機序

別ドメインへ移すときは、元事例の各根拠要素と対応する要素を表で示す。対応を示せない場合、草稿を作ってはならない。

## 安全な縮約

- 派生固定試験事例の `mode_constraint` は `read_only` に固定する。
- 元事例がファイル作成、command 実行、外部 API 呼び出しなどの副作用を含む場合、実行者には実行させない。選択する操作、出力先、変更案、または検証計画を回答させる形へ縮約する。
- 縮約後も、元事例の決定点、判断に使う根拠の役割、pass / fail の境界を出力だけから判定できなければならない。
- 報告には、縮約後の判定対象と、実行結果や作成物など検証できなくなった性質を分けて書く。縮約した草稿を元事例の完全な再現として扱ってはならない。
- `read_only` へ縮約できない元事例から草稿を作ってはならない。

## 重複の除外

- 各草稿に、他の草稿と異なる検査軸を1つ付ける。検査軸は、その変形によって露出しうる失敗の違いを示す。
- リポジトリ名、識別子、ファイル名、題材、path の区切り文字だけが異なり、同じ出力が同じ理由で pass する候補は重複とする。
- path 表記自体が決定点でない限り、区切り文字やドライブ名の変更を独立した検査軸にしてはならない。
- 重複候補は1件にまとめる。候補数を満たすために派生を増やしてはならない。

## 手順

1. `target_selection_status` と `execution_route` を確定する。
2. `execution_route: survey_only` なら適格候補を一覧化し、候補一覧だけをローカル報告へ保存して終了する。規則だけが指定された場合、その規則に関係し、観測事実を引用できる失敗記録だけを候補に含める。
3. `execution_route: derive` なら、対応する元事例を特定し、観測事実、決定点、根拠構造、成果物判定を引用付きで抽出する。
4. 事例族を作り、保存する条件と変えてよい条件を明記する。
5. `read_only` への縮約可否を判定し、縮約後の判定対象と検証しない性質を固定する。
6. 表層変更または別ドメイン移植の候補を作り、重複を除く。1事例族あたり3件を上限とするが、件数を目標にしない。
7. 各候補について、実行者向け本文、採点者向け判定基準、元事例との対応表、検査軸を草稿として作る。
8. 草稿に捏造検査例を付け、誤った行動が fail になり、期待行動が pass になるか確認する。
9. 草稿と採点基準を同じファイルに置くが、実行時に採点基準を実行者へ渡してはならないことを明記する。
10. 派生 agent が草稿を人間の採択待ちとしてローカル報告へ保存する。fixture 実行者には保存させない。

## 利用制限

- 草稿は頑健性確認だけに使う。
- 派生件数を観測事例数、失敗率、効果量、基準率の根拠として使ってはならない。
- 介入の採否は元の独立観測事例族ごとに判断する。
- 派生固定試験事例草稿を `opencode-prompt-dev/fixtures/` 配下の追跡対象固定試験事例パックへ直接追加してはならない。
- 元事例の確認なしに、一般論や想像から草稿を作ってはならない。

## 派生 agent の報告保存先

## 禁止

- prompt、skill、command、追跡対象 fixture、失敗記録を編集してはならない。
- 実行者向け本文で、ファイル作成、command 実行、外部 API 呼び出し、環境変更を要求してはならない。
- 対象指定がない実行で派生固定試験事例草稿を作ってはならない。
- `execution_route: survey_only` の報告に、実行者向け本文、採点者向け判定基準、捏造検査を含めてはならない。
- 観測済みと派生を同じ数値に集計してはならない。
- 元事例の private な本文や秘密情報を追跡対象へ転記してはならない。
- hidden chain-of-thought を出してはならない。
