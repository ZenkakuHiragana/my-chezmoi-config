# 失敗事例から評価 fixture への変換手続き

## 目的と位置づけ

- 観測された失敗事例（セッション引用、失敗記録）を、blind eval で定量評価できる試験項目（fixture）へ変換する手続きを固定する。
- 「失敗記録は fixtures の在庫である」（`fundamental-problem-map.md` P5 管理戦略）の在庫→計器の変換工程にあたる。
- fixture は機械可読な固定書式を持たない。下位モデルが解釈できる平易な自然言語で書く。
- 適用例: `opencode-prompt-dev/fixtures/revealed-criteria-scenarios-v0/`（本手続きで作られた最初のパック）。

## 用語

- episode: 失敗の観測1件。セッション中のやりとり、失敗記録、ユーザーの証言。
- fixture: 凍結された試験項目。実行者向け本文と採点基準の組。
- 実行者: fixture の本文だけを与えられて作業する fresh context のエージェント。
- 採点者: 実行者の出力と採点基準だけを与えられて判定する、実行者とは別のエージェントまたは人間。
- 判定基準: 実行者の出力が持つべき、観測可能な性質の二値判定。
- 汚染: 実行者が試験の意図や採点基準を事前に知ってしまうこと。

## 事例の選別条件

次をすべて満たす episode だけを fixture にする。

- 失敗が実際に観測されている（想像上の失敗クラスから作らない）。
- 決定点を特定できる。「ここで別の行動を取っていれば失敗しなかった」という最小の分岐が1つに絞れる。
- 期待挙動を観測可能な出力の性質として言える。「慎重にやる」「よく考える」は不可。「出力に X が含まれない」「Y を実装せず質問として分離する」は可。
- 再現に必要な文脈をおおむね1画面（インラインのコード断片＋依頼文）に圧縮できる。

向かない episode: 長時間セッションの文脈崩壊そのものが原因のもの（fixture の単発実行では再現しない）、実環境での実行結果が判定に必須のもの（検証計画の設計品質を採点する形に縮約できないなら諦める）。

## 変換手順

1. **決定点の抽出。** episode を読み、失敗を生んだ最小の分岐を1つ選ぶ。分岐が複数あるなら fixture を分ける。1 fixture 1決定点。
2. **最小再現文脈の構築。** その分岐の判断に必要な材料だけをインライン化する。実在リポジトリへの参照は切り、構造を保ったままドメインを差し替えてよい。
3. **盲検化。** 試験対象の挙動を示唆する語を実行者向け本文から除く。ファイル名・見出し・パックのフォルダ名も漏洩面になる。依頼文は実際のユーザーが書きそうな粒度に保つ（過剰に親切な依頼文は現実の分布から外れる）。盲検化が遮断するのは試験メタ情報（パック名、採点基準、試験であるという事実）であって、運用の配送形式ではない。実行者への配送は運用と同じ形式（assignment packet 等）で行い、その配送形式ごと fixture の一部として凍結する。ただし配送形式の field に判定基準と重なる誘導を書いてはならない。wrapper が試験対象の挙動を指示すると天井効果で計器が無効になる。
4. **判定基準の設計。** 3〜5個。各基準は1文・二値・観測可能。内観、態度、自信、文体を採点対象にしない。「〜を含まない」「〜を実装せず提案として分離している」「〜を仮定として明記している」の形を優先する。
5. **捏造検査（基準自体の較正）。** episode の実際の失敗を模した悪い出力を採点基準にかけ、fail することを確認する。期待挙動どおりの良い出力が pass することも確認する。どちらかが崩れる基準は表層語句への Goodhart 化を起こしているので書き直す。
6. **分離。** 実行者向け本文と採点基準を別ファイルに置く。採点基準・出所・試験意図は採点側にだけ書く。
7. **出所の記録。** session id、日付、失敗記録番号を採点側ファイルに書く。実行者側に置いてはならない。
8. **凍結。** fixture パックに版名（`-v0` 形式）を付けて追跡ファイルとして保存する。以後の文言変更は新しい版として扱い、版を跨いだ結果比較をしない。

## 定量化の規約

- 判定基準ごとに pass / fail を記録し、基準別の通過率で報告する。scenario 合計点は補助指標に留める。
- 1回の pass / fail はノイズ。プロンプト変更のゲート判断は腕ごとに10回以上を目安にする。規則の効果を主張する統計的判断は15〜30標本を床とする。
- 実行と採点は分離する。実行者は Sonnet 等の下位〜中位モデルの fresh subagent、採点者は別呼び出し。実行者の応答を採点者に渡すとき、実行者のセッション文脈は渡さない。

## 陳腐化の扱い

- リポジトリ規則の更新により「常に pass する」ようになった fixture は、無価値になったのではない。回帰カナリアとして残し、`opencode-prompt-dev/sunset-ablation-procedure.md` の入力にする。対象規則を外した状態で実行して fail に戻るなら、その規則の価値の測定になっている。fail に戻らないなら規則側が減価している。
- 古い episode 由来であることは fixture の欠陥ではない。episode の日付を採点側に残し、解釈時に参照する。

## 手続き化できない残余

以下の2工程は判断作業であり、チェックリストでは置き換えられない。強いモデルまたは人間が担う。

- **決定点の抽出と文脈圧縮**（手順1〜2）。何を削っても分岐の判断が保たれるかの見極めは圧縮判断そのもの。緩和策: 圧縮の変種を2〜3個作って比較し、分岐が保たれる最小のものを選ぶ。1個だけ作って採用しない。
- **判定基準の Goodhart 耐性**（手順4）。表層語句で通ってしまう基準は書いた本人には見えにくい。緩和策: 手順5の捏造検査を省略不能にする。捏造検査に使った悪い出力はパックに残し、採点者の較正材料として再利用する。

## 汚染への注意

fixtures は追跡ファイルであるため、このリポジトリで作業するエージェントのコンテキストに混入しうる。fixture を実行する実行者は、fixture パックを読んでいない clean context で起動する。実行者に与えるのは scenario 本文の貼り付けだけとし、ファイルパスを教えない。
