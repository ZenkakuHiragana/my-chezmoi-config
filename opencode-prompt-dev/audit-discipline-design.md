## 1. 目的と設計の中心

監査装置が産出すべき商品は欠陥検出ではなく、正当化された注意配分である（trust-collapse §1）。人間が「どこを疑えばよいか」を安く知れる状態を作り、依頼外の混入1件が全数詳細レビューへの転落を強いる構造を断つ。

設計は次の対応を骨格とする。

| 三性質                           | 検査装置                             | 台帳                |
| -------------------------------- | ------------------------------------ | ------------------- |
| 対応の健全性（根拠が本物である） | 証明書規律 + resolver                | 根拠台帳            |
| 左全域性（依頼外の混入がない）   | 分母の機械列挙 + 複式突合 + 判定被覆 | 変更台帳 + 認可台帳 |
| 右全域性（言われたことをやった） | 条項と変更の対応表（契約粒度に律速） | 認可台帳            |
| 装置自身の信頼                   | 較正記録                             | 計器台帳            |

証明書規律と全域性の監査は別装置ではない。同じ対応表（変更と意図の二部関係）を、実装者の自発申告と機械列挙という2つの主体が、作成時と turn 終了時という2つの時点で埋める1つの装置である（trust-collapse §4）。本書の設計はこの同一性を保つ。

保護 glob（作成時ブロック）は検査範囲の全体ではない。リポジトリ内の全変更が変更台帳と判定被覆の対象であり、保護 glob は作成時に証明書を要求する範囲だけを定める（§6.1）。

この装置の真の故障モードは論理破綻ではなく警報疲労である。したがって本書の設計判断は一貫して、人間へ届く雑音（孤立・保留・帰属不明）の量を SN比として会計し、還流経路で減らす形をとる（§7）。

## 2. 制約条件

すべて既決。実行時に緩めてはならない。変更は人間の決定と上位契約の改訂を要する。C1〜C8 は旧 B1 設計書から引き継ぎ、C9 を追加した。

- C1 移植性: 対象 harness（エージェント実行環境）は OpenCode / Oh-My-Pi / Claude Code / Codex の4つ。OS は Windows を含む。計器は Node.js コアモジュール + 薄い CLI + MCP server の3層で提供し、照合・記録・突合のロジックはコアモジュールだけが持つ。公開配布、PATH 登録、ExecutionPolicy 等のホスト設定に依存する配布形態を採用してはならない。
- C2 git: エージェントに git の破壊的操作を許可しない。コミットは人間が行う。したがって pre-commit / CI はエージェントループ内の発火機構にならない（人間コミット時の補助ゲートとしてのみ任意採用）。状態差分の取得は read-only の git 照会に限る。
- C3 帳簿分離: 証明書、実行記録、判定記録はタスクの消費財であり、非追跡のローカル規約ディレクトリに置き、バージョン管理に載せない。fixtures、resolver、schema 文書、cert-policy、常設許可規則、較正記録は資本財であり、リポジトリで追跡する。
- C4 統治予算: 常駐プロンプト指示は最小限に留め、教示の大半は差し戻しメッセージ（発火時にだけコンテキストへ入る）に載せる。規則追加時は同量の削除または移設を明示する。
- C5 モデル配分と委譲: fixtures と schema の作成、LLM 判定器の較正基準の策定は強いモデルまたは人間が行い、下位モデルへ委譲してはならない。resolver・journal・突合の実装は、凍結済み fixtures と機械的完了条件を前提に下位モデルへ委譲できる。委譲の assignment packet にセッション経緯を書いてはならない（B3 制約）。
- C6 プロンプト階層: 常駐指示はグローバル AGENTS.md に置かず、該当 skill 本文へ置く。グローバル層に分岐を書かない。
- C7 言語・語彙: prompt 本文へ持ち込む新しい制御語彙は `opencode-prompt-dev/english-token-allowlist.md` へ登録する。義務表現は `opencode-prompt-dev/obligation-vocabulary.md` の正規マーカーを使う。
- C8 プロジェクト分離: 計器と書式の枠組みはプロジェクト非依存。schema の中身、保護 glob、must-cite レジストリ、常設許可規則の中身は各プロジェクトのリポジトリで定義する。
- C9 帰属戦略: 記録型を採用する（ユーザー決定 2026-07-09）。適用先プロジェクトの成果物（GMOD アドオン）は決められた場所への配置が必要であり、作業空間を隔離する分離型が使えない。worktree 分離を帰属確保の主手段として採用してはならない。代償として観測点の網羅保守を払い、hook の観測網羅率を計器台帳の較正対象とする（§4.4）。

## 3. 全体アーキテクチャ

### 3.1 構成要素

| ------------------- | ------------------------------------------ | ------ | -------------------------------------------------------------------------------------------------- |
| 証明書              | `.opencode/work/audit/certs/`              | しない | 実装者が作成時に埋める照合可能な表。JSON 1ファイル=1証明書                                         |
| 実行記録（journal） | `.opencode/work/audit/journal/`            | しない | 書き込みの都度追記される JSONL。帰属の供給源                                                       |

| 常設許可規則        | プロジェクトごと `permanent-permits.json`  | する   | 再発する許可パターンの登録簿（§8）                                                                 |

ローカル監査ディレクトリは `.opencode/work/audit/` とする。配下に `certs/`、`journal/`、`verdicts/` を置き、git 管理に載せない。OpenCode を第一接続 harness とするため、監査生成物の規約名は harness 中立にしない。

CLI・MCP・hook adapter は照合・記録・突合ロジックの重複実装を持ってはならない。検証は常にコアモジュールに対して `node` で実行できる。

### 3.2 イベントと台帳の接続

監査は3種のイベントで駆動される。発火の判定にタスクの複雑さやエージェントの自己申告を使ってはならない。

1. 書き込み前（PreToolUse 型）: 発火述語 F1〜F2（§6.1）に該当する書き込みを、証明書の照合を通らない限りブロックする。根拠台帳の作成時衝突。
2. 書き込み後（PostToolUse 型）: 発火述語 F4（§6.1）により実行記録へ1行追記する。変更台帳の帰属供給。ブロックしない。
3. turn 終了（Stop 型）: 状態差分を列挙し、実行記録との複式突合、保護対象の証明書再照合、証明書改竄の検査を行う。変更台帳の全域検査。発火述語を持たず、turn 終了そのものが発火である。

PreToolUse 型の4 harness 対応は公式仕様で確認済み（§12）。PostToolUse 型・Stop 型の各 harness での実仕様（発火の確実性、payload、turn 終了の拒否可否）は実装時確認項目とする（I1）。

## 4. 四台帳の設計

### 4.1 変更台帳

問い: 何が変わったか。誰の、どの作業単位による変更か。説明できない残差は何か。

- 分母: 状態差分（read-only の git status / diff。untracked を含む）。リポジトリに物質化した変更に限る。リポジトリ外の副作用（外部 API 呼び出し、環境変更、リポジトリ外ファイル）は境界外であり、本装置は扱わない。
- 帰属: 実行記録。書き込みの都度、次のフィールドを持つ行を追記する。
  - `ts`（ISO 8601）、`harness`、`session_id`、`task_id`（task file または依頼の識別子。不明時は `unknown`）、`agent`（親か subagent か）、`tool`（ツール名）、`path`（対象パス）、`content_hash`（変更後内容のハッシュ）、`prev_hash`（変更前内容のハッシュ。取得できた場合のみ）
- 複式突合: 状態差分の各要素を実行記録の合計と突き合わせ、説明されない変更を残差として出力する。残差は観測欠損であって意図問題ではない。人間編集の混入、Bash 等フック外経路の書き込み、hook 不発のいずれかへ人間の裁定で分岐する。
- 個体化: 初版はファイル粒度とする。差分塊の粒度は隣接編集や整形で同一性が壊れ、再判定を誘発するため、細分は粒度別の再判定率を実測してから判断する（trust-collapse §6.1）。
- 完全観測は証明できない。買うのは観測漏れの残差化である（trust-collapse §5.2）。Bash 経由の書き込みは PreToolUse 型で保守的に検査し（保護 glob への書き込み・リダイレクトを含むコマンドは deny して Edit / Write 経由へ誘導）、検査を逃れた書き込みは Stop 型の複式突合が残差として受ける。

### 4.2 認可台帳

問い: その変更は何によって許されるか。要求されたことは実現されたか。

- 意図集合の供給源: task contract / `Requirement contract` の条項、ユーザー発言の記録、常設許可規則、task contract の write_set。意図の完全性は前提にしない（完備化禁止。trust-collapse §5.3）。
- 判定: 列挙された全変更に判定5区分（認可済・派生・孤立・保留・帰属不明）のいずれかを付ける（判定被覆）。判定被覆は「全変更が認可されている」ことではなく「全変更が分類済みである」ことを意味する。
- 判定器は3層で構成し、安い層から順に適用する。
  1. 機械判定: write_set 内の変更 → 認可済。常設許可規則に一致 → 派生。生成物の再生成一致（記録された生成器・入力・版から再生成して内容が一致） → 派生。実行記録に対応行がない → 帰属不明。
  2. LLM 判定器: 残った変更を条項へ帰属させる。初版は同じモデルの fresh context を使い、変更1件と条項集合を渡す狭い問いに限る。モデル族を跨ぐ脱相関は必須にしない。判定不能は保留とする。
  3. 人間裁定: 保留と孤立の処置（削除すべき発明か、仕様へ昇格すべき派生要件か）。
- 右全域性（未実現条項の検出）は契約の粒度に律速される。条項の精密化は目的適合（精密化で上がる SN比がコストを超えるときに限る）で止める。質問による条項の追加は `intent-elicitation-discipline` の発火・停止条件に従う。

### 4.3 根拠台帳（証明書規律）

問い: その主張、対応関係は何に支えられているか。旧 B1 設計書の内容をここへ引き継ぐ。

- 証明書の書式（作業仮説。確定は WI3c の fixtures 成文化で行い、実物と食い違った場合は実物を優先して本節を改訂する。この改訂は本書の改訂として完結し、上位契約が固定する範囲（証明対象、work item 構成、受け入れ条件）には触れない）:

```json
{
  "schema_version": 1,
  "target": "source-of-truth | contract | external-identifier",
  "task_id": "対応する task file または依頼の識別子",
  "covers": ["この証明書が免許する書き込み先パスの一覧"],
  "created_at": "ISO 8601",
  "rows": [
    {
      "claim": "主張（日本語自由文）",
      "source_path": "出典パス（cert-policy の root 相対）",
      "range": { "start": 1, "end": 10 },
      "quote": "range の逐語引用",
      "identifiers": ["claim に含まれ quote 内に出現すべき識別子"]
    }
  ]
}
```

- 機械照合列は `source_path` / `range` / `quote` / `identifiers`。`claim` は自由文で、関連性判定（分離検証者への狭い問い）と抜き取り照合の対象。鮮度は照合時刻のファイル内容で担保する（原本が変わっていれば quote 不一致として落ちる）。
- 照合仕様（audit core）:

| 検査             | 内容                                                                               | 失敗の意味             |
| ---------------- | ---------------------------------------------------------------------------------- | ---------------------- |
| V1 パス実在      | `source_path` が root 配下に実在する。シンボリックリンク解決後も root 内           | 出典の捏造             |
| V2 引用一致      | `quote` が `range` の行内容と逐語一致。正規化は改行コード（CRLF/LF）と行末空白のみ | 引用の捏造・原本の変化 |
| V3 識別子出現    | `identifiers` の各要素が `quote` 内に単語境界で出現                                | 主張と引用の乖離       |
| V4 covers 束縛   | 書き込み予定パスが `covers` に含まれる（作成時発火のみ）                           | 別作業の証明書の流用   |
| V5 schema 妥当性 | 必須フィールドと型が書式に適合                                                     | 書式不備               |

- 出力契約: stdout に JSON `{ "verdict": "pass" | "fail", "failures": [{ "row": n, "check": "V1..V5", "expected": "...", "actual": "..." }] }`。exit code は 0 = pass、1 = fail、2 = 入力・設定エラー。`failures` は差し戻しメッセージ生成の入力になる。
- 生成後捏造（先に作られた成果物へ逆算された、全行真だが成果物と不適合な証明書）は resolver では落ちない。抜き取り照合の適合検査（成果物のこの箇所はこの行の主張に適合するか）と分離検証者が受ける。fixtures の標準クラスとして維持する。
- 提出時（調査結論・助言・完了報告）は default-deny により、根拠レコードを伴わない主張を「仮定」として明示分類する。これは skill 層の指示であり、機械強制しない。全主張への証明書強制は儀式化とオーバーヘッドを生むため行わない。

### 4.4 計器台帳

問い: 上の3台帳を測る装置は、今も信用できるか。

すべての計器は建てた瞬間から較正負債を持つ。較正の切れた計器は計器なしより悪い（誤った安心を生産する）。較正記録（`calibration.md`）に、計器ごとの {較正方法, 較正周期, 最終較正日} を保持し、周期を過ぎた計器は較正切れとして扱う。

| 計器                       | 較正方法                                                                   | 較正の契機                      |
| -------------------------- | -------------------------------------------------------------------------- | ------------------------------- |
| hook adapter（観測点）     | 試験入力（既知の書き込み系列）を流し、実行記録との一致率＝観測網羅率を測る | harness の版更新ごと            |
| audit core（V1〜V5、突合） | fixtures 全通過・全検出                                                    | fixtures 追加ごと、実装変更ごと |
| LLM 判定器                 | 人間裁定との突合による誤り率。生成後捏造 fixtures の held-out 検出率       | モデル世代交代ごと              |
| 質問チャネル               | 回答の覆り率（`intent-elicitation-discipline` §9）                         | triage ごと                     |
| fixtures 自体              | 失敗記録からの供給が続いているか（在庫の鮮度）                             | triage ごと                     |

較正基準の策定（何を誤りと数えるか、held-out セットの構成）は C5 に従い強いモデルまたは人間が行う。較正の実施（測定と記録）は下位モデルへ委譲できる。

較正切れの計器の判定は保留へ降格し、監査レポートに較正切れとして明示する。「監査レポートが静かになった」を品質改善と観測系の腐敗のどちらか判別できない状態にしてはならない。

## 5. 深度グラデーション

作業面ごとに、回す台帳と深さを固定する。発火はすべて §3.2 のイベント駆動であり、省略を判断する主体を置かない。

| 作業面                                           | 変更台帳                               | 認可台帳                          | 根拠台帳                                  |
| ------------------------------------------------ | -------------------------------------- | --------------------------------- | ----------------------------------------- |
| 保護面への書き込み（保護 glob / must-cite 該当） | 実行記録 + Stop 型突合                 | 判定被覆                          | 作成時ブロック（証明書必須）              |
| 保護面外のリポジトリ書き込み                     | 実行記録 + Stop 型突合                 | 判定被覆（turn 終了時の事後判定） | 非発火。提出時 default-deny のみ          |
| 読み取りのみ・会話・リポジトリ外調査             | 非発火（書き込みイベントが発生しない） | 非発火（分母が空）                | 非発火                                    |
| 調査結論・助言・完了報告の提出                   | —                                      | —                                 | default-deny 分類（非ブロック、skill 層） |

日常の軽い依頼（壁打ち、雑談、リポジトリ外調査）が壊れない根拠は構造的である。全台帳の発火述語は書き込みイベント駆動であり、書き込みが発生しない依頼では分母が空になり、どの台帳も行を持たない。ブロックが起きるのは保護面への証明書なし書き込みだけであり、判定被覆の要求はブロックではなく turn 終了時の記録として満たされる。省略可否を判断する主体が存在しないため、P4 汚染なしに非発火となる。

## 6. 発火機構

### 6.1 発火述語

作成時（ブロック型。§3.2 の PreToolUse 型イベントで発火する）。次のいずれかに該当する書き込みは、照合を通る証明書なしに実行してはならない。

- F1: 書き込み先が cert-policy の保護 glob に一致する。該当する `covers` を持つ証明書の全行照合通過が必要。
- F2: 候補内容に must-cite パターンに一致する識別子が含まれる。その識別子を `identifiers` に含む照合通過行が必要。

記録時（非ブロック型。§3.2 の PostToolUse 型イベントで発火する）。リポジトリ内への書き込みはすべて実行記録の対象とする（F4）。F4 は判定条件を持たない。書き込みイベントそのものが発火である。

F1・F2 のどちらにも該当しない操作（些細な問答を含む）にはブロックを要求しない。task contract の write_set は作成時ブロックの条件にしない。write_set は認可台帳の機械判定層で扱い、write_set 外の変更は turn 終了時の判定被覆で孤立または保留として浮かせる。

本リポジトリの cert-policy 初期値では、保護 glob を実行時挙動を変えるプロンプト運用面に限定する。初期値は次の面とする。

- `.opencode/commands/`
- `dot_config/opencode/AGENTS.md`
- `dot_config/opencode/agents/`
- `dot_config/opencode/commands/`
- `dot_agents/skills/`
- `.chezmoitemplates/opencode/`

must-cite パターンの初期値は、`opencode-prompt-dev/english-token-allowlist.md` の制御語彙のうち、skill 名、agent 名、schema field、status 値として使う識別子に限定する。自然言語の一般語は含めない。

git 管理下全体を保護 glob にしてはならない。保護 glob は「後で全変更を判定する範囲」ではなく「作成時にブロックして証明書を要求する範囲」である。リポジトリ内の全変更は F4、Stop 型突合、認可台帳の判定被覆で扱う。設計文書、失敗ログ、fixtures、実験記録を初期保護面に含めると、証明書が不要な変更にも作成時ブロックが発火し、SN比を下げる。

### 6.2 harness 接続

主経路は PreToolUse 型（ブロック）+ PostToolUse 型（記録）+ Stop 型（突合）の3点接続とする。PreToolUse 型は4 harness すべてで候補内容の受領とブロックが公式仕様で確認済み（§12）。

| harness     | ブロック（確認済み）                                                        | 記録・突合（実装時確認: I1）                                   |
| ----------- | --------------------------------------------------------------------------- | -------------------------------------------------------------- |
| Claude Code | `PreToolUse`: `tool_input` 受領、exit 2 または `permissionDecision: "deny"` | `PostToolUse` / `Stop` 相当イベント                            |
| OpenCode    | `tool.execute.before`: `output.args` 受領、throw でブロック                 | `tool.execute.after` 相当。subagent バイパスのバグ状況を再確認 |
| Oh-My-Pi    | `tool_call`: `{ block: true, reason }`                                      | 対応イベント                                                   |
| Codex       | `PreToolUse`: `permissionDecision: "deny"` または exit 2                    | `PostToolUse` 相当イベント                                     |

- 第一接続 harness は OpenCode とする。MCP server は公開配布せず、OpenCode の設定からローカルの `node opencode-prompt-dev/audit/mcp.mjs` を直接起動する。
- Stop 型で turn 終了を拒否できない harness では、監査失敗を `.opencode/work/audit/` 配下へ永続化し、終了直後の次メッセージで未処置の監査失敗への対応を要求する。未処置の監査失敗が残る間、次 turn の書き込みは新規作業へ進めず、監査失敗の解消を先に行う。
- pre-commit は人間がコミットする時点の補助ゲートとして任意採用する。エージェントループ内の強制ではないため主経路にしない。

### 6.3 差し戻しメッセージ契約

ブロック時のメッセージは core の `failures` JSON から機械生成し、次を含める。

- 失敗した検査（V1〜V5 / F1〜F2）と、期待・実際の値。
- 証明書の置き場所と書式の最小例、または schema 文書へのパス。
- 再試行の手順（証明書を作成または修正してから同じ書き込みを再実行する）。

教示の大半をここへ載せることで、常駐プロンプトの統治予算消費を避ける（C4）。

## 7. SN比の監視と警報疲労の管理

- SN比: 信号 = 認可済 + 派生、雑音 = 孤立 + 保留 + 帰属不明。判定記録からタスク単位で機械集計する。
- 人間へ提示するもの: 残差、孤立、保留、較正切れ計器、および認可済・派生からの抜き取り。認可済・派生の全量提示はしない。人間の役割は全読ではなく抜き取り照合と計器への信頼配分である。
- 警報疲労の閾値: SN比（信号:雑音）が直近5タスクの移動平均で 7:3 を下回ったら、常設許可規則の収穫レビューを発火する。この値は理屈による仮置きであり、装置導入の可否を決める測定項目（trust-collapse §9）の実測で較正する。閾値の妥当性自体を計器台帳の対象とする。
- 誤判定は発生源別に計数する: 契約の書き漏らし、人間編集に由来する帰属不明、個体化による再判定。
- 雑音への還流は2経路に限る: 常設許可規則への収穫（§8）と、人間の裁定による残余コスト計上。タスクごとの契約細分化で雑音を埋めてはならない（完備化禁止）。

## 8. 常設許可規則

- 形式: プロジェクトごとの追跡ファイル `permanent-permits.json`。各規則は次のフィールドを持つ。
  - `id`、`kind`（`glob` | `generator` | `operation-class`）、`pattern`（glob または生成器・操作の識別）、`reason`（許可の根拠。1文）、`harvested_from`（収穫元の判定記録または失敗記録への参照）、`since`（導入日）、`review_at`（見直し日）
- 判定への効き方: 一致する変更は判定器の機械層で「派生」となる（§4.2）。
- 収穫条件: 同一クラスの孤立・保留が2つ以上の別作業で再発し、人間が承認したときに限り収穫する。1作業からの収穫をしてはならない（1事例からの規則化の禁止と同型。設計原則5）。
- `intent-elicitation-discipline` の許可条項（don't-care の同値類）は、再発時にここへ収穫される。質問回答の三値保存形式との接続点である。
- 成長曲線（規則数の推移）を triage で監視する。増加が収束しない場合は、意図集合の供給源か個体化の粒度に構造問題があるとみなし、本書の見直し根拠とする。

## 9. work item breakdown

上位契約の work item 1〜6 を細分化する。担当区分は C5、順序は上位契約に従う（work item 2 と 3 は独立並行可）。WI2〜WI6 は WI1 の完了を前提とする。§11 の確定事項を、該当 work item の入力として扱う。

### WI1: 本書の確定（強いモデルまたは人間が起草し、人間が承認する）

- 完了条件: 上位契約の受け入れ条件と本書の整合が確認され、§11 の確定事項が本文へ反映されている。

### WI2: 変更台帳（下位モデルへ委譲可）

- WI2a 実行記録: journal の書式（§4.1）と追記 API を core に実装し、hook adapter で1つの harness の書き込みイベントへ接続する。
- WI2b 複式突合: 状態差分の列挙（read-only git 照会）、実行記録との突合、残差の出力を core に実装する。
- WI2c 実演: Bash 迂回の書き込みが残差として検出されることを実演し、記録する。
- 完了条件: fixtures（実行記録つき）で突合と残差計算が期待判定に一致する。上位契約の変更台帳受け入れ条件の実演が記録されている。

### WI3: 根拠台帳

- WI3a fixtures 作成（強いモデルまたは人間）: 失敗記録 `.opencode/local-failure-logs/20260628-1706-review-feedback-overapplied.md`（原本・外部識別子）と `.opencode/local-failure-logs/20260705-1118-scope-shrunk-persistent-subtasks.md`（契約）から証明書の実物を手書きする。各 fixture は自己完結ディレクトリとし、`sources/`（疑似原本ツリー）、`cert.json`、`expected.json`、必要に応じて `artifact/` と実行記録を含む。外部リポジトリの実コードを持ち込まず、失敗記録の構造を再現した合成ファイルを使う。最小セット: 各証明対象（source-of-truth / contract / external-identifier）につき正例2枚以上・捏造例2枚以上。捏造例には引用改変（V2 で落ちる）、識別子不出現（V3 で落ちる）、covers 外書き込み（V4 で落ちる）、存在しない task identity への言及、生成後捏造（resolver pass が期待値。分離検証者の held-out テスト用）を含める。
- WI3b resolver 実装（下位モデルへ委譲可）: 凍結済み fixtures に対し V1〜V5 を実装する。LLM を使わない。
- WI3c schema 成文化と cert-policy 定義（強いモデルまたは人間）: 生き残った fixtures から列定義を帰納し `cert-schema.md` に成文化する。全列に「この列を要求した fixture」への参照を付け、fixture が要求しなかった列を追加してはならない。cert-policy.json の書式（保護 glob、must-cite パターン、監査ディレクトリ位置、除外）を定義し、F1・F2 が機械判定できる形で固定する。schema・rubric の命名は、対応する失敗記録の pattern_tags が存在する語だけを必要時に収穫する。
- 完了条件: 上位契約の根拠台帳受け入れ条件（正例全通過・捏造例全検出、3層アダプタ疎通、fixtures 枚数、schema 列参照）。

### WI4: 発火機構の接続

- 第一接続 harness は OpenCode とする。hook adapter はブロック（F1・F2）、記録（F4）、Stop 型突合の3点を接続し、照合ロジックを持たない。差し戻しメッセージは §6.3 に従う。
- 完了条件（実演を記録する）: (a) 保護 glob への証明書なし書き込みがブロックされる。(b) 差し戻し文の教示に従って作成した証明書で同じ書き込みが通る。(c) Bash 迂回の書き込みが Stop 型突合で残差として検出される。(d) ゲート通過後の証明書改竄が Stop 型突合で検出される。

### WI5: 認可台帳（判定器と集計）

- 判定器の機械層（write_set・常設許可規則・再生成一致・実行記録欠落）を core に実装する。LLM 判定器の問い（変更1件×条項集合）と assignment packet 書式を定義する。初版の LLM 判定器は同じモデルの fresh context を使う。判定記録の書式と SN比の集計を実装する。`permanent-permits.json` の書式（§8）を固定する。
- 担当区分: LLM 判定器の問いと assignment packet 書式の定義は強いモデルまたは人間が行う（C5）。機械層・判定記録・集計の実装は下位モデルへ委譲できる。
- 完了条件: 1つの実タスクで全変更が判定5区分に分類され、SN比が集計されている（上位契約の認可台帳受け入れ条件）。

### WI6: プロンプト統合とコスト実測

- 常駐指示を該当 skill 本文（`implementation` / `investigation` 等）へ最小限（発火述語への言及と schema 参照。目安1〜2行）追加し、統治予算の支払いを差分に明示する。default-deny の提出時分類も skill 層の指示として整備する。持ち込む制御語彙を english-token-allowlist.md へ登録する。
- 実測: 初回ブロック率、ゲート通過までの turn 数、タスクあたりの追加トークン、SN比、残差率、常設許可規則の成長曲線。プロンプト統合は blind eval（凍結シナリオ、盲検実行者、分離採点者）で指示あり／なしを比較する。
- 完了条件: 指示差分が統治予算の支払いつきで適用され、blind eval の結果と実測値が記録されている。

## 10. 検証とこの設計自体の反証条件

- 機械検証: fixtures 全通過・全検出（WI2・WI3）。発火機構の4実演（WI4）。
- 経験的検証: blind eval とコスト実測（WI6）。SN比、残差率、常設許可規則の成長曲線、個体化の再判定率、判定器の較正安定性は trust-collapse §9 が固定した測定項目であり、理屈で先取りしてはならない。
- 反証条件。次の観測は本書または上位契約の書き換えを要求する。
  - 指示ありでも初回ブロック率が下がらない: 常駐指示か差し戻し文の再設計。
  - 生成後捏造の held-out 検出率が低い: 適合検査の問いの再設計、または証明書規律の適用縮小。
  - SN比が閾値を恒常的に下回り、常設許可規則の収穫で上がらない: 意図集合の供給源か個体化粒度の再設計。
  - 残差率が高止まりする（観測網羅率が上がらない）: 記録型の観測点構成の再設計。C9 の帰属戦略自体の再検討は人間の決定事項とする。
  - コスト増分が失敗分布の改善（P2 主因の減少）に見合わない: 発火点の限定強化、または投資停止と上位契約の書き換え。

## 11. 確定事項と実装時確認

### 確定事項

- 監査ディレクトリは `.opencode/work/audit/` とし、配下に `certs/`、`journal/`、`verdicts/` を置く。harness 中立名にはしない。
- 第一接続 harness は OpenCode とする。
- 本リポジトリの cert-policy 初期値は §6.1 の保護 glob と must-cite パターンに従う。git 管理下全体を保護 glob にしない。
- MCP server は公開配布せず、OpenCode の設定からローカルの `node opencode-prompt-dev/audit/mcp.mjs` を直接起動する。
- Stop 型で turn 終了を拒否できない場合は、監査失敗を永続化し、終了直後の次メッセージで未処置の監査失敗への対応を要求する。
- 警報疲労の閾値は SN比（信号:雑音）で表し、初期値は直近5タスクの移動平均で 7:3 とする。
- LLM 判定器は初版では同じモデルの fresh context を使う。モデル族を跨ぐ脱相関は必須にしない。

### 実装時確認（着手をブロックしない）

- I1: PostToolUse 型・Stop 型イベントの4 harness 実仕様（発火の確実性、payload に書き込み内容が含まれるか、turn 終了の拒否可否）。WI2・WI4 の実装中に一次資料で確認し、§12 に追記する。

## 12. 調査記録（2026-07-05 確認。旧 B1 設計書 §7 から引き継ぎ）

hook 仕様の一次資料確認。最新性が効く主張のため、実装時（特に数か月後）には再確認する。PreToolUse 型（ブロック）のみ確認済みであり、PostToolUse 型・Stop 型の記録系イベントは未確認（I1）。

- Codex: hooks は GA で既定有効（`[features] hooks = false` で無効化する側の設定）。`PreToolUse` は `tool_name` / `tool_use_id` / `tool_input` を受け、`permissionDecision: "deny"` / exit 2 でブロック、`updatedInput` で書き換え可。公式に「PreToolUse は Bash、apply_patch による file edits、MCP tool calls を intercept できる」と明記。出典: <https://developers.openai.com/codex/hooks>。注意: 第三者記事には「Bash のみ intercept」とする公式と矛盾する記述が流通している。一次資料を採る。apply_patch の payload に patch 本文が含まれることは構造上ほぼ確実だが引用例では未確認であり、WI4 実装時に実物で確認する。
- OpenCode: plugin の `tool.execute.before` が `input.tool` と `output.args`（filePath 等のツール引数）を受け、throw でブロック。出典: <https://opencode.ai/docs/plugins/>。注意: subagent のツール呼び出しを intercept しないバグ報告 <https://github.com/anomalyco/opencode/issues/5894>。WI4 で修正状況を再確認する。
- Oh-My-Pi: `tool_call` イベントが `toolName` / `input` / `toolCallId` を受け、`{ block: true, reason }` でブロック。handler の throw も fail-safe でブロック側に倒れる。出典: <https://github.com/can1357/oh-my-pi/blob/main/docs/hooks.md>。
- Claude Code: `PreToolUse` が `tool_input`（Write/Edit の書き込み内容込み）を受け、exit 2 または `permissionDecision: "deny"` でブロック。出典: <https://code.claude.com/docs/en/hooks>。
