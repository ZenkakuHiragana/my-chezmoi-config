# 要件レビュー手順

## 0. 本文書の位置づけ

- 本文書は、`Requirement contract` の品質を検査する手順を固定する。
- 上位文書は `opencode-prompt-dev/fundamental-problem-analysis/trust-collapse.md`（問題分析の正本）とする。trust-collapse.md は AI 委任における信頼崩壊の機序を整理し、変更と意図の対応関係の性質（対応の健全性、左全域性、右全域性、検証の適合性）を定義する。本文書は、そのうち検証の適合性と右全域性の契約版を担保する手順を扱う。
- 読者は要件レビュー手順の設計・実装・保守を行う人間とエージェント。本文書だけで、何をどう検査するか、どう運用するかを判定できることを目標とする。
- 本文書は手順の設計を固定する。実装（skill の作成、発火機構の実装等）は別途行う。

## 1. 全体像

要件レビューは、`Requirement contract` を対象に、`code-review` と同じ手続き（fresh context、`Review finding record`、severity）で行う観点別レビューである。ただし対象がコードではなく契約であるため、`code-review` とは別の skill として実装する。共通する手続き（`Review finding record` の形式、severity の定義）は `.chezmoitemplates` で共有し、二重管理を避ける。

発火条件は `Requirement contract` の凍結直後である。`context-clarification` が契約を凍結したとき、要件レビューが発火する。

要件レビューは2層で構成される。

- **verification 層**: 書かれた条項は妥当か。検証の適合性（trust-collapse §4）を含む。R1〜R5 の rubric で検査する。
- **validation 層**: 意図が要求する条項がすべて書かれているか。右全域性の契約版（trust-collapse §4）を担保する。転写照合表、盲目導出、差分照合で検査する。

DO-178C は verification（要件どおり作ったか）と validation（要件が正しいか）を別工程に分離する。要件レビューの2層は、この分離を契約レベルで適用したものである。

## 2. verification 層（R1〜R5）

verification 層は、`Requirement contract` の各条項を5つの観点で検査する。各観点は「はい / いいえ / 不明」で判定し、指摘は `Review finding record` として severity 付きで返す。

- **R1 文言だけで合否判定できるか**: 条項が客観的に確認できる形で書かれているか。「適切に」「なんとかして」のような主観的表現は不合格。
- **R2 verification method の実行経路は、依頼文が含意する実際の経路と一致するか**: 条項の verification method が、依頼文から読み取れる実際の使用経路と一致しているか。単体テストで十分か、統合テストが必要か。
- **R3 複数環境の要求は条項として分割されているか**: 依頼が複数環境（Windows/Linux 等）を含む場合、各環境の要件が別々の条項として書かれているか。
- **R4 何を観測して合格とするかが書かれているか**: 条項が exit code だけでなく、何を観測して合格とするか（アサーション、ログの内容等）を明示しているか。
- **R5 消費者が未特定なら known unknown として明示されているか**: 条項の消費者（誰が・どの経路で使うか）が未特定の場合、unknown と明示されているか。不明確なまま確定していないか。

R1〜R5 は過去の失敗（CMake/PowerShell 型、Win/Linux 境界の未決等）から収穫した観点である。観点の追加・削除は、実測された取りこぼしに基づいて行う。

## 3. validation 層

validation 層は、意図が要求する条項がすべて契約に書かれているかを検査する。書かれた条項の妥当性（verification 層）とは別の問いである。

validation 層は3つの構成要素からなる: 転写照合表、盲目導出、差分照合。

### 3.1 転写照合表

転写照合表は、`context-clarification` が `Requirement contract` を凍結するときに生成する。契約凍結手順の1ステップであり、独立した agent ではない。

転写照合表は、セッションのユーザーメッセージを順に全件なめ、各メッセージが契約に反映されたかを確認する表である。`Requirement contract` の外部化ファイルに一緒に保存する。

列は次の3つである。

- **メッセージ列**: ユーザーメッセージの逐語引用。契約作成者が抽出する。
- **処遇列**: そのメッセージの内容が契約にどう反映されたか。反映、除外、覆りのいずれか。
- **条項参照**: 処遇の根拠となった契約の条項番号または位置。

転写照合表は、`Requirement contract` と一緒にユーザーに提示する。ユーザーは、メッセージ列（ユーザー自身の発言）と処遇列を一瞥で確認できる。

フレーム相関とは、契約作成者が自分の判定を再読するとき、同じ解釈枠が再適用され、元の判定を正当化する方向に読んでしまう現象である。フレーム相関がある列は、自己監査では守れない。

転写監査表の各列は、フレーム相関の有無で2群に分かれる。

フレーム相関がない（自己監査で守れる）:

- **メッセージ列の行の存在**: ユーザーが何を言ったか。ユーザーの発言に対応する行が立たないことは構造上起こらない。
- **メッセージ列の引用の正確さ**: ユーザー発言の正確な複写。機械的な複写であり、読者はユーザー自身であるため、誤引用は一瞥で分かる。

フレーム相関が残る（自己監査で守れない）:

- **処遇列**: 反映/除外/覆りの判定。契約作成者の判定を、同じ作成者が再読しても同じ目で読む。一貫性圧が自分の判定を守る方向に働く。

処遇列の誤りは、3.2 の盲目導出と、ユーザーが転写監査表を眺めることで受ける。

転写監査表の効果は、落とし込み済みの依頼を植えたセッション fixture で、照合表あり/なしの契約凍結を盲検比較することで測る。効果が null なら切り、別手段を探す。

### 3.2 盲目導出（subagent）

盲目導出は、`Requirement contract` を読まずに、依頼引用と territory だけから導出骨格を生成する。fresh context の subagent が行う。

`Requirement contract` に依頼引用欄を追加する。依頼引用欄は、ユーザーが自分のメッセージに意図的に貼り込んだテキストの逐語引用を保持する。ユーザーが採用したテキストは、著者が誰であれ意図の一次証拠に数える。ユーザーが採用していないエージェントの語りは汚染であり、引用欄に混ぜてはならない。

subagent への入力は次の2つである。

- **依頼引用**: `Requirement contract` の依頼引用欄。
- **territory**: 対象リポジトリまたはディレクトリへの参照。subagent が自分で読む。

subagent には `Requirement contract` を渡さない。セッション経緯も渡さない。問いは「この依頼から実装を始めるために、何が確定していなければならないか。導出骨格を書け」である。

成果物は導出骨格（リスト）である。

依頼引用欄の忠実性は、3.1 の転写監査表が transcript と突き合わせて監査する。引用欄が間違っていれば、3.3 の差分照合で差分が大きくなり、引用欄の問題が疑える。

### 3.3 差分照合

差分照合は、3.2 の盲目導出が生成した導出骨格と、`Requirement contract` を突き合わせ、差分を分類する。

差分は3方向に分類される。

- **欠如候補**: 導出骨格にあって契約にない。意図が要求する条項が契約に書かれていない可能性。
- **過剰**: 契約にあって導出骨格にない。契約レベルの無認可混入の候補。
- **ずれ**: 両方にあるが食い違う。解釈のずれ。

差分の分類は意味判断であるため、LLM が行う。3.2 の盲目導出とは別の fresh context で行い、循環を減らす。

差分が多い場合は、3.2 の依頼引用欄の忠実性に疑いを持つ。転写監査表で引用欄と transcript を突き合わせて確認する。

## 4. 差分質問 → 三値回答

差分照合で検出された差分は、質問リストとしてユーザーに返す。自動で条項化しない（完備化禁止、trust-collapse §2）。

各差分について「この次元はどう扱うか」を聞く。回答選択肢は3つである。

| 回答     | 意味                 | 契約への反映                                       |
| -------- | -------------------- | -------------------------------------------------- |
| 制約条項 | 要件として追加       | 契約に条項を追加                                   |
| 許可条項 | 実装者の裁量に任せる | 契約の許可欄に記録。再発時は常設許可規則へ収穫候補 |
| 仮定     | 今は決めない         | 契約に保留と明記                                   |

これは intent-elicitation-discipline の三値形式である。

## 5. don't-care の収穫

don't-care は失敗ではない。したがって既存の失敗事例記録（`.opencode/local-failure-logs/`）の経路には乗らない。

don't-care の本体は沈黙である。興味のないことをユーザーが書き起こすことはない。したがって don't-care の検出は、欠如の検出の一種である。

沈黙は don't-care（裁量委任）と暗黙の選好（逆鱗）の両方を含み、書かれていないという事実からはどちらか判別できない。判別が生まれるのは、沈黙している次元が強制的に可視化され、人間が反応する瞬間——衝突点だけである。

要件レビュー手順には、衝突点が2つ組み込まれている。

- **衝突点1: 差分質問への三値回答**: 差分照合で検出された差分に対する質問に、ユーザーが「許可条項」と答えることが don't-care の顕示である。
- **衝突点2: 契約外変更への承認裁定**: 質問チャネルが目的適合で止まった先の次元は、実装者が勝手に埋め、レビュー・検収で契約外変更として浮く。ユーザーが承認したイベントは don't-care の顕示である。

don't-care の収集は、既存のセッション採掘（`extract-failure-patterns` 系）に「物質化された衝突の結果」の抽出観点を1つ追加する。抽出対象は、差分質問への三値回答（特に許可条項）と、契約外変更への承認裁定である。ユーザーの追加動作はゼロである。

採掘は複数セッションを横断して読むため、2作業以上での再発判定に構造的に向いている。

## 6. 常設許可規則

同じ同値類への don't-care 回答が2作業以上で再発したら、常設許可規則へ収穫する。

常設許可規則は「このパターンの次元は、過去の要件レビューで許可条項と答えられた」という記録である。収穫の流れは、採掘レポート → 人間承認 → 常設許可規則（追跡ファイル）である。失敗ログ → triage → codify と同じ既存経路を使う。

常設許可規則に収穫された次元は、以後の要件レビューで差分質問として発火しなくなる。発火条件は「答えの消費者を指名できる」であり、既に許可済みの類はこの条件を満たせないからである。

質問数は常設許可規則の蓄積とともに単調に減る設計である。減らない場合（成長曲線の発散）は trust-collapse §9 が経験的な問いとして挙げている。

## 7. 未解決の運用課題

- **転写監査表の効果測定**: 3.1 に記載した効果測定で測る。
- **差分照合の LLM 判定の較正**: 差分の分類は LLM が行う。較正は、回答の覆り率（intent-elicitation-discipline の計器）で測る。
- **発火条件の運用との接続**: 発火条件は契約締結直後で確定している。運用との接続（implementation が契約を読みに行く、レビュー中に契約を加味するか等）は別途設計する。
- **転写監査表の処遇列の自己監査限界**: 処遇列はフレーム相関が残り、自己監査では守れない。処遇列の誤りは盲目導出とユーザーの目視で受けるが、完全な解決ではない。

## 8. 残存リスク

- **依頼引用欄の忠実性**: 引用欄の忠実性は転写監査表が transcript と突き合わせて監査するが、監査を行うのも LLM であるため、完全ではない。
- **don't-care の収穫は採掘周期に依存する**: 採掘の習慣が絶えれば収穫も絶える。取り逃しの帰結は「同じ質問がもう一度来る」という有界の煩わしさであり、無音の損害ではない。
- **常設許可規則の立ち上がりは遅い**: 収穫条件（2作業以上 + 人間承認）が本来的に持つ遅さと同じ性質のものである。
