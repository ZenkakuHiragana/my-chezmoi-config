# sunset ablation 手順

## 目的

この手順は、モデル世代の更新や規則体系の整理時に、推論補償の規則がまだ失敗を防いでいるかを測る。対象規則を外した状態で凍結済み fixture または回帰シナリオを実行し、外しても通る規則を削除候補にする。

## 適用範囲

対象にするもの:

- prompt、agent、skill、command の本文にある推論補償の規則。
- 注意分散、指示追従、手順漏れ、自己申告、文脈取り違えを補う規則。
- fixture や回帰シナリオで失敗再発を観測できる規則。

対象にしないもの:

- ローカル仕様や path を配送する知識物流。
- 実行環境の真実へ届く ground truth 計器。
- ユーザーの意図を取得する相互作用プロトコル。
- 公式仕様、公開 API、ランタイム制約など、モデル世代では消えない外部基盤の制約。

対象外の層を削除候補にする場合は、先にその層が推論補償として振る舞っていることを示す fixture を作る。

## 入力

- 対象規則の path と該当範囲。
- 対象規則が防ぐ失敗の説明。
- 対応する fixture または回帰シナリオ。
- 現行規則ありでの基準結果。
- 対象規則だけを外す差分。

fixture が存在しない場合は、先に `opencode-prompt-dev/episode-to-fixture-procedure.md` に従って作る。fixture なしで規則を削除してはならない。

## 手順

1. **候補を列挙する。** 対象規則、置き場所、義務レベル、防ぐ失敗、対応する `problem_classes` を記録する。
2. **層を分類する。** 対象規則を、減価する層か、減価しない層かに分類する。分類は推測だけで決めず、規則が防ぐ失敗と fixture の観測可能な判定基準で説明する。
3. **基準結果を固定する。** 現行規則ありで fixture または回帰シナリオを実行し、判定基準ごとの pass / fail を記録する。既存の同条件結果を使う場合は、実行条件と版が一致することを確認する。
4. **除去差分を作る。** 対象規則だけを外す。別の規則、fixture、採点基準、配送形式を同時に変えてはならない。
5. **除去条件で実行する。** 現行規則ありの実行と同じ配送形式、同じ実行者種別、同じ採点手順で実行する。
6. **結果を比較する。** 対象規則を外して fail に戻るなら、その規則は残す。対象規則を外しても基準結果と同等に pass するなら、削除候補にする。
7. **削除または保留を決める。** 削除候補は、統治予算の回収量と残存リスクを記録してから削除する。結果が割れる場合は保留にし、追加標本または fixture の見直しへ戻す。

## 判定

| 結果                             | 判定           | 次の行動                                         |
| -------------------------------- | -------------- | ------------------------------------------------ |
| 除去条件で fail に戻る           | 規則は有効     | 規則を残し、fixture を回帰カナリアとして保持する |
| 除去条件でも pass する           | 規則は削除候補 | 規則削除と統治予算回収を検討する                 |
| 結果が割れる                     | 保留           | 標本を増やす、または fixture の判定基準を見直す  |
| fixture が現行規則ありでも不安定 | 計器不良       | 規則判断を止め、fixture を較正する               |

## 記録するもの

- 対象規則の path、範囲、義務レベル。
- 対応する fixture または回帰シナリオ。
- 現行規則ありの結果。
- 対象規則なしの結果。
- 採用した判定と理由。
- 削除した場合は、回収した統治予算と残す監視点。

## 完了条件

- 対象規則だけを変えた比較になっている。
- 実行者、採点者、配送形式、fixture の版が比較可能である。
- pass / fail を判定基準ごとに記録している。
- 削除、残存、保留のいずれかの判定を出している。
- 削除する場合、関連する参照と回帰カナリアの扱いを更新している。
