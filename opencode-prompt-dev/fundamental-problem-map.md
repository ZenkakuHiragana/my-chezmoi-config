# 根源課題の地図と検証中心の設計原理

## 概要

コーディングエージェントへの委任で繰り返し発生する失敗を、還元不能な5つの根源課題に分類し、改善介入の採否をその地図に基づいて判断するための設計原理を固定する。『コンテキスト管理と compaction の調査報告』が文脈の保持を、『根拠収集と知識検索の調査報告』が根拠の収集を扱ったのに対し、本報告は両者を含む全体の見取り図と、次に投資すべき中心領域を定める。

本報告は2部構成をとる。第1部は課題の構造と設計判断の理由を扱う概念整理で、主に人間の保守者に向ける。第2部は用語、設計原則、介入評価手順、実装課題を、実行エージェントが単体で設計判断に使える明示度で定める。

### 結論

- AI 委任の根源課題は5つに分類できる（P1〜P5）。うち P1（意図の伝達不能性）と P4（認識の自己申告不能性）は解消できず、管理だけができる。
- P2（検証の非対称性）が他のすべての課題の管理コストを決める。現行システムは検査が安い失敗クラスを大幅に減らし、残存失敗は検査が高いクラスに集中している。
- 次の中心投資は証明書搬送型の作業規律である。立証責任を実装者へ移し、検査を行単位の照合に変える。照合機構（resolver）は1つで済み、certificate schema はドメイン固有の安価なデータになる。
- レビューの独立性は監査制度として設計する。観点表（rubric）は成果物クラスが所有し、委譲する親は観点を追加してよいが削除・置換してはならない。レビュアーに与える文脈は成果物の消費者の文脈に一致させる。
- 進捗と停止は失敗分布で測る。各失敗記録に `problem_classes`（P1〜P5 の順序付き分類）を付与し、主因分布が P1/P4 に集中したらプロンプト体系側の改良を停止する。
- 規則はモデル世代ごとに ablation で再検証し、なくても通る規則は削除する。

## 背景

### 事実の系列

本報告の判断は、次の観測事実に基づく。

1. 2026-06-25 の blind A/B eval で、実装に移行するだけの十分な準備ができているかどうかを自己申告させる層 (以下、readiness 閂) の追加が gate-rate を改善しないという null 結果が得られた（baseline = commit 02229fd、3シナリオ、盲検実行者＋分離採点者）。プロンプト層の追記は、明白な挙動クラスでは限界効用がほぼゼロである。
2. `.opencode/work/fuck.jsonl` に記録されたセッション（deferred renderer 実装、2026-06-28 終了）では、仕様書に明記済みのレジスタ割り当てが実装で破られ、複数のレビューサブエージェントがそれを見逃した。同セッション終盤の `ID_*` 事件では、「単一使用の定数は局所化せよ」というレビュー指摘の一般論が、`ID_*` は Lua 正本 `inktype.lua` の `parameters[i]` が定める row schema であるという事実の確認なしに適用され、既存設計を破壊した（失敗記録 20260628-1706）。
3. 同セッションでユーザーは「contract を結ぶ指示はすでに出している。単にそれを守らない」と指摘した。規範がプロンプトに存在しても、実行時圧力の下で破られる。triage 20260628-1546 の C003（レビューの concern 分割が明示規則にもかかわらず発火しない）も同じ現象である。
4. `.opencode/work/complaint.md` は、改善された失敗クラス（公開情報を調べない虚偽、スタブによるテスト偽装）と残存する失敗クラス（意図の取り違え、成果物の検査不足）を区別して記録している。

### 残存失敗の共通構造

改善されたクラスと残存クラスを分けるものは、失敗の種類ではなく検査のコストである。引用の有無やスタブの存在は機械または一目で検査できる。レジスタ意味論や意図適合の検査は、再製作に近いコストがかかる。プロンプト体系は失敗を「検査が高いクラス」から「検査が安いクラス」へ移すことしかできず、現行システムは安いクラスを刈り尽くした地点に到達している。残る改善は、検査が高い領域を安くする投資に集中する。

## 第1部 概念整理

第1部は、コーディングエージェントへの委任で繰り返し発生する失敗の構造を整理し、改善介入の採否を判断するための概念地図を提供する。第2部の実装課題の理解を前提としない。

### 用語（第1部）

第1部で使う概念の定義を置く。第2部の実装用語とは独立に扱う。

- 認識地図: 主体が、対象の知見・事実・条件・制約をどう認識しているかを示す地図。territory 認識地図（実在するコード、仕様、環境、挙動）、user 認識地図（ユーザーの意図と暗黙知）、model 認識地図（モデルの推定と誤補完）、consumer/verifier 認識地図（成果物を読む・検査する側の文脈）の4種がある。
- certificate: 正否を決める主張のクラスについて、実装者が作成する照合可能な表。各行は出典パス、範囲、引用を持つ。
- resolver: certificate の行を機械照合する機構。出典パスの実在、引用と実体の一致、識別子の出現を検査する。ドメイン非依存。
- 衝突: 成果物、主張、前提が、それを反証しうる対象（機械照合、コンパイル、実行、消費者、現実）と突き合わされる事象。
- 衝突の予定: 衝突が回避不能な形で工程に組み込まれていること。resolver の照合、merge を止める quiz、意図の持ち主への意図ミラー提示がこれにあたる。
- 捏造コスト反転: 成果物を正直に作るコストが、偽装して作るコストを下回るように設計された状態。
- 命題的な穴: 真偽を持つが未確認の前提。証明責務として列挙できる。
- 認識的な穴: 見れば判別できるが言語化されていない基準。命題として列挙できない。
- 忠実度: 検査結果が territory の実態と一致する度合い。
- フレーム汚染: 前段の要約や問題設定に混入した誤りが、後段で疑われずに増幅される現象。
- unknowns: 認識地図と territory の差。Known Knowns、Known Unknowns、Unknown Knowns、Unknown Unknowns の4種に分類される。
- 一貫性圧: 生成器が文脈全体と整合する出力を優先する性質。未解決の unknown を文脈に整合する値で無音に埋める機序の源。

### 根本課題と支配原理

根本課題は、人間・モデル・現実のズレを、安く衝突させられる表現へ変換することである。

ズレは複数の認識地図の間で開く。territory 認識地図は実在するコード、仕様、環境、挙動である。user 認識地図はユーザーが意図していることと暗黙に知っていることである。model 認識地図はモデルが推定していることと誤って補完していることである。さらに、成果物を読む、実行する、検査する側が持つ consumer/verifier 認識地図がある。委任の品質問題は、これらの認識地図の差をどれだけ安く検出できるかという問題になる。

支配原理は後述の根源課題 P2（検証の非対称性）である。委任は「検査が製作より安い」場合だけ経済的に成立する。どの根源課題を攻めても、最終的に「検査を安くする」に還元される。P2 は課題の1つであると同時に、他の課題の管理コストを決める貨幣単位である。

### 中心法則: 衝突の予定

成果物は、産出時か消費時に回避不能な「衝突」が予定されているときに限り、品質装置として機能する。
衝突が予定されていない成果物は、実行時圧力の下で儀式化する。義務の記述が prose に存在するかどうかは、この判定を変えない。

ここでいう衝突とは、成果物、主張、前提が、それを反証しうる対象（機械照合、コンパイル、実行、消費者、現実）と突き合わされる事象である。
衝突の予定とは、衝突が回避不能な形で工程に組み込まれていることである。

この法則は、効いた装置と効かなかった装置の対比から導かれる。証明書と照合機構の組は、産出時に引用と出典パスの機械照合が予定されており、出典欄を埋める義務が正本の参照を強制して誤った主張が作成時点で成立しなくなる。意図ミラー、quiz、prototype は、消費時に意図や理解や好みを唯一持つ人間が安く反証できる経路を予定し、言語化できない基準に届く。失敗ログから fixtures への還流は、事後の高価な衝突を安い計器の較正に使う経路を予定する。一方、readiness 閂は同じ文脈から接地なしに生成でき、watchdog の状態自己申告は省略も偽装もでき、prose の契約義務や concern 分割義務は実行時圧力の下で破られる。後者の3つは衝突が予定されておらず、観測でも効かなかった（参考にした失敗事例の節を参照）。

産出時の衝突は、成果物を作る行為そのものが territory との接触を強制するときに成立する。証明書の出典欄は、埋めるために正本を開くことを要求し、開いた時点で誤った主張が死ぬ。照合機構の抜き取り照合は、偽装の発覚確率を上げて偽装の期待コストを引き上げる。両者が揃うと、正直に作る方が偽装より安いという捏造コスト反転が成立する。この構造はシグナリング理論（Spence 1973）と同型である。儀式化の機序は Goodhart の法則（指標が目標になると指標は指標でなくなる）の実例である。

消費時の衝突は、反証手段を持つ消費者に必ず届くなら成立する。意図ミラーの消費者は意図の持ち主であり、解釈のずれを一目で反証できる。quiz の消費者は自分の理解度を答案の正誤で反証される人間である。prototype の消費者は「見れば分かる」基準の持ち主である。反証手段を持たない消費者への送達は衝突にならず、成果物の価値は送達ではなく反証可能な受領で決まる。

介入とレビューの採否では、次を順に問う。第1問として「この成果物の衝突はいつ、誰と予定されているか。正直に作る方が偽装より安いか」を先に立てる。残りは第2部の介入評価手順の要素である。

1. この成果物の衝突はいつ、誰と予定されているか。正直に作る方が偽装より安いか。
2. 対応する `problem_classes` を特定する。
3. 検査コストを下げるかを判定する。
4. artifact / resolver として実装できるかを prose より先に検討する。
5. 検証方法を定義する。
6. 統治予算の支払いを決める。

第1問に答えられない介入は、宛先の課題分類が何であれ、null 結果の系譜に加わる見込みが高い。

### 還元不能な5つの根源課題

#### P1: 意図の伝達不能性

意図は依頼者の内部にあり、言語はそれを過少決定する。契約理論はこれを不完備契約として定式化しており（Grossman & Hart 1986）、あらゆる契約は不完備である。仕様を分解して完備化する試みは、分解の深さに上限（レビュー負担、注意の限界）があるため必ず途中で止まる。

管理戦略: 完備化を目標にしない。主張がチェック可能になった粒度で分解を止め、残余は相互作用で埋める。着手前に解釈と補完判断を明示する（意図ミラー）、決定が要る点だけ質問する（`grill-me`）、凍結した前提を契約として可視化する（`Requirement contract`）。

#### P2: 検証の非対称性

委任は「検査が製作より安い」場合だけ経済的に成立する。ドメインによっては検査コストが製作コストに近く（GMOD のレンダリング挙動が典型）、ground truth が実行環境の中にしかない。

管理戦略: 検査を安くする投資に集中する。証明書規律（後述）、汎用 resolver、ドメイン償却型の計器（実機プローブ、golden image）、検査しやすさに向けたプロダクト側の設計（エンジン接触面を薄いアダプタに隔離し、ロジックをテスト可能な側へ寄せる）。

#### P3: 注意と文脈の希少性

コンテキストは有限の奪い合い資源であり、規則は作業内容と同じ予算から支払われる（統治予算）。規則を増やすと指示追従全体が劣化するため、限界的な1規則の期待値はある点を超えると負になる。位置による参照劣化も加わる。機序の詳細は『コンテキスト管理と compaction の調査報告』が扱う。

管理戦略: プロンプト階層規律、リファクタトリガーによる削除、義務の prose から artifact / resolver への移設、compaction の置き換え。

#### P4: 認識の自己申告不能性

モデルは生成と知識の境界を内部に持たず、知らないことを申告できない。自信の較正は事実 QA 単発では部分的に機能しても（Kadavath et al. 2022）、長いエージェントループとロングテールのドメイン知識で崩れる。自己申告に依存する監視は、必要なときに限って機能しない。watchdog の状態申告の不成立、readiness 閂の null 結果、C003 は、同じ壁の別の面である。

管理戦略: 内観に問わない。根拠レコードを伴わない主張を既定で「仮定」に分類し（default-deny）、検証は fresh context の分離検証者と機械照合に置き、現実との衝突で誤りを露出させる。

#### P5: 蓄積の不在

セッションは記憶を持たない使い捨ての労働者であり、学習はモデル内部に蓄積しない。改善はすべて外部成果物に蓄積するしかない。また、1事例からの規則化は分散が大きく、局所最適化された規則の堆積を生む。

管理戦略: 学習の単位を規則ではなく計器（回帰シナリオ、resolver、fixtures）にする。失敗記録は fixtures の在庫である。wiki が知識の複利であるのと同様に、計器は検証の複利である。

### 課題間の関係

P1 と P4 は解消できない。P3 と P5 は工学的に管理できる。そして P2 が全体のボトルネックになる。他の課題の管理装置がすべて検査コストに依存するからである。P1 の管理（意図確認と契約）は検証が安くないと回らない。P4 の代替（証明書と分離検証）は照合が安いことが前提になる。P5 の蓄積先（計器）は検証装置そのものである。どの課題を攻めても、最終的に「検査を安くする」に還元される。

### P1/P4 の残余: 命題的な穴と認識的な穴

P1 と P4 は解消できず管理のみができる。この管理対象は内部構造を持ち、命題的な穴と認識的な穴の2種に分かれる。処方が異なるため、穴の種類を判別してから管理装置を選ぶ。

命題的な穴は、真偽を持つが未確認の前提である。API の再入可能性、識別子のスコープ、仕様の一部がこれにあたる。証明責務として列挙でき、証明書、出典照合、default-deny で扱える。

認識的な穴は、見れば判別できるが言語化されていない基準である。視覚設計の好み、文体の許容範囲、「これは違う」としか言えない要件がこれにあたる。命題ではないため、責務としての列挙も質問による抽出も原理的に届かない。届くのは prototype、reference、実物比較というプローブであり、消費時の衝突の特殊形である。

2分類の取り違えには実害がある。認識的な穴に主張表を当てると、表は埋まるが基準は露出しない。命題的な穴にプローブを当てると、確認1回で済む事実に試作コストを払う。

### unknowns と一貫性圧

認識地図と territory の差は unknowns として現れる。unknowns は4種に分類される。Known Knowns は提示済みの前提、Known Unknowns は未決定だが自覚されている前提、Unknown Knowns は見れば分かるが言語化されていない基準、Unknown Unknowns は気づいていない盲点である。ズレが顕在化するのは、主に Unknown Knowns と Unknown Unknowns が残ったまま作業が進むときである。

強いモデルほど、unknowns が静かに致命傷になる。旧世代のモデルでは実行誤差が支配的であり、unknowns を雑に埋めても実行が壊れるため仕様誤差は目立たなかった。実行誤差を潰したモデルでは、残差の支配項が仕様誤差に移る。そして一貫性圧の強い生成は、未解決の unknown を自由度として扱い、文脈に最も整合する値で無音に埋める。埋めた事実は申告されず、埋めた値は周囲と整合するため、局所検査で浮かない。これが、基幹条件を飛ばしてもうまく補完して人災が大規模化する機序である。

この機序から、強いモデルは unknowns の露出手順でもあり、同時に unknowns の増幅器でもある。問いを再定義し、構造を圧縮し、散らばった前提を統合する能力は、誤った前提を含んだまま美しい構造にまとめる能力でもある。したがって強いモデルの能力は、unknowns を減らす解決策ではなく、衝突が予定された装置と組み合わせて初めて価値を生む。

unknowns の種類に応じて、届く管理装置が異なる。Known Unknowns には質問と決定リストが届く。Unknown Knowns には prototype と reference というプローブが届く。Unknown Unknowns には blindspot pass（盲点を列挙するのではなく、一部を Known Unknowns へ昇格させる手続き）と、現実との衝突が届く。未知の未知は内観ではなく衝突でしか見つからない。

問題設定そのものが間違っていることに気づく経路も、unknowns 管理の一部である。局所修正が累積しているとき、ルーティングが効かないとき、同じ失敗が再発するときには、現在の問題設定の中で前提を埋める作業（context-clarification）ではなく、問題設定そのものを疑う上位の問い（reframing）が要る。

### 証明書搬送型の作業

#### 原理

Proof-carrying code（Necula 1997）は、信頼できない生成物の監査を「生成者が証明を添付し、小さく信頼できる検査器が証明だけを照合する」形に反転させた。根底には、証明の発見は高価だが検証は安価だという計算の非対称性がある。これを作業一般に拡張する。

- 実装者は、正否を決める主張のクラスごとに certificate を作成する。certificate は {主張, 出典パス, 範囲, 引用} を列に持つ表である。
- 検査側は表を行単位で照合する。開放的な発見（レビュー）が、閉じた照合（チェック）に変わる。

#### 効果の機序

第一に、検査コストが下がる。「このシェーダーは正しいか」という開放的な問いが、「この表の各行の引用は実在し、主張と対応しているか」という照合に変わる。

第二に、作成の時点で多くの誤りが主張不能になる。証明書の出典欄を埋める義務は、実装者が正本を開くことを強制する。正本と矛盾する主張は、出典欄を埋める行為の中で成立しなくなる。立証責任が、誤りを事後発見ではなく事前抑止する。

第三に、レビューの負担構造が変わる。レビュアーは発見の義務（高コストで、見逃しが不可視）から解放され、照合の義務（低コストで、見逃しが可視）を負う。

#### 機構と schema の分離

certificate schema はドメイン固有だが、照合機構は共通化できる。certificate の1行を受け取り、(1) 出典パスの実在、(2) 引用と実体の一致、(3) 主張に含まれる識別子の引用範囲内での出現、を照合する resolver は、ドメイン非依存の機構として1回だけ実装すればよい。以後、新しいドメインの追加コストは schema という表定義の作成だけになる。ドメインごとに検査器を自作する必要はない。

#### 限界

certificate には、それらしい行で表を埋める偽装のリスクがある。対策は2つで、(1) schema の列を機械照合可能な形式（引用とパス）に限定する、(2) resolver による抜き取り照合を常設する。また、引用が主張を本当に支持するかという関連性の判定は機械照合の外にあり、fresh context の分離検証者に狭い問い（この引用はこの主張を支持するか）として渡す。この層には絶対の保証がなく、床を上げる装置であって天井を消す装置ではない。

### 検査の独立性

#### 相関する見逃し

多重チェックによる見逃し確率の低減は、見逃しの独立性を前提とする。独立に開発されたプログラム同士でも難しい入力で相関して失敗することが実証されており（Knight & Leveson 1986）、同一モデル・同一文脈・同一フレーミングのレビュアー同士の相関はさらに強い。相関した検査層を重ねても低減は頭打ちになる。買うべきは層の枚数ではなく脱相関である。

検査は形式照合と意味照合の2種に分かれる。形式照合はパス、行番号、引用、識別子を機械で照合し、意味照合は根拠が主張を本当に支持するかを LLM または人間が判断する。後者では独立性が品質を支配する。

脱相関には文脈の脱相関と事前分布の脱相関の2種があり、コストと効果が異なる。文脈の脱相関は、fresh context の検証者、フレーミングの変更、文脈スライスの変更で買える。安いが、届く範囲は文脈由来の相関までである。同族モデルは fresh context でも訓練由来の事前分布を共有する。事前分布の脱相関は、モデル族を跨ぐ、機械照合へ落とす、人間が見る、territory と直接接触させる、のいずれかでしか買えない。一般常識とドメイン真実が衝突する領域では事前分布の脱相関が要る。検査設計では、どちらの脱相関を買っているかを明示する。

なお、検査層の多重化には責任の拡散という既知の病理が伴う。各層の担当観点を明示的に分離しない多重化は、層あたりの注意を下げる。`code-review` の concern 分割義務は、並列化の手段である以前に、この病理への処方（責任の非重複割当）である。

#### 監査の独立性

委譲する親エージェントは、自分の成果物の監査を発注する被監査者にあたる。親がレビューの範囲や焦点を記述すると、親の盲点がレビュアーに継承される。観測された実例では、文書レビューの委譲で親が「前回の修正で生じた不整合が解消されているか」という焦点を与えた結果、レビュアーは版差分の確認に最適化し、本文完結性（文書は本文のみから理解できる）という常設規範の違反を見逃した。

処方は監査制度と同じ形をとる。

- rubric は成果物クラスに紐づき、レビュー skill が所有する。版管理された固定リストとして、委譲のたびに全項を適用する。
- 親は観点を追加してよい。削除・置換・縮小してはならない。
- 委譲文（assignment packet）にセッション経緯を書いてはならない。渡すのは成果物、rubric 参照、追加観点のみとする。この制約は packet という成果物への静的検査で機械的に確認できる。

#### 消費者文脈の原理

上記の実例で破られた規範「本文完結性」は、文脈を持たない読者に対して定義される性質である。文脈を持つレビュアーには、この違反は構造的に見えない。一般化すると、レビュアーに与える文脈は、生産者の文脈ではなく、その成果物の消費者の文脈に一致させる。文脈ゼロの読者に読まれる文書は文脈ゼロのレビュアーに検査させ、コンパイラと将来の保守者に消費されるコードはその視点で検査させる。レビュアーへ渡す文脈は委譲の副産物ではなく、成果物クラスごとの設計パラメータである。

### 衝突による unknown unknowns の発見

抜けている前提は、定義により列挙できず、内観でも見つからない。発見の唯一の経路は現実との衝突（実行、テスト、プローブ、照合、外部レビュー）である。したがって品質向上は、衝突の頻度を上げ、1衝突あたりのコストを下げることに帰着する。

衝突にはコストの階段がある。

意図ミラー < certificate 照合 < resolver < 分離レビュー < コンパイル < 実機プローブ < 利用者の手戻り

システムの仕事は、矛盾の発見をこの階段の左へ移すことである。ただし階段の左端（意図ミラー、証明書照合）は territory そのものではなく、territory の写像（仕様書、正本、証明書の引用元）と衝突している。写像が古いか誤っているとき、安い衝突は偽陰性を返す。忠実度を供給するのは右端の高価な衝突（コンパイル、実機プローブ、利用者の手戻り）だけである。これらは territory と直接衝突し、写像の誤りごと露出させる。

したがって右端の衝突は削減対象であると同時に、定期的に支払う較正コストである。失敗ログから fixtures を経て計器の較正へ至るパイプラインは、右端で1回だけ支払った忠実度を左端の安い計器へ輸送する装置である。右端をゼロへ最適化すると左端の較正が止まり、安い衝突の忠実度が沈黙のうちに劣化する。衝突は記録されなければ複利にならない。失敗記録を fixtures として回帰シナリオと resolver の較正に使う経路が、衝突の複利化装置であると同時に、忠実度の還流経路でもある。

### 会計と停止規則

改善の終わりが定義されていないと、どこまで進んでも不足に見える。停止規則を失敗分布で定義する。

- 各失敗記録に `problem_classes`（P1〜P5 の順序付き分類）を付与する。
- 主因分布が P2 / P3 / P5 に偏っているうちは、工学的な改善余地が残っている。
- 主因分布が P1 / P4 に集中したら、prose 規則への追加投資を停止し、残存失敗は相互作用と運用で吸収する。

この停止規則には自己言及的な穴がある。`problem_classes` の分類は失敗発生後の観測に基づく判断であり、判断者がモデルであれ人間であれ、真因への直接アクセスを持たない。分類という行為自体が P4 の制約下にある。さらに、一貫性圧の強いモデルは、文脈に情報が存在したのに上書きされた失敗（P3/P4 起源）や、検査が高くて無音の補完を捕まえ損ねた失敗（P2 起源）を、「意図が伝わっていなかった」という P1 の顔で記帳しうる。したがって主因分布の P1/P4 集中は、「工学的余地が尽きた」と「モデルが失敗のクラスを偽装するのが上手くなった」を区別しない。

この区別をつけるため、P1 判定には観測条件を課す。意図は実際に取得不能だったのか、それとも文脈に存在したのに一貫性圧で上書きされたのか。後者は P1 ではなく P3 または P4 である。また、停止するのは prose 規則への追加投資であり、プローブ経路、計器、相互作用プロトコルへの投資ではない。P1/P4 の管理装置はそれ自体が改良対象であり続ける。

進捗指標は失敗件数の絶対値ではなく、分布の左方移動（P2/P3/P5 比率の低下）と、矛盾1件あたりの発見コストの低下で測る。リポジトリの目的が「モデルの問題解決性能を+1世代向上させる」である以上、最終指標は「所与の品質水準を達成できる最弱のモデル」であり、blind-eval ハーネスで測定できる。

#### 消費財と資本財の帳簿分離

このリポジトリの産物は消費財と資本財に分かれ、会計が異なる。消費財はタスクの出力であり、そのタスクの成功で測る。資本財は計器（resolver、`check_vocabulary.py`、blind-eval ハーネス）、schema、fixtures、地図文書であり、将来の衝突単価をどれだけ下げるかで測る。資本財の価値は産出時点では確定せず、held-out の回帰評価に接続されて初めて測定可能になる。この接続を持たない資本財の増設は、局所過適合した規則の堆積と同じ経路をたどる。

sunset ablation は資本財の減価償却テストであり、モデル世代の進歩で価値を失った規則を除却する。「所与の品質水準を達成できる最弱のモデル」という最終指標は、資本ストックの総量測定である。失敗記録は fixtures の在庫、すなわち仕掛品である。

### 文脈と注意の物流

コンテキストは有限の奪い合い資源である（P3）。コンテキストを増やすと検証能力が壊れる問題が中心であり、単なる量の不足ではない。検証に必要な差分だけが、発火点で、消費者文脈に合う形で現れることが求められる。

文脈の崩壊は複数の機序で起きる。入力の中央部が忘れられる現象、長期コンテキストでの指示劣化、エージェントの目的の漂動、指示の漂動がある。これらに対する現行の処方は、履歴を圧縮して保持する仕組み（historian + decay rendering + emergency drop）と、作業前に必要な文脈を構造化ゲートで客観化する仕組み（コンテキスト収集規則）の2層構造である。前者は保持価値を事後判定し、後者は必要性を事前構造化する。この2層は補完関係にあり、片方だけでは不十分である。

文脈を積む操作はできても取り除く操作がない問題は完全には解けず、現在は部分的な対応に留まる。また、文脈の物流と同じ希少性制約が人間側にもあり、こちらは増設できない。成果物（証明書、record、報告）が増えるほど、人間の消費側で形式的な承認が起きる。証明書時代の人間の役割は全読ではなく、抜き取り照合と計器への信頼配分である。

### 要約連鎖とフレーム汚染

複数モデル、複数セッションにまたがる分析ワークフローには、単一セッションにはない伝播経路がある。一貫性圧の強いモデルは、確立した主張への圧力には反論する一方、自信をもって提示されたフレームには従順に乗る。前段のモデルが書いた要約は、後段のモデルにとって自信をもって提示されたフレームであり、要約に混入した誤りは疑われずに一貫的に増幅される。

要約は表現変換の一種であり、変換のたびに何が保存され何が落ちるかが変わる。要約の連鎖は、この損失を検査なしに累積させる構造を持つ。対策は3つで、いずれも衝突の予定を要約境界に置く形をとる。第一に、要約には出典パスを残し、主張の照合先を切らない。第二に、局所修正の累積時、同じ失敗の再発時、分析の段の切り替わりで、問題設定そのものを疑う問いを発火させる。第三に、要約の消費者に、要約ではなく一次資料から芯を再構成させ、要約との差分を検査する。

### 陳腐化の管理

プロンプト体系の各層は、モデル世代の進歩に対する耐久性が異なる。

- 減価する層: 推論の欠損を補償する規則（指示追従の drift 対策、注意の分散対策、詳細な手順規則）。モデルの向上とともに価値が減り、統治予算だけを消費するようになる。
- 減価しない層: ドメイン知識の物流（どのモデルの重みにも入らないローカル仕様）、ground truth 計器（実行環境の中の真実へ届く装置）、意図の相互作用プロトコル（意図は定義上プライベートであり続ける）。

層の帰属は推測で決めず、測定で決める（sunset ablation）。モデル世代の更新時に、回帰シナリオパックを対象規則を除去した状態で実行し、除去しても通る規則を削除する。規則は世代ごとに再検証される仮説として扱う。この運用のため、減価する層に属する規則は分離可能な include に隔離する。

### 理論的裏付け

- 正当化の成果物化: 知識の古典的定義（正当化された真なる信念）のうち、外部から監査できるのは正当化だけである。真理は直接見えず、信念は内部状態として近づけない。証明書規律は、監査可能な唯一の成分を成果物として要求する設計である。
- 蝶番命題: あらゆる探究は、それ自体は疑われない前提の上で回る（Wittgenstein『確実性の問題』）。全前提の明示化は原理的に不可能であり、選べるのは「どれを固定し、どれを監査可能に保つか」だけである。`Requirement contract` は固定した前提の明示化にあたる。
- 反証主義: 仮説は検証ではなく反証によってしか淘汰されない（Popper）。内観で前提の抜けを探すエージェントは抜けを発見せず、行動して現実に裏切られるエージェントだけが発見する。「短時間で試せる主張は、主張せずに試す」はこの帰結である。
- シグナリング理論: 信号が信頼できるのは、偽の信号を出すコストが正直な信号を出すコストを上回るときに限られる（Spence 1973）。証明書の出典欄を埋める義務が正本の参照を強制する構造は、捏造コストを正直コスト以上に上げることで信号の信頼性を担保する。
- 指標の目標化: 指標が目標になると指標は指標でなくなる（Goodhart）。prose の義務は「義務を満たしたように見える出力」という指標を最適化させ、衝突が予定された成果物だけがこの最適化を無効にする。

### 参考にした失敗事例

中心法則と各所の議論は、次の観測事実に基づく。各事例の詳細は出典欄の一次記録を参照。

- readiness 閂の null（2026-06-25 blind A/B eval、baseline = commit 02229fd）: コンテキスト収集規則の上に置いた readiness 判定ゲートが、gate-rate を改善しなかった。同じ文脈から接地なしに生成できるゲートは、衝突を予定しない。
- watchdog の状態自己申告の不成立: 状態 YAML を自己申告させる監視機構が、申告を出さないモデルに対して成立しなかった。申告は省略も偽装もできるため、衝突が予定されない。
- C003（明示規則が発火しない）: レビューの concern 分割を prose で義務化しても、実行時圧力の下で破られた。prose の義務は指標化されて儀式化する。
- 識別子のスコープ取り違え（2026-06-28）: ドメイン正本の row schema である識別子を、一般論の「単一使用の定数は局所化せよ」というレビュー指摘の適用対象と取り違え、既存設計を破壊した。証明書の出典欄を埋める義務があれば、正本を開くことを強制され、事前抑止された事例である。

## 第2部 設計原理と実装課題

第2部は、このリポジトリで設計・実装を行うエージェントが単体で使える明示度で書く。第1部の理解を前提にしない。

### 用語

ここで導入する語を prompt 本文へ持ち込む場合は、`opencode-prompt-dev/english-token-allowlist.md` へ登録する。

- `certificate`: 正否を決める主張のクラスについて、実装者が作成する照合可能な表。各行は {主張, 出典パス, 範囲, 引用} を持つ。
- `certificate schema`: certificate の列定義。ドメインごとに定義する。
- `resolver`: certificate の行を機械照合するスクリプト。出典パスの実在、引用の一致、識別子の出現を検査する。ドメイン非依存。
- `rubric`: 成果物クラスに紐づく常設のレビュー観点表。レビュー skill が所有し、版管理する。
- 消費者文脈: 成果物を実際に消費する読者・システムが持つ文脈。レビュアーに与える文脈はこれに一致させる。
- 意図ミラー: 実作業の開始前に、依頼の解釈、指定されなかった判断点、その補完内容を依頼者へ提示する行為。`Readiness record` の仮定欄を提示形式として使う。
- default-deny: 根拠レコードを伴わない主張を、既定で「仮定」に分類する規約。仮定は成果物に明示列挙する。
- 衝突コスト階段: 矛盾が発見される段階のコスト序列。意図ミラー < certificate 照合 < resolver < 分離レビュー < コンパイル < 実機プローブ < 利用者の手戻り。
- `problem_classes`: 失敗記録に付与する根源課題の順序付き分類。先頭を主因、2番目以降を副因とする。値は `P1`（意図の伝達不能性）/ `P2`（検証の非対称性）/ `P3`（注意と文脈の希少性）/ `P4`（認識の自己申告不能性）/ `P5`（蓄積の不在）/ `unknown`。判断不能なときは `unknown` のみを置く。分類は失敗時の観測事実に基づける。たとえば P3 は、必要な情報、規則、または文脈が存在していたにもかかわらず、量、配置、圧縮、規則過多、注意分散によって扱われなかった場合に付ける。
- 統治予算: プロンプトに置ける規則の総量制約。規則は作業内容と同じコンテキスト資源から支払われる。
- sunset ablation: モデル世代更新時に、規則を除去した状態で回帰シナリオを実行し、除去しても通る規則を削除する手順。

### 設計原則

1. 改善介入は、検査コストを下げるかどうかで採否を決める。検査コストを下げないプロンプト追記を、検証済みの失敗根拠なしに追加してはならない。
2. 正否を決める主張には certificate を作成する。certificate には resolver を用意する。resolver は fixtures で較正する。失敗記録を fixtures の第一の供給源とする。
3. 規則の追加は統治予算からの引き落としとして扱う。追加時は、同量の削除、または prose から artifact / resolver への移設を伴わせる。
4. 仕様の完備化を分解の目標にしてはならない。主張がチェック可能になった粒度で分解を止め、残余は意図ミラー、質問、停止で埋める。
5. 学習は規則ではなく計器に蓄積する。1事例から共有規則を作ってはならない。1事例からは回帰シナリオを作る。
6. rubric は成果物クラスに紐づけ、レビュー skill が所有する。委譲する親は観点を追加してよい。削除・置換・縮小してはならない。
7. レビュー委譲の assignment packet にセッション経緯を書いてはならない。成果物、rubric 参照、追加観点のみを渡す。
8. レビュアーへ与える文脈は、成果物の消費者文脈に一致させる。
9. 減価する層（推論補償の規則）は分離可能な include に隔離し、sunset ablation の対象として管理する。知識物流、ground truth 計器、意図相互作用の層に優先投資する。

### 介入評価手順

新しい改善介入の案を評価するときは、次を順に確認する。

1. 対応する `problem_classes` を特定する。P1 / P4 が主因または重要な副因にある場合、その介入は課題を解消しない。管理装置（相互作用、default-deny、分離検証）であることを確認する。
2. 検査コストを下げるかを判定する。下げない prose 追記は、blind eval での検証なしに採用してはならない。
3. artifact / resolver として実装できるかを、prose より先に検討する（`prompt-principles.md` の介入種別の選択方針に従う）。
4. 検証方法（fixtures または回帰シナリオ）を定義する。定義できない介入は採用しない。
5. 統治予算の支払いを決める。規則を追加する場合、何を削除または移設するかを明示する。

### 実装課題

優先順に並べる。各課題は独立に着手できる。

#### B1: 汎用 resolver

- 目的: certificate 行の機械照合。P2 に対応する。
- 内容: {主張, 出典パス, 範囲, 引用} を受け取り、(1) パスの実在、(2) 引用と実体の一致、(3) 主張に含まれる識別子の引用範囲内での出現、を検査するスクリプト。LLM を使わない。`opencode-prompt-dev/check_vocabulary.py` と同じ位置づけの常設計器とする。
- 受け入れ条件: 既知の正例 fixtures を全通過し、既知の捏造例 fixtures を全検出する。
- 検証方法: fixtures を失敗記録から作成する。`ID_*` 事件（失敗記録 20260628-1706）の反実仮想（誤った主張と実在する出典の組で照合が失敗する例）を含める。

#### B2: certificate schema の初期整備

- 目的: 出血量の多い主張クラスから証明書規律を開始する。P2 に対応する。
- 内容: 次の2クラスの schema を定義する。
  - register / ABI / schema 所有権表: material parameter、shader stage、register 範囲、writer、reader、根拠パス。triage 20260628-1546 の P002、および fuck.jsonl 終盤の shader contract 表・data layout ownership 表の項目定義を正本とする。
  - 仕様適合表: 実装上の決定と、それを許す仕様書の行の対応。
- 受け入れ条件: 各 schema の全列が、resolver で照合可能な形式（パスと引用）で構成される。
- 検証方法: `empirical-prompt-tuning` の枠組みで、schema 義務の有無による blind A/B を行う。triage 20260628-1546 の P002 / P004 シナリオを流用する。

#### B3: assignment packet の独立性制約

- 目的: 親エージェントの盲点継承の遮断。P4 と監査の独立性に対応する。
- 内容: レビュー委譲の assignment packet に形式制約を定める。セッション経緯への言及の禁止、渡せる内容の限定（成果物、rubric 参照、追加観点リスト）。文書クラスのレビューではレビュアーに成果物単体を与える（消費者文脈）。rubric を `code-review` / `japanese-doc-review` の skill 側へ固定し、親が縮小できないことを明記する。
- 受け入れ条件: 経緯言及を含む packet が静的検査で検出できる。
- 検証方法: 本文完結性違反（版差分への言及が本文に混入した文書）を、経緯つき委譲と成果物単体委譲の両方でレビューさせ、検出率を比較する blind eval。

#### B4: 失敗記録への problem_classes 付与

- 目的: 会計と停止規則の実装。P5 に対応する。
- 内容: `dot_config/opencode/agents/report-failure.md` の失敗報告 template frontmatter に `problem_classes: []` を持たせる。値は `P1` / `P2` / `P3` / `P4` / `P5` / `unknown` の順序付き配列とし、先頭を主因、2番目以降を副因とする。P1〜P5 の分類基準を失敗記録と triage の手順に明記する。triage で主因分布と副因込み分布を集計する。既存の失敗記録には triage 時に遡及付与する。
- 受け入れ条件: 新規失敗記録に `problem_classes` が付与され、triage レポートに主因分布と副因込み分布が含まれる。
- 検証方法: 既存失敗記録から10件を抽出し、付与基準による判定の一致を確認する。

#### B5: sunset ablation 手順

- 目的: 陳腐化した規則の削除。P3 に対応する。
- 内容: モデル世代更新時の手順書を作る。対象規則の一覧化（減価する層のタグ付け）、回帰パックの規則除去実行、除去しても通る規則の削除。
- 受け入れ条件: 手順書が存在し、1世代分の実測が行われている。
- 検証方法: 現行モデルで1規則を対象に試行し、手順が回ることを確認する。

#### B6: 不変条件クラスの語彙収穫

- 目的: rubric と certificate schema の命名整備。P2 / P5 に対応する。
- 内容: fuck.jsonl 終盤で提案された停止語彙（schema / ABI / 墓標 / 根拠 / 鵜呑み / 混ぜるな）を、不変条件クラスの名前として rubric 項目と certificate schema 名へ転用する。
- 受け入れ条件: rubric 項目名が失敗記録の pattern_tags と対応づく。
- 検証方法: B3 の rubric 整備と同時に確認する。

### 適用範囲と限界

- 本報告の装置はすべて床を上げる。捏造を検出可能にし、仮定を可視化するが、モデルの設計判断そのものを賢くはしない。開いた設計判断の質は `grill-me` と人間の判断に残る。
- certificate 義務はトークンとターンのコストを増やす。義務の発火点は commitment point（実装判断の確定、仕様と矛盾する主張、外部 API の書き下し）に限定し、コスト増は実測する。
- resolver の照合は形式まで届く。引用が主張を支持するかの関連性判定は分離検証者に残り、そこに絶対の保証はない。
