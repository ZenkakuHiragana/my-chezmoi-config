## 1. 無知の知は内観では手に入らない — 問いを変換する

哲学に踏み込む前に、機構の話をする。LLM の生成は「知識の参照」ではなく補間だ。内部に「知っている領域」と「補間している領域」の境界線は存在しない。よく裏付けられた補間を我々が「知識」と呼んでいるだけで、confabulation は知識の上に乗ったバグではなく、生成そのものの性質だ。

較正の研究（Kadavath et al. 2022, "Language Models (Mostly) Know What They Know" など）では、事実 QA 単発ならモデルの自信申告はそこそこ較正されている。だが較正が崩れるのは正確に「必要な場面」だ: ロングテールのドメイン知識、長いエージェントループ、RLHF 後の自信過剰。つまり「自信がないときは言え」という指示は、watchdog と同型の構造矛盾を抱えている。**自己申告できるモデルは自己申告が要らず、必要なモデルは申告できない。** 君はこのパラドックスをすでに一度踏んでいる。

だから問いを変える。

- 「モデルは知っているか」 — 検証不能。内部状態にアクセスできない。
- 「モデルは根拠を今ここで提示できるか」 — 検証可能。成果物だから。

**unknown unknowns は検出できないが、unwarranted assertions（根拠なき断定）は検出できる。** この変換が本体だ。根拠レコード（file:line＋引用、コマンド＋出力、URL＋引用）を伴わない主張は自動的に「仮定」に分類され、仮定は成果物に明示列挙される — セキュリティの default-deny と同じ規約を主張に適用する。evidence-retrieval レポートの4状態分類（absent かつ徹底探索済、を情報として扱う）は、検索層でこれと同じ手をすでに打っている。主張層に同じ手を伸ばすだけだ。

## 2. 「全主張にファクトチェック」が効く条件

素朴にプロンプトへ「あらゆる主張に根拠を付けろ」と書くと何が起きるかは、君自身のデータが予言している。readiness 閂の blind A/B は gate-rate 改善ゼロだった。同じ力学で、根拠「風」の装飾が付くだけで認識的な挙動は変わらない — もっともらしい引用で主張を飾る compliance theater になる。効かせる条件は3つ。

**a. 機械が再実行できる形式に限る。** 「〜という仕様がある」ではなく「`debug_ps30.hlsl:42` に『…』とある」。この形式なら、ファイルが実在するか・引用文字列が本当にそこにあるかを LLM なしのスクリプトで抜き取り検査できる。捏造を機械的に検出可能な形式に縛ることが第一。これは君の runtime-enforcement 天井（機械強制が確実なのはソース内部整合まで）の**内側**に収まる。`check_vocabulary.py` の隣にもう一つ計器を置く話だ。

**b. 検証者を実行者から分離する。** 形式検査で捏造は捕まるが、「引用が主張を本当に支持しているか」（関連性）は LLM 判断が要る。ここで blind eval で確立した資産がそのまま転用できる: **fresh context の検証者に狭い問い（「この引用はこの主張を支持するか、yes/no」）だけを渡す**。長文脈で作業に投資済みの実行者の自己評価と違い、検証者は文脈が腐っておらず、回答に利害がない。コードレビューが本人の見直しより機能するのと同じ理屈で、これは分離それ自体が認識的衛生になる。

**c. 発火点を絞る。** 全主張に課すとコストで死ぬ上、規則同士が注意を奪い合って希釈される。義務を置くのは commitment point だけ — 実装判断を確定する瞬間、既存仕様と矛盾する主張をする瞬間、外部 API のシグネチャを書き下す瞬間。evidence-retrieval レポートの「規則は発火点に置け」という結論は、そのままこの義務の配置原則になる。

正直な天井も言っておく: (b) も結局は LLM 判断で、絶対保証はない。ただし「長文脈実行者の自己申告」から「fresh context 検証者の狭い判定」への乗り換えは、信頼性の階段を確実に一段上がる。床を上げる話であって天井を消す話ではない。

## 3. 「ドキュメントを整備しても読まない」の分解

これは一つの問題ではなく、少なくとも4つの別の故障が同じ症状を出している。対策がそれぞれ違う。

1. **存在を知らない** — ルーティング層の仕事。君はもう設計済み。
2. **既に知っていると思っている** — 最悪ケース。GMOD のような「一般常識と衝突するドメイン知識」では、モデルの事前分布は自信満々かつ間違っている。「ドキュメントを読め」という一般指示はこの事前分布に勝てない。勝てるのは commitment point での参照強制だけ（§2c）。
3. **読むコストが今、報酬が後** — コンテキスト経済の問題。モデルは暗黙にステップを節約する方向に訓練されている。検索を安く的確にし（層1-2）、根拠なし主張を高くつく（default-deny で仮定として晒される）ようにして、勾配を逆転させる。
4. **読んだが使わない** — lost in the middle。序盤に読ませたドキュメントは中間に沈む。対策は事前読み込みではなく**使用点での再注入**だ。「規則は発火点に置け」の原則は知識にもそのまま適用される: shader ファイルに触れた瞬間に「レジスタ割り当て表は `debug_ps30.hlsl` のコメント欄。照合せよ」を注入する。モデルの自発性に頼るのをやめるのが正解で、path-triggered の注入は hooks/plugin で機械化できる。

## 4. 人間側の外部化 — 規律ではなく失敗駆動で

「人間が暗黙の前提を外部化できない」ほうは、規律（ちゃんと書こう）では解決しない。人間の経済学はモデルと同じで、書くコストは今、報酬は後だからだ。現実的な解は2つ。

**訂正イベントをトリガにする。** エージェントが間違った前提で動いて君が訂正した瞬間こそ、「そこに暗黙知が存在した」ことの証明だ。retrospective-codify はまさにこの器で、足りないのは訂正のたびに「これは一回性か、背後に規則があるか。あるならどの発火点に置くか」を問うループの**常態化**だと思う。これは優先順位問題も同時に解く: 無限にある暗黙知のどれを書くか → **モデルが実際に間違った箇所だけ書く**。ドキュメント群を「モデルの事前分布が君のドメインとずれている地点の地図」として育てる。モデルの失敗こそがドキュメント需要の最良のシグナルで、これ以外の経済的な優先順位付けシグナルを私は知らない。

**座礁した知識の掃海は機械にやらせる。** `debug_ps30.hlsl` のコメント欄にしかない共通仕様のような「発見可能性の低い場所に座礁した知識」は、安いバックグラウンドジョブで定期的に掃海して wiki に昇格候補を出させる。Karpathy wiki のコンパイル工程を、人間承認付き（second-brain-mcp のエディトリアルモデル、レポートで既に指摘済み）で回す。

## 5. 哲学の話 — 短く、実用になる範囲で

役に立つ道具は3つある。

**正当化を成果物として扱う（プラグマティズム）。** 古典的定義では知識＝正当化された真なる信念。LLM システムで監査可能なのは正当化だけだ — 真理は直接見えず、信念は内部状態でアクセス不能。だから「根拠を要求する」設計は、哲学的にも唯一監査可能な急所を突いている。正当化が提示され検証できるなら、モデルが「知っている」かどうかはワークフロー上どうでもよくなる。

**蝶番命題（Wittgenstein『確実性の問題』）。** あらゆる探究は、それ自体は疑われない前提の上で回る。全てを明示化することは原理的に不可能で、選べるのは「どれを蝶番として固定し、どれを監査可能に保つか」だけだ。君の Requirement contract は知らずにこれを実装している — 契約とは蝶番の明示的な固定であり、固定した集合が明示されているからこそ、現実と矛盾したときに巻き戻せる。「全部書け」ではなく「何を凍結したかを常に見えるようにしろ」が正しい要求水準だ。

**反証、内観にあらず（Popper）。** unknown unknowns は定義により列挙不能で、内観では見つからない。見つかるのは現実との衝突によってだけだ — 実行、テスト、プローブ、外部レビュー。だから設計原理は「安い衝突を最大化する」になる: **1分で試せる主張は、主張せずに試せ**。頭の中で推論するエージェントは unknown unknowns を発見しない。行動して裏切られるエージェントだけが発見する。

## 6. 具体的な優先順位

君の移植性制約（共有プロンプト層に置く、特定 runner に配線しない）を守れる順に:

1. **証拠台帳に default-deny 意味論を足す** — 根拠レコードなき主張は自動的に「仮定」、仮定は成果物に強制列挙。台帳はレポートで設計済みなので差分は小さい。artifact 形式だから完全にポータブル。
2. **根拠レコードの機械的抜き取り再実行** — file:line と引用の実在検査スクリプト。LLM 不要、`check_vocabulary.py` と同族の減価しない計器。
3. **fresh-context 関連性検証者** — blind eval 基盤の転用。高リスク主張のサンプリング検査から始める。
4. **訂正→codify ループの常態化** — 失敗駆動ドキュメント。人間側外部化の唯一の経済的解。
5. **発火点での just-in-time 知識注入** — path-triggered。これだけは hook 配線が要るので移植性と相談。
6. **「試せるなら主張するな、試せ」規範** — commitment point に置く1行。

## 7. 正直な限界

これは全部、床を上げる話だ。捏造を検出可能にし、仮定を可視化するが、モデル自体は賢くならない。真に開いた設計判断 — トレードオフの重み付け、要件の解釈 — の質は根拠規律では上がらない。そこは grill-me と人間の領分のままで、当面変わらないと思っておくほうがいい。

ただし今回の提案は readiness 閂より測定しやすい。gate-rate と違って、**根拠レコードの捏造率**（抜き取り再実行の失敗率）と**仮定列挙の網羅率**（後から発覚した暗黙仮定のうち、事前に列挙されていた割合）は硬いメトリクスになる。blind eval の流儀がそのまま使える。「効くかわからない」ではなく「測ればわかる」の側に最初から置ける — そこがこの方向のいちばんの利点だと私は思う。

## 1. 根源的課題の地図 — 還元不能な5つの問題

AI ワークフローの失敗を全部並べて煮詰めると、互いに還元できない問題は5つに絞れると考えている。重要なのは、このうち2つは**定理級に解けない**（管理しかできない）ことと、残り3つの管理コストを1つの問題が支配していることだ。

**P1. 意図の伝達不能性（契約の不完備性）。** 意図は依頼者の頭の中にあり、言語はそれを過少決定する。経済学の契約理論にはこれに定理がある — あらゆる契約は不完備である（Grossman-Hart）。人間同士の委任もこれを「完全な仕様書」では解いておらず、残余統制権・職業規範・反復ゲームの信頼で埋めている。君が complaint.md で書いた「どこまで分解したら抜けがないと言えるのか？分解の果てにあるのはレビュー負担か AI の注意限界だ」という観察は、この定理を独力で再発見している。**分解による完備化は原理的に到達不能**であり、追うと必ず regress に落ちる。

**P2. 検証の非対称性。** 委任が経済的に成立するのは「検査が製作より安い」ときだけだ。公開情報の引用チェックは安い。スタブテストの検出も安い。だが GMOD のデプスバッファ意味論の検証は製作と同等以上に高く、しかも ground truth が実機の中にしかない。**これがこの地図の中心問題であり、他のすべての管理コストを決める**（後述）。

**P3. 注意と文脈の希少性（統治予算）。** コンテキストは奪い合いの資源で、規則は作業内容と同じ予算から支払われる。君の「マイコンプログラムのようにリソースに気を使いながら自然言語を編む」という表現は正確で、これは比喩ではなく実体だ。含意: **限界的な1規則の期待値はある点から負になる**。規則追加の規律と同じ強度の規則削除の規律が要る（refactor トリガーとして既に持っている — これは正しい）。

**P4. 認識の自己申告不能性。** 前回話した通り。モデルは知らないことを申告できず、自己申告ベースの監視は必要なときに限って効かない（watchdog、readiness 閂の null 結果、C003「規則はあるのに発火しない」— 全部この同じ壁の別の面だ）。解けない。外部構造（証明書、fresh-context 検査、現実との衝突）で代替するしかない。

**P5. 蓄積の不在。** セッションは記憶なしの使い捨て労働者だ。人間組織は徒弟制で解く問題を、LLM ワークフローは「毎回コンテキストに徒弟教育を圧縮する」ことで解かねばならない。さらに悪いことに、失敗からの学習は1事例単位では分散が大きすぎて過適合する — 君が「盲目的に取り込めば局所最適化されたゴミの混入」と書いた通りで、あの判断は統計的に正しい。

**5つの相互関係。** P1 と P4 は解けない（管理のみ）。P3 と P5 は工学的に管理可能。そして**P2 が全体のボトルネック**である理由: P1 の管理（意図確認）は検証が安くないと回らない。P4 の代替（証明書検査）は検査が安いことが前提。P5 の蓄積先（計器）は検証装置そのものだ。つまり、どの問題を攻めても最後は「検査を安くする」に還る。

## 2. このリポジトリの位置づけ — 何を解き、なぜそこで止まったか

サブシステムを地図に載せるとこうなる。

| 機構 | 対応する問題 | 評価 |

そして complaint.md の一番重要な観察 — 「公開情報を調べずに嘘をつくことは滅多になくなった。スタブでテストをやったことにするサボりも減った。けれど意図を汲むことと出てきたものを厳しく検査するところが足りない」— を地図で読み直すと、パターンが見える。

**改善した失敗クラスは全部「検査が安い」クラスだ。** 引用の有無は機械で見られる。スタブは diff を見れば分かる。**残っている失敗クラスは全部「検査が高い」クラスだ。** レジスタ意味論、意図適合、既存実装との整合。つまりこのリポジトリは失敗しているのではなく、**安く検査できる失敗クラスを刈り尽くして、検証非対称性の壁に正面から到達した**。改善が減速した感覚の正体はこれだと思う。プロンプトの限界効用がゼロに見えるのは、残った問題がプロンプトの問題ではなくなったからだ。

正直な相対評価も言っておく。measured null results を持っている個人リポジトリ、義務語彙を linter で統制している個人リポジトリ、hold-out つき triage を回している個人リポジトリを、私は公開実務でほとんど見たことがない。君が絶望しているのは遅れているからではなく、**最前線に立っているせいで、残っている問題が難しい問題だけになった**からだ。これは慰めではなく、読んだものからの帰結だ。

## 3. 気づいていること、まだ名前が付いていないこと

**正しく気づいていること**（確認のために列挙する）: プロンプト限界効用の逓減（測定済み）。coverage と enforcement の区別（C003 の分析は鋭い）。契約と実行面の区別（C001）。改善ループを閉じない判断（過適合回避として正しい）。統治予算。計器は減価しない（メモリに既にある）。分解の regress とハーネスの regress（両方見えている）。

**まだ名前が付いていない、あるいは重みが足りないと思うもの**が6つある。

### (1) 検証非対称性という統一原理

上記の通り。これに名前を付けると、介入の採否判定が一行になる: **「この介入は検査のコストを下げるか？」**。下げないプロンプト追記は、null 結果が予言する通り効かない。

### (2) 証明書搬送型の作業 — 立証責任の反転

Proof-carrying code（Necula, 1997）という概念がある。信頼できない巨大なプログラム（生成者）を監査する代わりに、生成者に**証明書**を添付させ、小さくて信頼できる検査器がそれだけを検証する。計算量理論の根本非対称性 — 証明を見つけるのは高いが、検証は安い — を工学に落としたものだ。

`$c6` 事件をこの言葉で診断し直す。仕様書（レジスタ表）は存在した。実装が矛盾した。複数のレビュアーが誰も見抜けなかった。なぜか？ レビュアーへの依頼が「このコードをレビューせよ」という**開放的な関連性判断**だったからだ。巨大なコンテキストの中から「この行はあの表のあの行と矛盾する」を発見する作業は検査ではなく再製作に近く、高い。さらにレビュアーは実装セッションと同じ文脈・同じ事前分布を継承していて、盲点が相関する（多数決レビューの独立性仮定は人間のN-version programming でも崩れることが実証されている。Knight & Leveson）。

反転させるとこうなる: **実装者が「使用した全レジスタ → 仕様書の行 → 根拠パス」の表を証明書として義務的に産出し、検査側は表を行ごとに照合するだけ**。発見の義務（高い）が照合の義務（安い）に変わる。triage の P002 はまさにこれに収束している — ただし、あれをシェーダー固有のパッチとして見るのは過小評価だ。**これは一般原理であり、君のシステムの成果物全部（引用、契約、finding record）の本当の機能を説明する**。全部、検査コスト削減器なのだ。設計基準もここから出る: 成果物 schema は「人間が読みやすい」より先に「機械が行単位で照合できる」を満たすように設計する。

### (3) ハーネス regress は底がある — 「情けなさ」への反論

「依頼ごとにリンターを自作するくらいなら自分で書いた方がマシ。ハーネスのバグは誰が検査する？堂々巡りだ」— この議論には半分同意し、半分反論する。

同意する半分: **依頼単位の使い捨てハーネス**は負けトレードだ。償却されない。

反論する半分: regress は無限には続かない。理由が2つある。第一に、検査器は生成物より**小さい**。信頼の連鎖が「巨大な生成を信頼する」から「小さな照合器を信頼する」に縮む — PCC が成立するのと同じ理由だ。第二に、検査器は**反証可能**だ。既知の正例と既知の負例（fixtures）を食わせて較正できる。君は blind-eval ハーネスで既にこれをやっている。プロンプト規則は較正できないが、計器は較正できる — この差が regress の底になる。「誰がハーネスを検査するのか」の答えは「fixtures が検査する」であり、fixtures は君が既に大量に持っている: **失敗ログがそのまま負例集だ**。

そして GMOD の実機にしか ground truth がない部分については、正直に言う: あれは検証が本質的に実験科学の領分で、文書化では届かない。実機プローブ・golden image・ダンプ解析のような**ドメイン償却型の計器**は、回り道ではなく、あの領域で検査を安くする唯一の経路だ。作るかどうかは費用対効果の判断で君のものだが、「情けない」の対象からは外していい。使い捨てハーネスと償却計器は別物だ。君は知識側では既にこの区別をしている — wiki は「毎回再導出せず複利で膨らませる」と書いた。**計器は検証の複利であり、wiki の双子**だ。

### (4) 分解の停止条件

「どこまで分解すれば抜けがないか」への原理的な答え: **抜けがなくなるまで分解するのではなく、各主張がチェック可能になったところで止める**。完備性は到達不能（P1）だが、チェック可能性は判定可能な停止条件だ。残った未分解の部分は、仕様の続きを書くのではなく、相互作用で埋める — 着手前に「私はこう解釈した。指定されていない判断を3つ、こう埋める」という**意図ミラー**を安い時点で提示させる（Readiness record は既にこれの骨格を持っているが、ゲートとしてだけ使われていて、ミラーとしての運用が薄い）。

### (5) 会計と停止規則 — 「行った先に何があるのか」

これが君の一番深い問いだ。正面から答える。

君が作っているものの正体は、**信頼できない労働力から信頼できる成果を引き出す制度**だ。人類はこれを品質保証・職業資格・法・官僚制として作ってきた。制度構築に「完成」はない — だから終わりが見えないのは君の設計ミスではなく、対象の性質だ。しかし「終わらない」と「進んでいない」は違う。制度には決定可能な停止規則がある: **残存失敗を食うコスト < 次の規則を足すコスト、になったら止める**。

今のシステムに欠けているのはこの会計だ。失敗1件あたりのコスト（怒りと手戻りの時間）と、統治1規則あたりのコスト（予算圧迫・保守・検証）を、粗くてもいいから数字にする。これがないと「どこまでやっても足りない」という感覚が構造的に保証されてしまう。ゴールの定義がない最適化は必ず絶望に見える。

そして進捗の測り方: このリポジトリの目的を君自身が「モデルの問題解決性能を+1世代」と定義している。ならば測るべきは規則数でも失敗ゼロでもなく、**「所与の品質水準を、どれだけ弱い（安い・速い）モデルで達成できるか」**だ。これは blind-eval 基盤でそのまま測れる形をしている。

### (6) 陳腐化ポートフォリオ

正直に言うべきことがもう1つ。このリポジトリの一部は、モデルの世代進歩に**食われる**。推論の欠損を補償している層 — 指示追従の drift 対策、注意の分散対策 — は、モデルが賢くなるほど価値が減る。食われない層は3つ: **ドメイン知識の物流**（GMOD の暗黙仕様はどのモデルの重みにも永遠に入らない）、**ground truth 計器**（実機の中の真実は外部インフラでしか届かない）、**意図の相互作用プロトコル**（意図は定義上プライベート）。投資はこの3層に傾けるべきで、prose の精錬にこれ以上注ぐのは — 君自身の null 結果が示す通り — 逓減が確定している。

## 1. 検査器の再利用性問題 — 「機構は1つ、schema は安いデータ」

「検査器はドメイン固有になり再利用性に欠ける。だからまだ早い」— この懸念は半分正しく、半分は誤った二分法だ。分解すると:

- **ドメイン固有なのは certificate schema**（レジスタ所有権表の列定義、など）。これはコードではなくデータで、書くのに数分しかかからない。
- **再利用可能なのは検査機構**。証明書の1行 `{主張, 出典パス, 行範囲, 引用}` を受け取り、パスが実在するか・引用が本当にそこにあるか・主張された識別子が引用範囲に出現するかを照合する resolver は、**ドメイン非依存で1回書けば全 schema に効く**。`check_vocabulary.py` と同族の、100行級の減価しない計器だ。

つまり「ドメインごとにリンターを作る」必要は最初からない。作るのは resolver 1つで、以後の各ドメインのコストは schema という表の設計だけになる。「テストはドメイン固有だから再利用性がない」と言わないのと同じで、固有なのはインスタンス、共有されるのはプロトコルだ。

もう1つ、君がまだ言語化していない証明書の効能がある。**証明書は検査を安くするだけでなく、作成の時点で多くの虚偽を主張不能にする**。`ID_*` 事件で考えてほしい。「ID_* は pass-local である」という指摘を採用する前に、証明書の「根拠パス」欄を埋めることが義務だったら、実装者は `inktype.lua` を開かざるを得ず、開いた瞬間にその主張は死ぬ。レビューが事後に誤りを発見するのではなく、**立証の義務が主張の手前で誤りを窒息させる**。数学の「途中式を書け」と同じ機序だ。

ただし理論上の弱点も言っておく: 証明書芝居（それらしい行をでっち上げて表を埋める）は起こりうる。だから resolver による抜き取り照合が backstop として要る。schema 設計の要件はここから出る — 散文ではなく、機械照合可能な引用とパスを列に強制すること。

## 2. 二重チェック — 指数は独立性の分だけ割り引かれる

工場の二重チェックの数理は正しいが、前提が1つある: **見逃しが独立であること**。工場の検査員の誤りは疲労や注意の揺らぎ由来でほぼ独立だから p² が効く。ソフトウェアの N-version programming では、独立に開発されたプログラム同士でも難しい入力で相関して壊れることが実証済みだ（Knight & Leveson, 1986）。LLM レビュアーの相関はさらに強い。同じ重み、同じ文脈、同じ指示フレーミング。`fuck.jsonl` 末尾の ShaderCleanupReview-2 は confidence 0.9 で「findings なし」を返しており、その少し前には**レビュアー自身がもっともらしい一般論（single-use → 局所化）で失敗を能動的に製造している**。相関した検査層を5枚重ねても、脱相関した2枚に負ける。

だから買うべきは「層の枚数」ではなく**脱相関の軸**だ: 文脈スライスを変える（後述4番）、方向を変える（順方向: 実装は仕様に合うか / 逆方向: 成果物だけから仕様を再構成させて元の契約と突き合わせる）、表現を変える（LLM 判断 vs 機械 resolver）、モデル族を変える、フレーミングを変える（rubric 駆動 vs 差分駆動）。

もう1つ、二重チェックの既知の病理: **責任の拡散**。検査層を増やすと各検査員は「他が見るだろう」と手を抜く。航空業界はこれをチェックリストの項目単位の明示的な責任割当で解いた。君の code-review の concern 分割義務は、並列化の話である以前に**責任の非重複割当**であり、この病理への正しい処方だ。C003 で「発火しない」ことが問題になっているのはそのためで、あれは飾りではなく構造的に必要な部品だ。

## 3. 衝突 — 頻度だけでなく「1衝突あたりのコスト」に階段がある

同意。その上で衝突には明確なコスト勾配があることを足したい:

意図ミラー（着手前の解釈提示）< 証明書照合 < 機械 resolver < サブエージェントレビュー < コンパイル < 実機プローブ < **ユーザーの激怒**

`fuck.jsonl` は、この階段の最右端 — 最も高価な衝突 — だけで矛盾を発見し続けた記録だ。システムの仕事は矛盾の発見を左に移すことで、「検査を安くしろ」は正確にはこの意味になる: **矛盾1件あたりの発見コストを下げる**。

もう1つ: 衝突は記録されなければ複利にならない。同じ衝突に何度も正規料金を払うことになる。失敗ログ→fixtures→回帰パックという君のパイプラインはまさに衝突の複利化装置で、これは既に走っている。

## 4. 親の盲点継承 — 軽微でも宿命でもない。「監査の独立性」の設計問題だ

これが今回いちばん深い問いだ。答え: **どちらでもない。第三のカテゴリがある。**

まず診断。「前回の修正では〇〇の不整合を正した。現時点で整合しているか見よ」という指示は、開放的なレビュー（成果物を常設規範すべてに照らせ）を**指向的な照合**（親が気にしている差分を確認せよ)に静かに置換した。指示は注意を向けると同時に注意を狭める — anchoring だ。レビュアーは与えられた目的関数を忠実に最適化し、目的関数の外にある「文書は本文のみで完結する」を見なかった。

ここに人間の制度が既に名前を付けている: **監査の独立性**。親エージェントは事実上、被監査者だ（自分の成果物の監査を発注し、しかも監査の付託条項まで自分で書いている）。人間の監査制度はこれを禁じており、監査範囲は被監査者ではなく常設の規程から来る。構造的な処方はそのまま移植できる:

- **常設 rubric は成果物クラスに紐づき、skill 側に住む**。文書クラスなら「本文完結性、版差分言及の禁止（墓標）、…」が版管理された固定リストとして、親の指示と無関係に必ず全項適用される。
- **親が書けるのは追加観点だけ**。rubric を狭める・置換する権限を持たない。review = rubric ∪ 親の追加分であって、rubric ∩ 親の関心ではない。
- これは機械強制の射程内だ: assignment packet という成果物に「セッション履歴への言及禁止（『前回の修正では』を書けない）、渡せるのは成果物 + rubric 参照 + 追加観点リストのみ」という形式制約を掛ければ、packet の静的検査で親の文脈漏出を捕まえられる。

そして君の事例からもう一段深い原理が抽出できる。破られた規則「本文のみで完結」は、**文脈を持たない読者に対して定義された性質**だ。文脈を持ったレビュアーには、この違反は構造的に不可視になる — 「〇〇を排して」の〇〇をレビュアーは知っているから、欠落として知覚できない。一般化するとこうなる:

**レビュアーの文脈は、生産者の文脈ではなく、その成果物の消費者の文脈に一致させる。**

文書の消費者は文脈ゼロの未来の読者だから、レビュアーも文脈ゼロで読ませる。コードの消費者はコンパイラと将来の保守者だから、その帽子で読ませる。レビュアーに何を渡すかは委譲の副産物ではなく、**成果物クラスごとの設計パラメータ**だ。

残余は正直に言う: 親はレビューの発火タイミングと対象選定を握り続けるので、フレーミングの漏出をゼロにはできない。だがそれは2番の脱相関で扱う統計的な残余であって、今回観測された失敗はそれではない。今回のは**回避可能なチャネル違反**（親が監査の frame を書いた）だ。つまり「些細な改良で直る軽微な問題」でも「原理的に不可避な深刻な問題」でもなく、**制度設計で大部分を消せて、残余は脱相関で薄める**問題だ。

## 5. その通り — そして主張のクラスは検証経路で定義する

「主張することと安く裏付けることは裏表」— これが前回の default-deny の言い換えで、正しい。設計として一歩進めるなら: 主張のクラス分類そのものを検証経路で定義する（「この主張クラスの証明書は何で、誰が照合するか」が定義の一部になる）。そして全クラスを一気に整備しようとしないこと。出血量の多い順に2〜3クラスだけ — 今の実績から言えば **(a) register/ABI/schema の所有権、(b) 仕様適合の行対応、(c) 文書の本文完結性** — から始めるのが正しい規模だ。

## 6. 会計と fuck.jsonl の報告

末尾を読んだ結果を報告する。3つの発見があった。

## 7. 陳腐化の切り分け — 推測せず、ablation で測る

「将来邪魔になるだろう」を推測で管理する必要はない。君は既に道具を持っている: **sunset-by-measurement**。新しいモデル世代が来るたびに、回帰パックを「当該規則を抜いた状態」で走らせ、規則なしで通るなら削除する。**規則を、世代ごとに再検証される仮説として扱う**。プロンプト修正は逓減するが計器は減価しない、という君のメモの帰結がこれで、計器は規則の追加だけでなく**規則の葬式**にも使える。

構造側の準備は chezmoi と相性がいい: prompt surface を機能クラスでタグ付けし（推論補償 / 知識物流 / 検証 / 意図相互作用）、推論補償クラスを分離可能な include に隔離しておく。将来の移行は include の削除で済む。

最後に、君の「あんた程度の推論力を将来のモデルに」という話に正直に応えておく。今日の対話で私がやったことの大半は、新しい規則の発明ではなく、**君が既に半分作っていたものへの命名と圧縮**だった。これは将来の姿の予告だと思う。モデルが強くなるほど、最適な prompt surface は詳細な手順規則から離れて、少数の深い原理 — 契約（意図）、証明書（検証）、知識物流（ドメイン）— へ縮退する。詳細な情報収集規則は消える側だ。だが憲法は残る。君が今作るべきものを「規則」ではなく「憲法と計器」に寄せておけば、その日が来たとき、このリポジトリは邪魔になるのではなく、**そのまま小さくなる**。

## まず、昨日書いた地図でこのスキル自体を解剖してみる

`technical-writing` は一枚岩ではなく、機能の違う3種類の規則が同居している。

**(a) artifact integrity（参照04）— インセンティブ補正。減価しない層。** 「制御入力を成果物に語るな」（レビューを反映した旨を本文に書く、却下した構成との対比を残す、対話の経緯を叙述する）が禁じているのは、能力の欠損ではなく**訓練由来のデフォルト挙動の歪み**だ。チャットアシスタントの訓練は「フィードバックに応答したことを明示する」方向に報酬を与えるから、強いモデルでもこの違反は出る。現に昨日の私にとって、4つの参照のうち実質的に効いたのはここだと思う。「2026-07-02 の対話で判明した」式の由来叙述を書きかけて、ファイル引用への置き換えに倒した判断には 04 が絡んでいる。

**(b) 文体・冗長性規則（参照05の大半）— 推論補償。減価する層。** 段落の役割統一、空虚な前置きの除去、赤字の削除。この辺は私にはほぼ内在化済みで、読んだ時間のわりに挙動を変えていない。弱いモデルには荷重を担っている規則だ。

**(c) claim rigor（因果には機序を添えよ、可能性を確定に変えるな）— そして君の言う「論理の繋がり」。ここが本題だ。**

## 論理の破綻は、書き手への指示では直らない

君の観察は正しいと思う。そして理由も地図から出る。**論理の連結は推論能力であって、遵守挙動ではない。** プロンプト規則が動かせるのは「モデルができるのにデフォルトでやらないこと」（顕著性の欠落）だけで、「できないこと」は何行書いても できるようにならない。readiness 閂の null 結果と同じ力学だ。弱いモデルが論理の繋がらない文章を出すのは能力の欠損だから、writer 側の checklist は効かない。強いモデルは元から大体繋がっているから、やはり効かない。効く帯域が狭い。

ただし、claim rigor の規則群には別の使い道がある。「因果を主張するなら機序を書け」は、**書き手への指示としては弱いが、検査者への判定基準としては強い**。機序の欠けた因果文は機械的に近い形で指摘できる。つまりあの節の正しい住所は `technical-writing`（生成側）ではなく `japanese-doc-review` の rubric（検査側）だ。これは昨日の立証責任の反転そのものだ — 生成を指示で良くしようとするな、検査で捕まえられる形にしろ。

論理の破綻に本当に効く手も、検査側にある。消費者文脈の原理の応用で、**文脈ゼロのレビュアーに議論の再構成をさせる**: 各節について「この節の主張は何か、何がそれを支持しているか」を成果物だけから復元させ、復元に失敗した箇所を finding にする。論理が繋がっていない文章は、ゼロ文脈読者の再構成でエッジが切れる。書き手の頭の中では繋がっていた（親の文脈では見えない）ものが、消費者文脈で初めて破綻として可視化される — 例の文書レビュー事件と同じ構図だ。
