# AI委任監査の観測面

## 0. この文書の位置づけ

本書は、`trust-collapse.md` で整理した問題構造に対し、何を観測すれば人間の確認範囲を限定できるかを整理する候補設計である。

ここで示す観測面は、恒久的な台帳を必須としない。タスク中だけ生成する一時成果物、既存のdiffやtest reportから作るview、機械的な検査結果として実装できる。独立ファイルやschemaを増やすのは、観測方法、失敗時の処置、保守者、寿命を分離しなければ事故が起きる場合に限る。

## 1. 最小の観測面

### 1.1 基準面

**問い**

- 今回の規範的内容は何に由来するか。
- 制約、許可、選好のどの役割を持つか。
- 確認済み、仮定、未確認のどの状態か。
- 合否、選択の認可、候補比較、再判断のどこで使われるか。
- 認可元にない精度を追加していないか。
- 代理指標は目的を表しているか。

**候補成果物**

- 依頼原文と規範的内容の差分。
- 規範的内容ごとの認可元。
- 制約、許可、選好の役割と確定状態。
- 未確定事項。

**人間へ出す条件**

- 元の入力より意味または強さが増えた。
- 数値境界や固定書式を追加した。
- 代理指標を合否基準として使う。
- 選好を受け入れ条件へ昇格した。
- 許可された範囲を、認可元より広く一般化した。
- 仮定または未確認の内容を、確認済みとして後段へ送った。

### 1.2 判断面

**問い**

- 判断材料から一意に決まらず、実装者が裁量で選んだ点は何か。
- 局所解か一般解か。
- 代替案と選択理由は何か。
- 間違っていた場合に何が壊れるか。

**候補成果物**

```text
選んだ内容:
未規定だった点:
主な代替案:
選択理由:
局所解 / 一般解:
壊れる条件:
確認した範囲:
```

**完全性の補助**

実装者の自己申告だけに依存せず、別の検証者が変更と実行結果から次を探す。

- 新しい公開APIや状態。
- 新しいfallback、上限値、所有権。
- 特定のworkloadに依存する定数。
- 計画からの逸脱。
- 一般解を装った局所解。

### 1.3 変更面

**問い**

- 何が変わったか。
- 各変更は何によって許されるか。
- 必要な変更が欠けていないか。
- 別作業や人間編集が混ざっていないか。

**候補成果物**

- 機械列挙した変更一覧。
- 認可済、派生、孤立、保留、帰属不明の判定。
- 未実現条項。
- 状態差分と実行記録の残差。

**人間へ出す条件**

- 孤立、保留、帰属不明。
- 未実現条項。
- 変更粒度が粗く、認可判定が割れる。
- 生成物を再現できない。

### 1.4 根拠面

**問い**

- 主張と対応関係は何に支えられているか。
- 根拠は実在するか。
- 根拠は主張を本当に支持するか。

**候補成果物**

- source path、range、quote、identifier。
- command、environment、exit status、出力。
- 対象版、commit、artifact digest。

**検査の分離**

- 形式照合: パス、引用、識別子、実行記録の実在。
- 意味照合: 根拠が主張を支持するか。

形式照合のpassを意味照合のpassへ流用しない。

### 1.5 検証面

**問い**

- どの観測が制約の遵守を、どの範囲で支持するか。
- 採用した選択は、どの許可によって正当化されるか。
- 選好は候補比較でどのように扱われたか。
- 後続判断が依存する仮定は、維持、反証、未確認のどれか。
- 実経路、対象環境、failure pathを含むか。
- 未検証領域は何か。

**候補成果物**

- 制約とverification methodの対応、および許可、選好、仮定と選択理由の対応。
- 実行した経路、環境、入力範囲。
- 観測結果。
- 支持できる結論の範囲。
- 未実行項目。

**人間へ出す条件**

- 制約とverification method、または許可、選好、仮定と選択理由の対応が弱い。
- 単体実行を統合経路へ一般化している。
- 一つのplatformの結果を複数platformへ流用している。
- narrow benchmarkの成功を一般解へ広げている。

### 1.6 計器面

**問い**

- 観測と判定に使った計器は、今も信用できるか。
- 既知の正例・負例を正しく判定するか。
- harness、model、environmentの更新後も発火するか。

**候補成果物**

- fixtureのpass/fail。
- 対応platformと未対応platform。
- 最終較正日時。
- 既知の誤検出、見逃し、迂回経路。

### 1.7 観測範囲と深さの選択

本書は観測面の候補を列挙するが、すべてのタスクで全観測面を同じ深さまで使うことは要求しない。

観測範囲と深さを選ぶ規則は未確定である。後続の検討では、`trust-collapse.md` §8が示す損失、後段での検出困難性、撤回困難性、後続判断への伝播、人間の確認面を縮小する効果と、本書の六つの観測面との対応を扱う。

## 2. 人間へ提示する最小出力

人間へ全観測面を読ませてはならない。通常は、次だけを提示する。

```text
今回の重要な裁量判断
未認可または説明不能な変更
検証方法と要求の不一致
未実行・未証明・未較正の高リスク部分
人間の決定が必要な項目
```

各項目には、根拠と、判断しなかった場合の影響を付ける。

機械的に認可済みで、既知fixtureにより較正され、再実行可能な項目は折り畳む。監査の目的は記録量を増やすことではなく、人間の注意を裁定が必要な場所へ集めることである。

## 3. 恒久記録に昇格する条件

一時的な監査成果物を、常設規則、fixture、ADR、schemaへ昇格するのは次の場合に限る。

- 同じ失敗または裁定が複数の作業で再発した。
- 人間の裁定を機械検査へ変換できる。
- 将来の確認費用を明確に下げる。
- 適用範囲と反例を固定できる。
- 既存の記録へ統合すると誤検出、見逃し、警報疲労、保守腐敗が起きる。

一件の失敗から恒久schemaを作らない。まずfixtureまたは一時的な観測項目として保持し、再発と移転可能性を確認する。

## 4. 評価指標

候補設計の採用可否は次で判断する。

- 高損失失敗の検出率。
- 人間が読んだ行数、項目数、時間。
- 誤検出、保留、帰属不明の割合。
- 追加token、turn、実行時間。
- fixtureへの再利用率。
- model、harness、platform更新後の回帰。
- 常設規則、例外、schemaの成長率。

人間の確認面が縮小しない場合、観測面の追加は失敗である。
