# 意図抽出の発火と停止の規律

## 0. 本報告の位置づけ

- 本報告は、Known Unknown にある事項を質問によって Known Known へ移す操作（以下、意図抽出）について、発火条件、停止条件、回答の保存形式、較正を固定する。
- 上位文書は `opencode-prompt-dev/fundamental-problem-map.md`（委任の4面、知識配置図、根源課題 P1〜P5、衝突の予定）。並列文書は `opencode-prompt-dev/fundamental-problem-analysis/trust-collapse.md`（監査台帳、仕様の不確かさの操作的定義、警報疲労）。本報告は両文書を変更しない。trust-collapse §4 は「Known Unknown にある事項を質問や決定によって Known Known へ移す作業に §5.4 の目的適合が停止規則を与える」とすでに述べており、本報告はその一文の展開にあたる。
- 新しい台帳・独立分類は導入しない（trust-collapse §8 の分類規律に従う）。本報告で導入する語は既存分類へ次のとおり畳む。作話条項は認可台帳の誤判定の発生源の1つ、三値保存形式は質問回答の記録形式、覆り率は計器台帳の較正指標である。
- 発端は 2026-07-07 の分析セッション。外部観察（コーディングエージェントが曖昧な指示を質問で解消せず実装コストの低い解釈を勝手に選ぶ、という不満と、grill 型ツールの流行。出典参照）を契機に、「意図抽出はいつ行い、どこで止まるべきか」を trust-collapse の語彙で操作化した。

## 1. 中心命題

意図抽出の終端状態は、Known Unknown 象限の空化ではない。空化は P1（意図の伝達不能性）により原理的に不可能であり、空化を目標に置くと停止規則が定義できない。

終端状態は次で定義する。

> 残存するすべての unknown に、質問より安い衝突が予定されている状態。

意図抽出の仕事は unknowns の削減ではなく、各 unknown を最安の衝突経路へルーティングすることである。質問はルーティング先の1つにすぎず、人間の注意予算（増設不能な最上位統治予算）を消費する高価な経路である。

## 2. 故障モード: 選好の作話は質問チャネルの警報疲労である

trust-collapse は監査装置の真の故障モードを論理破綻ではなく警報疲労と特定した——人間が札を機械的に承認し始め、札が信号でなくなる。過剰な意図抽出はこの双対であり、新しい現象ではない。質問が多すぎると人間は機械的に答え始め、答えが信号でなくなる。意図抽出は意図条項（変更・意図の二部関係の右側）を生産する監査装置であり、監査装置の故障分類を継承する。

意図面には「まだ決めていないが決められる」領域の外側に、選好そのものが存在しない領域がある。そこへの質問は知識を転送しない。選好を製造する。人間は質問されると答えを作話し、その場しのぎの答えが契約に凍結される。

作話された条項（作話条項）の行き先を認可台帳で追うと、雑音より悪いことが分かる。

- 偽の右全域性圧: 誰も欲していない条項が未実現条項として警報を鳴らし、雑音を増やす。
- 偽認可: 本来「孤立」として浮くべき変更が、偽の意図により「認可済」になる。これは偽信号であり、警報疲労すら起こさず黙って較正を狂わせる。
- 作話条項は記録された仮定より悪い。仮定は「可逆・要検証」のマークを持つが、作話条項は「ユーザーの意図」のマークを持ち、下流の誰もそれを疑わない。作話条項とは意図側の較正切れ計器である。較正の切れた計器は計器なしより悪い。

## 3. 発火条件

発火判定を内観（「曖昧だと感じるか」）に置いてはならない。一貫性圧の下では、すでに片方の解釈で不足を無音に補完したモデルこそ「複数の解釈がありうる」と申告しない。内観述語は readiness 閂と同じ null 系譜に入る。

操作化には trust-collapse §5.4 の仕様の不確かさを使う。すなわち、脱相関した解釈者の集団が合理的に下しうる判定のばらつき（統計的評価）、または構造的徴候（適用範囲の「含まないもの」の欠落、曖昧な語彙、量化子の不在。構造的評価）である。判定分布は統計量であり自己申告ではないため、P4 汚染がない。

質問の発火には次の3条件をすべて満たすことを要求する。

1. 割れが実在する。脱相関した解釈者が異なる値で埋める（統計的評価）、または構造的徴候がある（構造的評価）。
2. 割れが下流で「孤立」「保留」として着地する見込みがある。着地しない割れは監査に現れない差であり、質問に値しない。
3. 答えの消費者を指名できる。どの台帳、どの検査、どの分岐が答えを使うかを発火前に言えない質問は儀式である。

割れは必要条件であり十分条件ではない。解釈者はユーザーが選好を持たない次元でも割れる。don't-care の検出は発火判定ではなく回答形式（§5）が担う。

## 4. 停止条件

停止規則は目的適合（trust-collapse §5.4）である。

> 条項の精密化は、精密化によって上がる SN比が、精密化のコスト（質問の作成、人間の注意、警報疲労リスク）を超えるときにだけ行う。

契約の粒度は理想ではなく、下流の計器の検査精度が決める。どの計器も遵守と違反を区別できない精度の選好は、抽出しても死蔵される（消費者不在の field と同じ経路）。抽出してはならない。

完備化禁止（trust-collapse §2, §5.3）が上限を与える。質問の追加で雑音を埋め尽くそうとする運用は、タスクごとの契約細分化による雑音埋めと同じ規律違反である。

停止は、未解消の無知を放置することではない。未解消の無知は次のいずれかに保存され、衝突が予定される。

- 仮定 + 意図ミラー提示（`Readiness record` の仮定欄）
- 許可条項（§5）
- deviations log 経由の再交渉点送り（§6）

## 5. 回答の三値保存形式

質問への回答は次の三値のいずれかとして保存する。

- 制約条項: 実在する選好。契約に凍結し、下流の検査対象にする。
- 許可条項: don't-care の保存形。「この同値類のどれも認可済」。恣意的な一点を制約として凍結せず、同値類全体を認可する。SN比を上げ（その類の変更が今後「孤立」に落ちない）、偽制約を作らない。再発する don't-care 類は常設許可規則へ収穫する（trust-collapse §2 の還流経路）。
- 仮定: 判断の先送り。可逆マーク付きで記録し、意図ミラーまたは検証で衝突を予定する。

すべての質問に「こだわりなし——そちらで決めて許可条項として記録」を一級の回答選択肢として常設する。これは don't-care 領域の検出器であり、作話を制約条項として凍結する経路を回答形式そのものから消す。停止を prose の義務や自己申告に委ねず、形式に埋め込む。

## 6. タイミング

- 高 fan-out の質問（1つの答えが複数の設計セクションを変える）は、契約凍結境界の直前に集中させる。早すぎる質問は先行調査や別の答えで無効化され、注意予算の空撃ちになる。遅すぎる質問は答えの適用コストが跳ね上がる。
- 実装中に発見された新しい unknown は、都度質問に昇格させない。deviations log に記録し、次の再交渉点でまとめて意図ミラーに載せる。失敗記録 20260705-1118（再交渉への正規経路を持たない実行者が存在しない作業単位を発明した）は、必要なのが質問頻度の引き上げではなく再交渉経路の常設であることを示す。

## 7. レジーム適合

質問で直接確かめられるのは、命題的無知（真偽を持つ事実・前提を把握していないこと）である。非命題的無知（体験や提示を通さないと基準を認識できないこと）が優位のレジーム——試作の外観、文体、体感を探索する作業が典型——では、成果物そのものが最安のプローブであり、質問ゼロが §3〜§4 と同じ規則から導出される。「作って見せて反応を見る」ループの1周は、言語を介した質問1個より安く、かつ忠実である。

「質問する」対「自走する」の二値対立は誤った定式化である。regret と衝突コストで価格付けされた単一の規則が、質問集中レジームと質問ゼロレジームの両方を覆う。モードスイッチは要らない。

## 8. 既存の発火規則の操作化

`.chezmoitemplates/opencode/AGENTS.md` の文脈充足条件は、各文脈層に「同じ依頼から複数の妥当な成果物が考えられる場合、…が変わる判断点をそれぞれ判断項目として扱い、確定できない場合は文脈の不足として扱う」という条件を持つ（依頼文脈の判断項目、2026-07-07 時点）。

- 評価: この条件が名指しする量は正しい。「複数の妥当な成果物が考えられる」は §5.4 の統計的評価（独立した解釈者が異なる成果物を作りうる）の folk 版であり、後続の列挙（形式、入口、対象範囲、成功条件、品質基準など）は成果物候補間の diff を取るべき軸のリストとして読める。
- 欠陥: 現状は「考えられるか」という内観述語を、同一文脈の同一モデルが評価する。衝突が予定されていない prose であり、不足を補完した本人に、その不足の存在を訊く構造を持つ（§3 冒頭と同じ機序）。
- 修正の形: 内観を提示義務に置き換える。「複数考えられる」と主張するには、乖離した成果物スケッチを2〜3個実際に提示しなければならない。「一意である」という主張は、脱相関した第二解釈者（fresh context またはモデル族跨ぎ）が同じ要件から別のスケッチを提示すれば反証される。これは捏造コスト反転の構造を持つ——曖昧性の主張には変種の実在が要り、一意性の主張は反証可能になる。スケッチ間の diff が測定された割れであり、diff の走る軸は既存の判断項目列挙にそのまま着地する。
- 副次効果: 変種提示型の質問（変種つき意図ミラー)は、開放質問より注意予算に安く（想起ではなく再認）、作話耐性が高く、「B は違う、そういうのじゃない」という反応を通じて非命題的無知に部分的に届く。操作化は発火判定を改善するだけでなく、質問チャネル自体を一段安い衝突形式に置き換える。

## 9. 較正

質問チャネル自体を計器台帳の対象にする。較正指標は回答の覆り率——質問で得た条項が、後の手戻りで「やはり違う」と覆される率——である。覆り率の高い質問クラスは作話を採掘している証拠であり、質問チャネルの較正切れとして扱う。測定経路（手戻り記録と条項の突合）は未設計である（§10）。

## 10. 未決事項

変更候補。いずれも null 結果の教訓（発火測定なしに新しい分類ゲートを足さない）に従い、導入前に発火測定または blind eval を予定する。

1. `grill-me` への三値保存形式の追加。特に「こだわりなし→許可条項」の一級選択肢化。
2. 文脈充足条件（AGENTS.md 判断項目）の提示義務化。内観述語から変種 exhibit への置換。
3. 覆り率の測定経路。手戻り・レビュー往復と条項の突合方法。

経験的な問い。

- 変種提示のコスト（スケッチ2〜3個の生成）が質問価値を上回るのはどの頻度か。
- 判定の割れの安価な近似はどこまで届くか。fresh context 1本で足りるか、族跨ぎが要るか（文脈の脱相関と事前分布の脱相関の差）。
- 許可条項の成長曲線は常設許可規則と同様に有限で緩やかか。
