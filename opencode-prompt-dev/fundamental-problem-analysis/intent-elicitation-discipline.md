# 意図抽出における発見・提示・閉鎖の規律

## 目的

本書は、実装前に判断材料を増やし、露出した実装上の選択を処理する規律を定める。対象は、質問だけではない。正本調査、実行、計測、例示、差分、試作、実利用も発見経路に含む。

上位の不可避制約は `opencode-prompt-dev/fundamental-problem-map.md` に従う。基準化、判断、実現、実行、評価の五つの変換境界と、具体化の停止規則は `opencode-prompt-dev/fundamental-problem-analysis/trust-collapse.md` に従う。本書は、そのうち発見、提示、露出した判断の閉鎖を扱う。

完全な意図の復元、重要判断の完全列挙、Unknown Unknown の不存在は扱わない。五つの変換境界を使う発見経路も、現時点では未検証の実験候補であり、恒久promptの規則ではない。

## 保証する範囲

実行した有限の発見経路を集合 `R`、`R` から露出した受け入れ可否に関係する実装判断を集合 `D_R` とする。

`pass`は、露出した判断の処理を終え、実装へ進めるというreadiness判定である。本規律で `pass` が保証する範囲は次までである。

```text
すべての d ∈ D_R について、
  一つの選択が固定されている
  または
  選択集合 A_d が認可されている

かつ、D_R に未処理の判断がない
```

この保証は、実行した探索に相対的である。次は保証しない。

```text
D_R が、実際に重要な全判断の集合と一致する
```

したがって、`context-clarification` の責務は「全意図を発見すること」ではない。露出した判断を無認可のまま実装へ送らないことである。

## 判断材料と処置を分ける

### 判断上の役割と確定状態

判断材料は、判断上の役割と確定状態を別々に保存する。

| 軸           | 値       | 意味                                                     |
| ------------ | -------- | -------------------------------------------------------- |
| 判断上の役割 | 制約     | 観測結果の受け入れ可否を決める。                         |
| 判断上の役割 | 許可     | 実装者が追加承認なしに選べる範囲を示す。                 |
| 判断上の役割 | 選好     | 受け入れ可能な候補を順位づける。                         |
| 確定状態     | 確認済み | 認可元または正本により内容を確認した。                   |
| 確定状態     | 仮定     | 未確認命題を意識的に暫定採用し、反証時の戻り先を持つ。   |
| 確定状態     | 未確認   | 命題を暫定採用せず、質問、調査、提示、後段の観測へ送る。 |

`仮定`は抽出時に発見される内容ではない。次の順序で処理した結果として保存する。

```text
命題を抽出する
  → 確認済み、未確認、競合ありを判定する
  → 質問、調査、提示、暫定採用、後段送りを選ぶ
  → 暫定採用した命題だけを仮定として保存する
```

競合ありは処置を選ぶための診断であり、新しい確定状態ではない。

### 許可、裁量、未解決、仮定

同じ「実装者に任せる」ように見える処置でも、成立条件は異なる。

| 種別                   | 成立条件                                                                | 処置                                               |
| ---------------------- | ----------------------------------------------------------------------- | -------------------------------------------------- |
| 明示された許可         | ユーザーまたは権限を持つ認可元が、選択集合を明示した。                  | 認可元、範囲、適用条件を保存する。                 |
| 既存契約から導ける許可 | 固定済みの制約、許可、invariantから選択集合を導ける。                   | 導出根拠と範囲を保存する。                         |
| 低水準の実装裁量       | 既存の所有境界とinterface内で、どの選択も既知の受け入れ判断を変えない。 | 実装の所有先へ委ねる。仮定として保存しない。       |
| 未解決の判断           | 受け入れ判断を変える選択が残り、固定も認可もされていない。              | `pass`を止めるか、期限付きの再判断経路を固定する。 |
| 仮定                   | 未確認事実を、依存先と反証経路を明示して暫定採用した。                  | 仮定、依存する判断、反証時の戻り先を保存する。     |

低水準の実装裁量が受け入れ可否、公開interface、永続状態、実経路のいずれかを変えると判明した場合、その選択は `D_R` の判断として扱う。

## 許可の成立条件

許可は、反対証拠の不在から導かない。

```text
ユーザーが反対していない
モデルが問題ないと判断した
一般的な既存パターンである
```

これらだけでは認可元にならない。

実装者へ選択集合 `A` を委ねる場合、現在固定した契約に対して次を示す。

```text
すべての x ∈ A が、既知の受け入れ条件を満たす
```

示せない場合は `A` を狭めるか、一つの選択へ固定する。許可範囲外の選択は、その許可では正当化されない。直ちに禁止を意味するとは限らないが、実装へ進む前に認可境界へ戻す。

「こだわりなし」や「そちらで決めてよい」という回答も、提示済みの選択範囲に対する明示的な許可として記録する。提示していない判断全体への包括許可へ拡張してはならない。

## 発見方法を選ぶ

知識配置図は、主体面の不一致を説明する診断語彙である。Known Unknown、Unknown Known などの配置だけを根拠に発見経路を発火してはならない。提示前には、既に存在する暗黙の選好と、まだ形成されていない選好を区別できないためである。

命題的無知と非命題的無知の区別は、観測方法の選択に使う。

| 無知の種類   | 主な観測方法               |
| ------------ | -------------------------- |
| 命題的無知   | 質問、正本調査、実行、計測 |
| 非命題的無知 | 例、差分、試作、実利用     |

repositoryや公開一次資料で決まる事実は調査する。ユーザーだけが決められる判断は質問または提示へ送る。どちらか不明な場合は、情報の所有先を先に確かめる。

## 提示の忠実度を段階的に上げる

非命題的無知を扱う提示は、次の順に忠実度を上げる。

1. 結果差の説明
2. 局所的な例または差分
3. 模型または画面断片
4. 実行可能な断片
5. 統合経路での体験

低い忠実度で判断できない場合に一段上げる。この順序は「最小の提示で必ず判断できる」ことも、「最安の観測操作を事前に選べる」ことも保証しない。

A/B提示は、特定の差を判断してもらう局所手段として使う。AとBが共通の盲点を持つ場合や、候補Cがある場合を排除できないため、重要な判断軸を完全に露出した根拠にはしない。

提示後に保存するのは、提示内容、提示した選択範囲、その時点で得た認可、禁止、選好である。回答を「以前から存在した潜在意図の発見」として保存してはならない。提示順序や表現による誘導も消せないため、何を見せて得た判断かを残す。

## 露出した判断を閉じる

`D_R` の各判断は、実装前に次のどちらかへ処理する。

1. 一つの選択へ固定する。
2. 選択集合を認可する。

`Requirement contract`へ反映する場合は、最低限次を記録する。

- 固定した選択、または認可した選択集合
- 選択理由
- 認可元
- 依存する仮定
- 認可範囲外へ出た場合の戻り先

後段へ判断を送る場合も、送付までに許される行為の範囲を固定する。さらに次を記録する。

- 再判断する主体
- 判断に使う観測
- 再判断の期限
- 何を固定する前に認可境界へ戻るか

これらがない後段送りは未解決判断の先送りであり、処理完了に数えない。

実装中に受け入れ可否へ関係する新しい選択が露出した場合、その選択を新たな発見結果として契約固定境界へ戻す。実装者の内部判断だけで閉じてはならない。

## 二つの停止条件

### 探索範囲の停止

探索範囲は、今回選択した有限の発見経路 `R` を実行した時点で止める。ここから主張できるのは、`R` を実行し、`D_R` を得たことまでである。

候補全集合の同定、追加境界の不存在、重要判断の完全列挙は主張しない。探索を止めることと、探索が完全であることを同一視してはならない。

### 露出済み判断の具体化停止

`D_R` の判断をどこまで具体化するかは、`trust-collapse.md` の「完備化を避ける停止規則」に従う。各項目を個別に評価し、現在の判断を変えない項目は具体化しない。判断を変える項目は、損失、後段での検出性、可逆性、後続判断への伝播を評価し、レビュー面を縮小する効果が追加確認の費用を上回る範囲で具体化する。

- 現在の判断を変えるか。
- 誤りによる損失が無視できるか。
- 後段で検出しやすいか。
- 取り消しやすいか。
- 後続判断へ伝播するか。
- 追加確認の費用より、人間のレビュー面を縮小する効果が大きいか。

探索範囲の停止と具体化停止は別の条件である。一方を満たしたことを、他方を満たした根拠にしてはならない。

## 五つの発見経路は実験候補とする

次の経路は、五つの変換境界を有限の探索方向として使えるかを確かめるための一時的な実験契約である。発火候補は仮説であり、恒久promptへ入れる規則ではない。各経路の具体化は、前節の停止条件で止める。

### 基準化

- 入力: 依頼、対象、環境、認可元、既存の契約候補。
- 観測する差: 目的と代理指標、規範的役割、強さ、適用範囲、認可元の差。
- 出力: 制約、許可、選好の候補と、確認または処置が必要な判断。
- 発火候補: 数値、件数、固定形式、代理指標を契約へ加える場面。
- 具体化の停止: 意味または強さの差が受け入れ判断を変えなくなった地点。
- 主張できる範囲: 選択した入力と条項の意味差を検査したこと。
- 主張できない範囲: 暗黙の制約、許可、選好を全て発見したこと。

### 判断

- 入力: 固定した制約、許可、選好と、露出した実装候補。
- 観測する差: 一意に決まらない選択、局所解と一般解、選択理由、損失条件の差。
- 出力: 固定または認可が必要な実装判断と主要な代替案。
- 発火候補: 複数の実装方法がscope、一般化範囲、後続判断を変える場面。
- 具体化の停止: 主要な代替案の差が固定済み条件の合否を変えなくなった地点。
- 主張できる範囲: 列挙した候補が、名指しした判断軸で異なること。
- 主張できない範囲: 代替案集合が完全であること、局所解が一般解であること。

### 実現

- 入力: 固定または認可した選択、変更候補、invariant、状態とデータの所有先。
- 観測する差: 変えるもの、保つもの、実装しないもの、書き込み先、所有先の差。
- 出力: 必要な変更、禁止する変更、維持条件、所有先の候補。
- 発火候補: 複数の書き込み面、状態所有者、生成物、維持条件が関係する場面。
- 具体化の停止: 変更、不作為、維持条件の違いが受け入れ判断を変えなくなった地点。
- 主張できる範囲: 列挙した変更面と所有先を検査したこと。
- 主張できない範囲: 隠れた書き込み経路や必要変更が存在しないこと。

### 実行

- 入力: actor、tool、entry point、実経路、環境、正常経路、failure path。
- 観測する差: 想定経路と到達可能な実経路、単体経路と統合経路、対象環境と検証環境の差。
- 出力: 実行可能性、到達不能、未検証経路、環境依存に関する判断。
- 発火候補: 結果がactor、tool、入口、platform、統合経路に依存する場面。
- 具体化の停止: 実利用経路の選択または合否を変える差を特定した地点。
- 主張できる範囲: 名指しした経路と環境で観測した結果。
- 主張できない範囲: 全ての入口、環境、failure pathで同じ結果になること。

### 評価

- 入力: 受け入れ条件、検証方法、消費者文脈、観測結果、支持する範囲。
- 観測する差: 目的と代理指標、受け入れ条件と検証方法、観測範囲と結論の差。
- 出力: 条件と検証の対応、未支持の主張、再判断が必要な仮定。
- 発火候補: test成功や代理指標を受け入れ判断へ使う場面。
- 具体化の停止: 観測が支持する条件と範囲を限定できた地点。
- 主張できる範囲: 名指しした観測が、限定した条件を支持すること。
- 主張できない範囲: 成果物全体の品質、未観測範囲、別の消費者文脈への一般化。

## 計画の変更条件: 較正、盲検評価、採否

### 既知事例で手順を較正する

既知事例は、入力、不要質問、停止点を具体化する開発用資料として使う。有効性の証明やhold-outには使わない。

- paneler: 未認可の代理指標と、actor・toolが到達不能な経路。
- MatMul: 現在の入力に効く局所解から一般解への無認可な拡張。
- CMakePowerShell: 下位scriptの成功を、消費者が通る統合経路の成功として扱う誤り。

### 汚染を避けて盲検評価する

既知3事例を除外し、次を凍結する前に汚染と代表性を確認する。

- 未使用の失敗事例
- 観測済み失敗族から作る派生事例
- 追加質問、調査、保留を要しない正常対照

現行の `context-clarification` と、発見経路候補を加えた版を比較する。最低限、次を測定する。

- 実装前に検出した高損失の失敗
- 不要な質問、調査、保留
- 誤った契約条項
- 人間が読む量と時間
- 入力量、出力量、turn、所要時間
- 実装後の手戻り
- 同じ失敗への回帰利用可能性

### 現在の測定が示す範囲

`W6-RERUN-2` では、閉じた合成scenario上で次を観測した。

- doom-loop制御率: 25.0%から83.3%
- contract逸脱: 13件から0件
- validation入力文字数: 97.1%増加
- validation出力文字数: 56.6%増加
- 所要時間中央値: 34.9%増加
- hold-out critical: 4件失敗
- 採用判定: 不合格

この測定が示すのは、現行review構造が特定の工程制御を改善する一方、入力、出力、時間の負荷を増やし、固定した採用条件を満たさなかったことまでである。五つの発見経路の有効性、system全体の正確なtoken費用、成果物品質が83.3%へ上がったことは示さない。

### 採用条件

発見経路候補がhold-outで検出力を改善せず、費用だけを増やした場合は撤回または縮小する。改善を確認した場合だけ、次の順で反映する。

1. `context-clarification` の責務と発見手順
2. `Requirement contract` の固定選択と認可範囲
3. `requirement-review` の対応検査
4. `implementation` から新規判断を戻す経路
5. AGENTS.md の文脈収集規則との接続
6. 回帰fixtureと較正手順

採否が決まる前に、新しいstatus、schema、保証水準を追加してはならない。
