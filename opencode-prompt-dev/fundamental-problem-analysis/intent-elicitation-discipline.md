# 意図抽出の発火と停止の規律

## 0. 本報告の位置づけ

- 本報告は、Known Unknown にある事項を質問によって Known Known へ移す操作（以下、意図抽出）について、発火条件、停止条件、回答の保存形式、較正を固定する。
- 上位文書は `opencode-prompt-dev/fundamental-problem-map.md`（委任の4面、知識配置図、根源課題 P1〜P5、衝突の予定）。並列文書は `opencode-prompt-dev/fundamental-problem-analysis/trust-collapse.md`（入力から受け入れ判断までの五つの変換境界、監査装置に求める性質、完備化を避ける停止規則）。本報告は、§3.1「依頼・対象・環境から判断材料を作る」のうち、質問によって入力を増やす操作を詳しく扱う。
- 本報告で使う「作話条項」「二軸保存形式」「覆り率」は、意図抽出を分析するための局所語彙である。`trust-collapse.md` の正本語彙として再定義せず、同文書が所有する変換境界と停止規則を参照する。
- 発端は 2026-07-07 の分析セッション。外部観察（コーディングエージェントが曖昧な指示を質問で解消せず実装コストの低い解釈を勝手に選ぶ、という不満と、grill 型ツールの流行。出典参照）を契機に、「意図抽出はいつ行い、どこで止まるべきか」を `trust-collapse.md` §3.1と停止規則へ接続した。

## 1. 中心命題

意図抽出の終端状態は、Known Unknown 象限の空化ではない。空化は P1（意図の伝達不能性）により原理的に不可能であり、空化を目標に置くと停止規則が定義できない。

終端状態は次で定義する。

> 認識済みの未確定事項について、質問で閉じるものと、質問より安い衝突へ送るものの処遇が決まった状態。

意図抽出の仕事は unknowns の削減ではなく、認識済みの未確定事項を最安の衝突経路へルーティングすることである。質問はルーティング先の1つにすぎず、人間の注意予算（増設不能な最上位統治予算）を消費する高価な経路である。

## 2. 故障モード: 選好の作話は条項生成境界の故障である

`trust-collapse.md` §3.1は、制約、許可、選好と確定状態を区別し、認可元にない精度や固定書式を規範的内容へ足さないことを求める。過剰な意図抽出は、ユーザーが選好を持たない次元へ質問し、その場で作られた回答を制約として固定することで、この境界を壊す。

意図面には「まだ決めていないが決められる」領域の外側に、選好そのものが存在しない領域がある。そこへの質問は知識を転送しない。選好を製造する。人間は質問されると答えを作話し、その場しのぎの答えが契約に凍結される。

作話された条項（作話条項）は、単なる雑音より悪い。

- 誰も求めていない条項が、実装と検証の対象を増やす。
- 本来は裁量として露出すべき変更が、作話された回答によって認可済みに見える。
- 明示した仮定は可逆性と要検証性を残せるが、作話条項は「ユーザーの意図」として固定され、後段で疑われにくい。

## 3. 発火条件

発火判定を内観（「曖昧だと感じるか」）に置いてはならない。一貫性圧の下では、すでに片方の解釈で不足を無音に補完したモデルこそ「複数の解釈がありうる」と申告しない。内観述語は readiness 閂と同じ null 系譜に入る。

本報告では、内観の代わりに、複数の具体的な成果物候補が実際に割れること、または適用範囲、語彙、量化子などに構造的な不足があることを発火候補として観測する。この操作化は本報告の提案であり、`trust-collapse.md` が所有する定義ではない。

質問の発火には次の3条件をすべて満たすことを要求する。

1. 割れが実在する。別の具体的な成果物候補が異なる値を持つ、または構造的な不足がある。
2. 割れが現在の判断または受け入れ可否を変え、`trust-collapse.md` §8 の停止規則の少なくとも一つに該当する。
3. 答えの消費者を指名できる。制約による合否、許可による選択の認可、選好による候補比較、仮定に依存した再判断のどこで答えを使うかを発火前に言えない質問は儀式である。

割れは必要条件であり十分条件ではない。解釈者はユーザーが選好を持たない次元でも割れる。don't-care の検出は発火判定ではなく回答形式（§5）が担う。

## 4. 停止条件

停止規則は `trust-collapse.md` §8 に従う。

> 質問は、現在の判断を変えうる重要な割れを後段より安く閉じ、人間のレビュー面を縮小する場合に限る。

契約の粒度は理想ではなく、後段の検証が遵守と違反を区別できる範囲で決める。回答が、制約による合否、許可される選択範囲、候補間の順位、仮定に依存する再判断のいずれも変えない場合、質問で抽出しても死蔵される。質問してはならない。

`trust-collapse.md` §8 の完備化禁止が上限を与える。質問の追加で不確かさを埋め尽くそうとしてはならない。

停止は、認識済みの未確定事項を放置することではない。質問で閉じない事項は次のいずれかに保存され、衝突が予定される。don't-care は未確定事項ではなく、選好の不在と選択範囲の許可が確認された状態として§5に保存する。

- 仮定 + 意図ミラー提示（`Readiness record` の仮定欄）
- 未確認のまま deviations log 経由の再交渉点送り（§6）

## 5. 回答の二軸保存形式

質問への回答は、規範的内容と確定状態を分けて保存する。一つの回答に複数の内容がある場合、単一の分類へ押し込まず、それぞれを保存する。

規範的内容には次の役割がある。

- 制約条項: 回答が受け入れ可否を決める場合に、認可元と範囲を付けて契約へ固定する。
- 許可: 実装者が追加の承認なしに選べる範囲を保存する。範囲外の選択を禁止するとは限らない。
- 選好: 受け入れ可能な候補の順位を保存する。劣後しても直ちに不合格とはしない。

規範的内容には、次の確定状態を別に付ける。

- 確認済み: 認可元が内容を明示している。
- 仮定: 未確認の内容を暫定的に使用する。反証時に再判断へ戻せるよう、依存先と衝突経路を残す。
- 未確認: 内容を暫定採用せず、後続の調査または再交渉へ送る。

すべての質問に「こだわりなし——そちらで決めて許可として記録」を一級の回答選択肢として常設する。これは don't-care 領域の検出器であり、選好の不在と選択範囲の許可を保存する。作話を制約条項として凍結する経路を回答形式そのものから消し、停止を prose の義務や自己申告に委ねず形式に埋め込む。

## 6. タイミング

- 高 fan-out の質問（1つの答えが複数の設計セクションを変える）は、契約凍結境界の直前に集中させる。早すぎる質問は先行調査や別の答えで無効化され、注意予算の空撃ちになる。遅すぎる質問は答えの適用コストが跳ね上がる。
- 実装中に発見された新しい unknown は、都度質問に昇格させない。deviations log に記録し、次の再交渉点でまとめて意図ミラーに載せる。失敗記録 20260705-1118（再交渉への正規経路を持たない実行者が存在しない作業単位を発明した）は、必要なのが質問頻度の引き上げではなく再交渉経路の常設であることを示す。

## 7. レジーム適合

質問で直接確かめられるのは、命題的無知（真偽を持つ事実・前提を把握していないこと）である。非命題的無知（体験や提示を通さないと基準を認識できないこと）が優位のレジーム——試作の外観、文体、体感を探索する作業が典型——では、成果物そのものが最安のプローブであり、質問ゼロが §3〜§4 と同じ規則から導出される。「作って見せて反応を見る」ループの1周は、言語を介した質問1個より安く、かつ忠実である。

「質問する」対「自走する」の二値対立は誤った定式化である。手戻り損失と衝突コストで価格付けされた単一の規則が、質問集中レジームと質問ゼロレジームの両方を覆う。モードスイッチは要らない。

## 8. 既存の発火規則の操作化

`.chezmoitemplates/opencode/AGENTS.md` の文脈充足条件は、各文脈層に「同じ依頼から複数の妥当な成果物が考えられる場合、…が変わる判断点をそれぞれ判断項目として扱い、確定できない場合は文脈の不足として扱う」という条件を持つ（依頼文脈の判断項目、2026-07-07 時点）。

- 評価: 「複数の妥当な成果物が考えられる」という条件は、条項化前に割れうる判断軸を露出する。後続の列挙（形式、入口、対象範囲、成功条件、品質基準など）は、成果物候補間の差分を取る軸として読める。
- 欠陥: 現状は「考えられるか」という内観述語を、同一文脈の同一モデルが評価する。衝突が予定されていない prose であり、不足を補完した本人に、その不足の存在を訊く構造を持つ（§3 冒頭と同じ機序）。
- 修正の形: 内観を提示義務に置き換える。「複数考えられる」と主張するには、乖離した成果物スケッチを2〜3個実際に提示しなければならない。「一意である」という主張は、脱相関した第二解釈者（fresh context またはモデル族跨ぎ）が同じ要件から別のスケッチを提示すれば反証される。これは捏造コスト反転の構造を持つ——曖昧性の主張には変種の実在が要り、一意性の主張は反証可能になる。スケッチ間の diff が測定された割れであり、diff の走る軸は既存の判断項目列挙にそのまま着地する。
- 副次効果: 変種提示型の質問（変種つき意図ミラー）は、開放質問より注意予算に安く（想起ではなく再認）、作話耐性が高く、「B は違う、そういうのじゃない」という反応を通じて非命題的無知に部分的に届く。操作化は発火判定を改善するだけでなく、質問チャネル自体を一段安い衝突形式に置き換える。

## 9. 較正

`trust-collapse.md` §4.3と§9に従い、質問チャネル自体も較正対象にする。候補指標は回答の覆り率——質問で得た規範的内容またはその確定状態が、後の手戻りで「やはり違う」と改められる率——である。覆り率の高い質問クラスは作話を採掘している可能性がある。ただし、測定経路（手戻り記録と回答の突合）は未設計である（§10）。

## 10. 未決事項

変更候補。いずれも null 結果の教訓（発火測定なしに新しい分類ゲートを足さない）に従い、導入前に発火測定または blind eval を予定する。

1. `grill-me` への二軸保存形式の追加。特に「こだわりなし→許可」の一級選択肢化。
2. 文脈充足条件（AGENTS.md 判断項目）の提示義務化。内観述語から変種 exhibit への置換。
3. 覆り率の測定経路。手戻り・レビュー往復と回答の突合方法。

経験的な問い。

- 変種提示のコスト（スケッチ2〜3個の生成）が質問価値を上回るのはどの頻度か。
- 判定の割れの安価な近似はどこまで届くか。fresh context 1本で足りるか、族跨ぎが要るか（文脈の脱相関と事前分布の脱相関の差）。
- 許可が課題ごとに発散せず、再利用可能な範囲へ収束するか。
