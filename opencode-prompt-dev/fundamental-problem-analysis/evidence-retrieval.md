# 根拠収集と知識検索の調査報告

## 概要

コーディングエージェントに、正しい根拠を、正しい情報源から、正しい順序で、監査可能な形で検索させる方法を調査した。前回の『コンテキスト管理と compaction の調査報告』が「文脈をどう保持するか」を扱ったのに対し、本報告はその双対にあたる「必要な文脈と根拠をどう集めにいくか」を扱う。

発端は、GMOD 向けカスタムシェーダーの実装をエージェントに委任した際の失敗である。仕様の不遵守、既存実装の把握漏れ、GMOD 公開仕様の把握漏れという3つの失敗モードが観察された。このうち後ろ2つは、プロンプトの工夫だけでなく、エージェントが事実を収集しやすい環境を整えることでも再発を防げる。本報告は、公開情報の探索、既存ツールの比較、検索力学の整理、そして既存の作業規則とのすり合わせを記録する。

### 結論

- 検索失敗の根本原因は、表現のミスマッチと、根拠の振り分け・正本判断の失敗にある。ベクトル類似度は関連性ではなく、因果や呼び出し連鎖を表現できない。
- 全情報源を1つの自然文検索に統合しない。知識の種類ごとに表現を変え、ルーティング層で繋ぐ。
- 既存の作業規則は、問いを「何級の根拠が要るか」へ絞る前端と、出典付きで記録し矛盾を突き合わせる後端をすでに持つ。真の未整備領域は、両端に挟まれた検索の中央と、検索結果を証拠として扱う台帳の細部である。
- ルーティング層の中核は、coverage fallback（探す順）と canonical trust（矛盾時にどの情報源を信じるか）という2つの独立な軸、さらに absence を情報として扱う規律の3つで構成される。
- 「ある情報源に規則が存在すること」と「それが今回の課題で全粒度に正しく適用され解決済みであること」は同値ではない。規則は、どこに置かれ、いつ発火するかまで制御しないと効かない。

## 背景

### 発端: GMOD シェーダー実装での検索失敗

`.opencode/work/fuck.jsonl` に記録されたセッションで、GMOD 向けカスタムシェーダーの実装をエージェントに委任し、3つの失敗モードに陥った。

1. 指示した仕様を守らなかった。
2. 既存実装の詳細仕様を把握しきれなかった。
3. GMOD の公開外部仕様を把握しきれなかった。

1つ目は指示追従の問題で、コンテキスト管理の側面が強い。2つ目と3つ目は、エージェントが自ら事実を集めに行く過程の問題である。公開情報の探索は検索エンジンの精度向上でかなりカバーできるが、プロジェクトのドメイン知識とフレームワークの詳細仕様となると、Wiki を読ませるか、Markdown の束を探させるしかなかった。ripgrep で関連語彙を検索すれば分かることも多いが、検索範囲か検索語彙のどちらかを間違えると、仕様が書いてあっても拾えない。

### 問題提起: ドメイン知識と外部仕様の発見可能性

断片化した情報源が、エージェントが自然には探しに行かない場所に散らばっている。Lua の API 仕様は Facepunch Wiki が `?format=json` で記事を JSON 取得できるが、シェーダーのレジスタ割り当て一覧はデバッグ用に書いたシェーダー `debug_ps30.hlsl` のコメント欄にしかない。後者は GMOD Wiki にも載っていない公益性の高い共通仕様で、別途 GitHub Wiki にまとめ、ローカルにクローンして使う。プロジェクト固有の設計や計画は、また別に日本語で管理する。

結果として、参照すべき情報源は性質の異なる3つに分かれる。

- A. GMOD Wiki（HTTP で取得できる大規模な生リファレンス）
- B. SourceDocuments Wiki（ローカルの Markdown 束。小規模で高シグナルな合成知識）
- C. プロジェクト固有資料（ローカルの日本語設計書。小規模でプロジェクト限定の合成知識）

3つを1つの自然文検索に統合し、Exa Web Search のように振る舞わせるのがいちばん楽に見える。しかし後述する通り、知識の種類が違えば正しい表現も違い、無理な統合は構造情報を殺す。

### 調査の方向

2つの軸で整理した。

- 検索の力学: 語彙のずれをどう吸収するか、粒度をどう制御するか、似た別物をどう弾くか。
- 情報源の設計: 何をどこに置き、どう読ませるか、複数情報源をどう使い分けるか。

この2軸に沿って、公開情報の探索、既存ツールの比較、検索力学の既知手法、そして本リポジトリの既存作業規則とのすり合わせを進めた。

## 検索失敗のメカニズム

### 類似度は関連性ではない

ベクトル検索は「意味が近い」ものしか拾えない。因果で繋がっていても語彙が被らない情報は届かない。認証の不具合が、共通語彙を持たない設定ファイルに行き着くことは珍しくない。複数の文脈エンジニアリングの記述が、同じ点を指している。「similarity is not relevance」であり、ベクトル類似度は因果も呼び出し連鎖もモデル化しない。

これが検索力学の根本制約になる。検索システムは常に何かを返す設計になりがちで、corpus に存在しない事項を問うても、それらしい類似度スコア付きのチャンクを返してしまう。hallucination は生成層だけでなく検索層からも紛れ込む。

### 検索精度の2つの崩れ方

検索のランキングが弱いと、2つの災難のどちらかに崩れる。1つはノイズがコンテキストウィンドウを圧殺する方向、もう1つは正解を取りこぼす方向である。Sourcegraph が報告した実例では、100万行のモノレポで grep を1回走らせた結果、約4000件がコンテキストウィンドウに流れ込み、検索としては動いたが実用上はコンテキストウィンドウに対する DoS 攻撃になった。ランキングと件数の上限なしに検索を文脈に流すのは、自分のコンテキストウィンドウを攻撃するのと同じである。

「LLM の問題に見える失敗の多くは、実は文脈構築、すなわち検索の段階で起きている」。この観察は、前回の失敗事例の2つ目と3つ目にそのまま当てはまる。

### 表現のミスマッチ

同じ知識でも、種類が違えば正しい表現が違う。コードベースは時間的ダイナミクスを持つグラフであり、安定した構造層にはグラフ索引が向く。直前に変更されたファイルには、鮮度が要るためグラフより agentic な検索が向く。蓄積する合成知識は、raw ソースから毎回再導出するのではなく、複利で膨れる wiki として保守するのが正しい。自然文主体の設計資料は、語彙を固定しにくく表記揺れが発生するため、意味の近さを扱う検索が向く。逆にコードの識別子は厳密に区別しなければならず、似た別物を拾い上げるリスクの高い純粋なベクトル検索には向かない。

1つの検索手法で全種類をカバーしようとすると、どこかの種類の表現を殺す。これが「統合しない」理由である。

### 閉世界仮定の罠

1つの情報源を探索して見つからなかった時、「なかった」という結果自体が情報になる。これを「存在しない」と取り違えるのが、閉世界仮定の罠である。

具体例でいうと、Source SDK のソースコードは GMOD の派生元であり、GMOD 側で後から追加されたシェーダーはコードをいくら探しても見つからない。それでもそのシェーダーは GMOD に実在し、Wiki にも記事がある。コードでの不在を存在しないことと取り違えると、ここで見落とす。逆に、ライトマップのサンプル計算のようなエンジン内部の実装詳細は Wiki に載っておらず、ソースコードを解読しないと正しい情報は得られない。この2例は矛盾せず、どの情報源がその質問に権威を持つかが質問の種類で変わるだけである。

各情報源のクエリ結果は、4状態に分かれる。

- present かつ権威あり: 採用する。
- present かつ権威なし: 二次資料として記録し、権威ソースを照合する。
- absent かつ徹底探索済: 「ここにはない」を情報として扱い、別ソースへ進むか、文書化されていない事項と判断する。
- absent かつ探索不十分: 結論保留でより深く探す。

この4つ目を扱うには、「本当にない」を検出できる検索プリミティブが必要になる。dense vector は構造上 absence を検出できず、常に何かを返す。BM25、lexical、symbol の厳密照合だけが本物の空集合を返せる。dense を使うなら、類似度の下限を設けて閾値未満は「なし」として返さなければならない。下限がないと、権威ソースには実は存在しない事項を、別ソースの dense がそれらしく返して「あり」にしてしまうマスキングが起きる。

### 優先順位の2軸

情報源の優先順位を1本の線形リストで表すと、2つの別物が混ざる。1つは coverage fallback、どこにありそうかを順に当たる探索順である。もう1つは canonical trust、複数情報源が矛盾した時にどちらを信じるかという正本判断である。

GMOD Wiki、SourceDocuments Wiki、Source Engine のソースコード、Web 検索、を順に当たる線形 chain は coverage fallback としては妥当で、情報源間でカバーが被らなければ先勝ちで安全に機能する。しかし両方に載っていて意見が違う時、先勝ちでは間違ったほうを掴む。canonical trust は質問の種類別に決める2変数の表であり、「GMOD の追加機能」なら Wiki が正、「エンジン内部実装」ならソースコードが正、「プロジェクト設計判断」ならプロジェクト資料が正、となる。

この2軸を分けておかないと、線形 chain が矛盾時に黙って誤答を選ぶ。矛盾を安全に処理するには、coverage fallback の各段階で、その情報源の absent が徹底したものかを判定し、併せて権威ソースとの突き合わせを行う規律が要る。

## 比較した既存実装

### コード検索系 MCP

`zilliztech/claude-context` は、AST 単位のチャンク化、BM25 と密ベクトルのハイブリッド、Merkle tree による差分再索引を備える。`jeremymefford/agent-context-mcp` は同じ思想の Rust 実装で、multi-repo スコープと、意味・語彙・シンボルの3方式を融合するランキングを持つ。後者の README が失敗の正体を明示している。「エージェントが既存の実装パスを見つけられず、並行する別解をでっち上げ始めるのを防ぐ」。これは前回の失敗事例の2つ目そのものである。

この系統は、既存実装の把握、すなわち失敗モードの2つ目に直接効く。

### ドキュメント・知識オンデマンド系

Cursor の `@docs` が教科書的なパターンである。外部のドキュメントをクロールして索引化し、検索で関連チャンクだけを文脈に注入する。全部を前もって読ませるのをやめる設計で、「文脈予算の浪費」を防ぐ。`joutvhu/knowledge-shelf` が同じ思想の MCP で、マニフェスト単位で検索から該当節だけを取り出す。

この系統は、外部仕様の把握、すなわち失敗モードの3つ目に効く。

### 記憶・由来メタデータ系

`kruschdev/krusch-context-mcp` は、コードベースの現実と記憶を突き合わせてから行動する Zero-Trust Deep Search を持ち、`source_type` や信頼度のメタデータを付与する。`ipiton/agent-memory-mcp` は、型付き記憶、ドキュメント RAG、repo ツールを統合し、由来メタデータを重視する。

この系統は、複数情報源の由来を区別し、矛盾を検出する後端の規律に効く。

### グラフ・構造系

`tae2089/code-context-graph` は Tree-sitter でナレッジグラフ化し、`@intent` `@domainRule` `@sideEffect` の注釈で業務文脈による検索を可能にする。広い自然文質問に対する単発ベクトル検索を「証拠駆動の絞り込み層」で置き換える狙いである。本リポジトリに既に導入済みの Graphify は同系統にあたる。

この系統は、コード上の事実を構造と到達経路込みで確認する要求に効くが、グラフ化は安定した構造層に効く一方で、直近の変更には遅延が生じる。

### Karpathy LLM Wiki

Andrej Karpathy が2026年4月に提示した LLM Wiki パターンは、raw ソース、LLM がコンパイルした wiki、そして `index.md` とスキーマ設定ファイルの3層構造をとる。重要なのは、毎クエリで raw から再導出するのではなく、合成知識を1度コンパイルして保守し続ける複利 artifact にする点である。新しいソースが来たらエンティティページを更新し、矛盾にフラグを立てる。RAG との差はここにある。

このパターンは個人から小規模チームの、bounded なコーパス向けと明示されている。大規模では RAG がなお必要になる。本件の情報源 B と C は規模と性質がこの範囲に一致する。ただし「古い wiki は無いより悪い」点に注意が要る。人間が手書きして放置するとすぐに古くなり、エージェントが古い内容を確信をもって出力する。

### 検索エンジン・プラットフォーム

2026年時点で、主要なベクトルデータベースはハイブリッド検索を標準で持つ。真の BM25 とベクトルを同時に一級市民として扱うのは Weaviate、Vespa、Elasticsearch である。Qdrant、Pinecone、Milvus は BM25 風の語頻度を sparse vector として表現し融合する近似になる。Qdrant の v1.10 Query API は、dense と sparse の RRF 融合に続き ColBERT の多ベクトルで再ランキングするまでをサーバ側1本のクエリでこなす。ColBERT を再ランキング専用に使うなら HNSW 索引を無効化し、索引コストを下げられる。

完成品のプラットフォームが要る場合は RAGFlow が候補になる。ハイブリッド検索、出典付き回答、RAPTOR、GraphRAG、MCP 対応を Docker Compose 1発で入れる。代償は重いインフラで、最低16ギガバイト、実務では32ギガバイトの RAM を推奨する。LlamaIndex は同じ構成を code-first で自前組み立てるライブラリ側にあたり、問いごとの検索手法使い分けを auto_routed や composite retrieval で実装する。

## 検索力学の既知手法

### ハイブリッド検索

BM25 の厳密語彙照合と、密ベクトルの意味照合を、Reciprocal Rank Fusion で融合する。純粋な意味検索は `getUserById` のような厳密識別子を取りこぼし、純粋な語彙検索は言い換えを取りこぼす。両方を併せることで両方を拾う。2026年の実務では、ここに学習型 sparse を加え、上位候補に重い再ランキングを載せる3段構えが定石になっている。

### ColBERT と late-interaction

設計資料が抱える「厳密に把握すべき識別子」と「表記揺れする自然文」の2軸を、1つの仕掛けで同時に満たす手法が ColBERT の late-interaction である。チャンクを1つのベクトルに潰すのではなく、トークンごとのベクトルを保存し、クエリの各トークンと文書トークンの最大類似度を束ねてスコア化する。プール化で潰れる厳密トークン信号を保持したまま、文脈的な意味マッチも効く。

2026年の評価で、この手法は「意味段落の内側で識別子の一致が決め手になる、コードと自然文の混在文書」に特に効くとされている。本件の設計資料は識別子を含む自然文であり、まさにこの用途に当たる。付随的な効能として、どのクエリトークンがどの文書トークンに当たったかを示せるため、監査向けに根拠をハイライトできる。出典追跡の要求を同じ仕掛けで削れる。

ColBERT は万能ではなく、ストレージが単一ベクトルの4倍から10倍かかり、クエリが遅くなる。実務では BM25 かバイエンコーダで候補を絞ってから ColBERT で再スコアする2段構えにする。レアトークンやエラーコードでは依然として BM25 が強く、ColBERT は BM25 を補完するもので置換するものではない。

### RAPTOR と階層検索

粒度の問題、すなわち断片だけだと前提が抜け全文だと文脈が汚染される問題には、RAPTOR の階層検索が別軸の答えになる。チャンクを再帰的にクラスタリングして要約し、木を構築する。葉が詳細、上位ノードが全体像を担い、クエリ時に両方を同時に引ける。素朴な RAG が文書をシュレッダーにかけた状態で、日付の質問には効いても伝記は書けないのに対し、RAPTOR は階層的に意味を保ったまま粒度を行き来できる。索引時に要約コストを前払いし、クエリ時は木を潰して全層から引く。

### recall の梯子

ハイブリッド構成の段階的な効き目を実測した報告がある。BM25 単体の recall@10 を60%とし、密ベクトル単体で71%、両者を RRF 融合すると78%、学習型 sparse を加えて81%、ColBERT で再ランキングすると86%になる。フルスタックは BM25 単体に比べて25ポイント、密ベクトル単体に比べて15ポイント高い。ただしコストは2倍から3倍かかり、クエリ時で1.5倍から2倍になる。質の上がり幅は一貫して10〜25%である。

## 課題の構造化と既存機構のカバー

### 課題の6分類

根本課題を「与えられた問いに対する回答を、正しい根拠を、正しい順序で、監査可能な形で検索すること」と置き、6つの課題に分解する。

1. 問いを根拠要求へ変換する。何についての事実を求めているかを分類し、その問いに答える根拠の種類を決める。
2. 根拠要求から探索計画を作る。情報源の候補、探索順序とフォールバック条件、どこまで探せば未発見と扱うかを決める。
3. 情報源の性質に合った検索を行う。自然文の表記揺れを拾い、厳密語彙を取りこぼさず似た別物を混ぜず、コード上の事実を構造と到達経路込みで確認し、情報の粒度を前提の欠けも汚染もない範囲に調整する。
4. 検索結果を根拠として記録する。出典、範囲、検索語、更新日時を残し、探索した情報源ごとに発見と未発見を区別する。
5. 複数根拠を採否判断する。矛盾を検出し、問いに対する権威の適用範囲に従い採否を決め、採用・補助・未採用を把握できる形にする。
6. 情報源そのものと索引を保守する。情報源の追加・削除・更新に追従し、古い索引や古い根拠を検出し、検索の失敗を検索器・索引・分類規則の改善提案へ繋げる方法を用意する。

### 既存機構の流用可能性

本リポジトリの既存機構がこの6分類のどこを担うかを整理する。文脈収集規則は問いを「何級の根拠が要るか」まで毎ターン申告させ、作業段階の遷移条件を満たさないと次に進めない門を設ける。public-research は公開情報を引用付きで確認し、investigation は repo ローカルの根拠を観測事実として返す。失敗ログの workflow と retrospective-codify は、失敗を規則の改善へ繋げる。

カバーの概要は次の通り。

- 課題1（問いを根拠要求へ）: 文脈収集規則の4層文脈分類と required source class がカバーする。
- 課題2（探索計画）: 未カバー。規則は根拠の級まで決めるが、具体情報源の候補、順序、フォールバック、未発見の閾値までは決めない。
- 課題3（性質に合った検索）: 機械の領域。スキルは検索を呼ぶだけで手法を規定しない。コードの構造確認だけ、investigation と Graphify が部分カバーする。
- 課題4（記録）: 出典・範囲・検索語・更新日時の記録は public-research、investigation、根拠の紐づけ規則がカバーする。情報源ごとの発見・未発見の区別は未カバーである。
- 課題5（採否判断）: 矛盾検出は public-research の突き合わせ手順がカバーする。権威適用範囲は、文脈収集規則が一般優先順位を持つものの、質問タイプ別の正本表までは未符号化である。採用・補助・未採用の把握は、レビュー指摘向けの Review finding record が構造類似を持つが、検索証拠向けの台帳は不在である。
- 課題6（保守）: 失敗から改善への繋ぎだけカバーする。情報源の追加削除、更新頻度の差、古い索引の検出は未整備である。

### 規則の存在と正しい適用は同値ではない

カバー表の「カバー」は、種となる規則がその機構に存在することを意味するだけである。それが今回の課題で、すべての粒度において正しく発火し、解決済みとみなせることとは同値ではない。

public-research が出典付き記録の形式を持つことと、GMOD シェーダーの実装中にエージェントが Wiki の節、ファイルの行、更新日時を毎回正しく残すことは別の話である。後者は、規則が存在するだけでなく、その作業粒度で確実に発火するように置かれていなければ成り立たない。カバーは「種がある」までを示し、「実地で効いている」までは示さない。表の読み取りはこの差を前提にする。

### 適用粒度と配置制御

AGENTS.md に根拠を意識することが明示されていても、それがタスク遂行に必要な事実の探索の、すべての粒度に及ぶとは限らない。ワークフロー全体での根拠意識は、上位階層の規則として効く。しかし個別のファイル1行の事実確認、あるいは1つの検索クエリの出所押さえといった、きめ細かい粒度では、発火する位置に規則が置かれていなければ素通りされる。

規則は存在するだけでなく、どこに置かれ、いつ適用されるべきかまで制御しなければ正しく適用されない。これは本リポジトリのプロンプト階層の問題に直結する。グローバルな AGENTS.md に分岐を書くべきではなく、エージェントのシステムプロンプトやスキル本文、あるいはコマンドの該当段階に、発火すべき位置で規則を置く。検索に関わる規則も、文脈収集規則の中の該当段階、もしくはルーティング層の設定として、発火点に配置し直す必要がある。

## 設計指針

### 3層構造

検索を3層で構成する。

- 層0 必要性の検証: いま何が必要かを問う。文脈収集規則が担う、本リポジトリ独自の強みである。
- 層1 ルーティングと索引: 必要な事実がどこにあり、どの由来かを決める。現状の未整備領域である。
- 層2 表現ごとの検索: グラフ、agentic と grep、wiki、raw 参照を、情報源の性質に合わせて使い分ける。

前回の調査で、文脈収集規則は「必要性を事前に構造化された門で客観化する独自の強み」と結論づけた。検索層はその下請けであり、層0 の判定結果を受けて層1 が行き先を決め、層2 が実際に検索する。3層が揃うと、保持の問題と収集の問題が両方閉じる。

### ルーティング設定

層1 のルーティングは、設定ファイル駆動にする。各情報源に、名前、種別、取得方法、そして「この情報源が正本になる質問タイプ」のメタデータを持たせる。coverage fallback の順序と canonical trust の表を、共にリストで持つ。情報源の追加削除はこの設定を書き換えるだけで済む。Valve Developer Community のように bot 対策で取得できない情報源は、情報源エントリの取得可否を取得不可にしておけば、fallback が自動的にスキップする。

3つの規律を設定に組み込む。第1に、coverage fallback は情報源間でカバーが被らないか、被っても同意している時だけ先勝ちを許す。第2に、canonical trust は質問タイプ別の表で決め、矛盾時には線形順序ではなくこの表が採用を決める。第3に、absence を情報として扱い、各情報源の「見つからない」が徹底探索済かを判定可能にする。そのため、dense を使う情報源には類似度の下限を必ず設ける。

### 証拠台帳

検索結果を証拠として扱う台帳を、Review finding record の構造を検索証拠向けに翻案して作る。各証拠は、出典、範囲、検索語、更新日時、そして状態を持つ。状態は、採用、補助、未採用、未発見の4つである。未発見を状態として持つことで、「徹底探索して見つからなかった」ことを根拠として記録し、次の行き先の判断材料にできる。

台帳は、public-research と investigation の出力形式と整合させる。両スキルが発見の記録を担う一方で、未発見の記録は現在明示的に持たない。台帳のスキーマで未発見を1等級に扱うことで、この隙間を埋める。

## 残る課題と設計指針

### 規模別の領分

ハイブリッドスタックの価値は、コーパスの規模、識別子の密度、multi-hop 性の3つで効いてくる。情報源 B と C が数百ファイル以下の小規模であれば、BM25 か ripgrep と on-demand 読み、それに index.md を組み合わせるだけで、フルスタックの恩恵の大部分は届く。ColBERT や RAPTOR、Qdrant の投資に見合わない。absence の検出も、lexical なら自然に効く。

中規模以上で、識別子を含む自然文が主になる場合は、Qdrant を local で動かし dense と sparse と ColBERT の再ランキングを組むのが費用対効果の最適点になる。真の BM25 が欲しくなったら Weaviate に挿げ替える。RAPTOR は、multi-hop の質問で困ってから RAGFlow か LlamaIndex で追加する。どちらの領分に振れるかは、B と C の実際のファイル数と識別子密度を量れば決まる。この測定は今後の検証項目である。

### 規則の配置と発火制御

既存規則の存在と、今回の課題での正しい適用を区別した結果、規則を発火点に配置し直す作業が設計の本体になる。文脈収集規則のなかで、required source class の判定から具体情報源の候補と順序を導く橋渡しを書く。ルーティングの規則、canonical trust の表、証拠台帳のスキーマは、グローバルな AGENTS.md ではなく、発火する階層に置く。設定ファイルとして情報源リストを持ち、規則はそれを参照して動く形にする。

この配置作業を間違えると、規則はあっても素通りされる。前回の調査が「必要性検証層」として文脈収集規則を評価したのと同じ厳しさで、検索の規則も発火点で検証する必要がある。

### 保守運用

情報源の追加削除、更新頻度の差、古い索引や根拠の検出は、設定ファイルの設計と、Merkle tree やファイル監視による差分再索引で段階的に整備する。小規模で人間が管理する wiki については、編集はファイルを直すだけで済むが、放置すれば古くなる。second-brain-mcp のエディトリアルモデルのように、LLM が保守候補を出し、人間が承認する運用が、日本語で管理しリポジトリ外に置く本件の要件に合う。

### 今後の検証項目

導入後に実測で確認すべき点を列挙する。

- 情報源 B と C の実際の規模。ファイル数と識別子密度を量り、小規模 wiki の領分か Qdrant の領分かを確定する。
- Graphify の実効性。グラフ探索が concept discovery の遅れをどれだけ救うか。鮮度の要る変更への遅延と天秤する。
- Valve Developer Community の取得層の回避策。検索層ではなく取得層の別件として扱う。
- ルーティング規則の発火検証。文脈収集規則の該当段階に置いた規則が、実地で確実に発火するか。
- 証拠台帳の未発見状態。徹底探索済の absent が、次の行き先の判断に実際に使われるか。
