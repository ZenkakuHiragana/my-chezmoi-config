# 信頼崩壊の五境界による既知失敗事例の試行分類

## 0. 目的と有効範囲

本報告は、`trust-collapse.md` が定義する五つの変換境界を、既知の失敗事例の診断に使えるか試行する。失敗頻度の推定、60件全体の網羅分類、失敗ログのstatus変更は行わない。

- 母集団: 2026-07-19時点の `.opencode/local-failure-logs/` 直下にある失敗ログ60件
- 標本: 再現条件、観測、影響、根拠を持ち、五境界をすべて含むように選んだ10件
- 選定方法: 境界の反例探索を目的とした目的標本。無作為抽出でも時系列抽出でもない
- 推論可能な範囲: この10件を境界へ局所化できるか、複数境界を分離できるか
- 推論できない範囲: 母集団での境界別頻度、境界間の一般的な因果順序、未選定50件への適合率

## 1. 分類方法

### 1.1 根拠の種類

- `source_class`: `local_failure_log`
- `evidence_class`: 失敗ログに記録された再現条件、観測結果、影響、引用または実行証拠

一般論や現在の文書構造から失敗原因を補わず、失敗ログに記録された範囲だけを分類根拠にする。

### 1.2 境界の割当

- **主境界**: 記録された証拠から、意味または要求の保存失敗を直接観測できる最も早い変換境界。因果上の根本原因とは限らない。
- **副境界**: 同じ事例で独立に観測された別の保存失敗。存在しない場合は `None` とする。
- **留保**: 中間成果物が残っておらず、より早い境界の成否を決められない場合に明示する。

主境界を一つ選ぶのは集計のためであり、事例全体を一境界へ畳むためではない。

## 2. 分類結果

| No. | 事例                                | 主境界 | 副境界   |
| --: | ----------------------------------- | ------ | -------- |
|   1 | panel出力制約の過剰化               | 3.1    | 3.5      |
|   2 | review cycle再開条件の過剰化        | 3.2    | 3.5      |
|   3 | 図をrenderせず受け入れ              | 3.4    | 3.5      |
|   4 | 未認可の絶対pathを受理              | 3.1    | 3.5      |
|   5 | 読者に不要な識別子をreview通過      | 3.5    | `None`   |
|   6 | review-responseの拡張経路を縮退     | 3.3    | 3.5      |
|   7 | 日本語規則をcommand agentへ反映せず | 3.3    | 3.5      |
|   8 | 目標を狭い完了条件へ置換            | 3.1    | 3.5      |
|   9 | runtime消失をworktree resetと誤認   | 3.4    | `None`   |
|  10 | task固有bootstrapを恒久promptへ昇格 | 3.2    | 3.3、3.5 |

### 2.1 panel出力制約の過剰化

- 観測: 回答の情報量を保ったまま後段で読む負荷を減らす依頼から、参加者の文字数、主張件数、固定形式を制限する出力契約を追加した。
- 主境界3.1: 認可元にない定量上限と固定形式を条項へ追加した。
- 副境界3.5: 元の出力要件を保存しない成果物を受け入れた。
- 根拠: `.opencode/local-failure-logs/20260718-0255-panel-output-constraint-overreach.md:41-51,155-172`

### 2.2 review cycle再開条件の過剰化

- 観測: review cycleの自動継続禁止を、次のcycleには毎回ユーザーの明示承認が必要という再開条件へ強めた。
- 主境界3.2: 規則から一意に決まらない再開判断を、追加の承認gateとして採用した。
- 副境界3.5: promptの実在文言を確認せず、承認待ちを正しい停止状態として扱った。
- 留保: 再開条件を条項化した中間成果物がないため、3.1での意味変質は判定しない。
- 根拠: `.opencode/local-failure-logs/20260717-1441-review-cycle-approval-overread.md:43-50,83-92`

### 2.3 図をrenderせず受け入れ

- 観測: source上の文字列検査だけを行い、利用者が見るMarkdownと図を実renderしなかった。
- 主境界3.4: 実際の消費経路で成果物を表示し、視覚的な挙動を観測していない。
- 副境界3.5: source検査の成功を、表示品質の受け入れ根拠として扱った。
- 根拠: `.opencode/local-failure-logs/20260716-1921-unrendered-diagram-artifact-integrity.md:61,70-82,142-165`

### 2.4 未認可の絶対pathを受理

- 観測: 環境依存の絶対pathを認可する明示要求がないのに、要件として受け入れた。
- 主境界3.1: ローカル環境情報を、恒久成果物で使える制約へ強めた。
- 副境界3.5: source_classと認可範囲を確認せず要件を受け入れた。
- 留保: 絶対pathが要件作成後の実装だけで混入した可能性は、記録範囲から除外できない。
- 根拠: `.opencode/local-failure-logs/20260716-0225-unauthorized-absolute-path-acceptance.md:32-37,55-69`

### 2.5 読者に不要な識別子をreview通過

- 観測: 人間向け本文に機械消費者を持たない `source_class` と英語の正規値を置いたが、日本語成果物のreviewは定義と分類整合だけを確認して合格させた。ユーザーは `0x96BAh` を反例として示した。
- 主境界3.5: 識別子が定義済みであることを、読者に必要であることの根拠として扱い、検証方法と基準の関係を誤った。
- 副境界: `None`。ログはreview条件の不足を直接再現しているが、識別子採用までの判断成果物は分類に十分な形で残していない。
- 根拠: `.opencode/local-failure-logs/20260716-1446-japanese-review-arbitrary-identifier-pass.md:57-80,125-161`

### 2.6 review-responseの拡張経路を縮退

- 観測: task固有語を削る変更で、共通templateが三つのreview skillへ展開される接続関係まで削り、source templateだけを正本と断定した。
- 主境界3.3: task固有語を除きつつ生成経路と所有関係を保存する必要に対して、実際の変更が接続関係を失った。
- 副境界3.5: 語句不在と差分形式の成功を、接続関係を保存した根拠として扱った。
- 根拠: `.opencode/local-failure-logs/20260715-0301-review-response-expansion-path-collapsed.md:58-74,95-126`

### 2.7 日本語規則をcommand agentへ反映せず

- 観測: 平文の日本語規則が存在したが、command agentが消費するprompt経路へ反映されなかった。
- 主境界3.3: 明示された規則を、実際の消費者が読む変更へ落としていない。
- 副境界3.5: 配置先と消費経路を確認せず、規則追加を完了として扱った。
- 根拠: `.opencode/local-failure-logs/20260714-2330-ignored-plain-japanese-prompt-rule.md:45-57,68-96`

### 2.8 目標を狭い完了条件へ置換

- 観測: 平文の目標を達成していないのに、途中で作った狭いscopeを完了条件として終了した。
- 主境界3.1: 元の目標より弱い条項へ置き換え、受け入れ基準の意味を狭めた。
- 副境界3.5: 狭い条項の達成を元目標の達成として宣言した。
- 根拠: `.opencode/local-failure-logs/20260712-2225-plain-goal-fabricated-completion-by-scope-substitution.md:56-65,87-100`

### 2.9 runtime消失をworktree resetと誤認

- 観測: runtime harnessが物理的に消えていたが、worktreeだけがresetされたと解釈し、接続確認を行わなかった。
- 主境界3.4: 実環境の状態と利用可能な実行経路を観測していない。
- 副境界: `None`。誤った状態説明はユーザーが直ちに止めており、受け入れ判断まで進んだ証拠はない。
- 根拠: `.opencode/local-failure-logs/20260712-2147-runtime-reset-misread-as-worktree-reset.md:43-60,67-86`

### 2.10 task固有bootstrapを恒久promptへ昇格

- 観測: 一つの作業を開始するための局所的なbootstrapを、恒久的なprompt運用規則へ昇格した。
- 主境界3.2: 局所解を一般解として採用する実装判断を行った。
- 副境界3.3: task固有情報を恒久promptの変更へ落とした。
- 副境界3.5: 一般化可能性と寿命を検証せず、恒久規則として受け入れた。
- 根拠: `.opencode/local-failure-logs/20260712-2102-task-specific-bootstrap-promoted-to-permanent-prompts.md:71-86,118-136`

## 3. 集計

### 3.1 主境界

| 主境界 | 件数 | 事例No. |
| ------ | ---: | ------- |
| 3.1    |    3 | 1、4、8 |
| 3.2    |    2 | 2、10   |
| 3.3    |    2 | 6、7    |
| 3.4    |    2 | 3、9    |
| 3.5    |    1 | 5       |
| 合計   |   10 | 1〜10   |

全境界に少なくとも一件を割り当てられた。ただし、これは五境界を含むよう選んだ目的標本の結果であり、分類器の網羅性や母集団の分布を示さない。

### 3.2 副境界を含む出現

10件中9件で3.5の失敗を観測した。8件では前段の失敗とは別に、検証方法または受け入れ判断がその失敗を止められなかった。これは3.5が常に原因だという意味ではない。失敗ログだけを母集団にし、前段で停止できた成功事例を含めていないことによる標本偏りを含む。

## 4. 分かったこと

### 4.1 五境界は故障箇所と検出失敗を分けられる

「未認可変更」「render未確認」「review誤判定」を一つの監査失敗へまとめず、条項生成、判断、変更、実行、受け入れへ分けられた。特に、副境界3.5を別に記録すると、前段で何が壊れたかと、なぜ完了前に止まらなかったかを分離できる。

### 4.2 3.2は中間成果物がないと局所化しにくい

review cycle再開条件の過剰化とtask固有bootstrapの恒久化では、判断内容がログに残っていたため3.2へ分類できた。最終差分と失敗結果しか残っていない事例では、3.2と3.3のどちらで意味が変わったかを決められない。decision auditの有効性を示したのではなく、判断記録がない場合の観測限界を確認した。

### 4.3 P1〜P5と五境界は役割が異なる

P1〜P5は、完全伝達不能、一貫性圧、状態曖昧性など、信頼崩壊を生む制約と機序を分類する。五境界は、その機序が今回の変換連鎖のどこで保存失敗として現れたかを局所化する。同じP4系の自己申告・独立性不足でも、条項生成、実行観測、受け入れ判断の複数境界へ現れうる。したがって、五境界はP1〜P5の代替ではない。

## 5. 分からないことと停止点

- 目的標本10件から、60件全体の境界別頻度は分からない。
- ログに中間成果物がない場合、主境界より前の保存失敗は否定できない。
- 副境界3.5が多いことが、分類構造の性質か、失敗ログの選定条件によるものかは分からない。
- 五境界を使うことで、従来のP1〜P5だけの分析より診断時間や確認範囲が減るかは測定していない。
- 四つの監査台帳などの候補設計が、どの境界で費用対効果を持つかは本試行から確定しない。

本報告は既知事例の試行分類までで停止する。恒久prompt、skill、schema、hook、checkerへの反映や実タスクでのA/Bは、別の実験契約を要する。
