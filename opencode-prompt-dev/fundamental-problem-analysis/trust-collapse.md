# AI 委任における信頼崩壊と監査台帳

## 0. 本報告の位置づけ

- 本報告は、AI に複雑な作業を委任したときに起きる信頼崩壊について、前提、観察、概念、分類基準を固定する。実装案と実装計画は扱わない。
- 上位文書は `opencode-prompt-dev/fundamental-problem-map.md`（委任の4面、知識配置図、根源課題 P1〜P5、衝突の予定）とする。本報告は上位文書を変更しない。位置は「P2（検証の非対称性）へ投資する監査装置の設計に先立つ問題整理」である。
- 読者は監査装置の設計・実装・保守を行う人間とエージェント。本報告だけで、何が確定済みで何が未決かを判定できることを目標とする。
- 本報告は問題の性質（§1、§3〜§5）と、解の骨格（§2 の監査装置、§6 の4台帳）の両方を含む。解の骨格は解の一候補であり、問題の性質から一意に導かれるものではない。解決手段を選択する読者は、§2 の監査装置と §6 を候補の1つとして扱い、別経路の検討を妨げる根拠として使ってはならない。
- 用語は、本報告の内部で完結する定義を §2 に集約する。本文中で使う前に定義する。外部由来の概念（計量標準、要件トレーサビリティ）は、持ち出す直前に最小限の説明を添える。

## 1. 中心命題と信頼崩壊の機序

AI 委任の問題は、AI が誤ること自体ではない。AI がもっともらしい成果物を作る一方で、成果物の各部分が何に由来し、何に許され、何に支えられ、どこが未確認なのかが見えないため、人間が成果物全体を詳細レビューしなければ信頼できなくなることである。

中心課題は次である。

> AI が作った成果物について、人間がどこを疑えばよいかを安く知れる状態を作ること。

「安く」とは、人間による再製作に近い全数詳細レビューを要しないことを指す。

機序は次のとおり特定できた。

- 観察: 依頼していない要素の混入。複雑で制約の多いタスクで頻発し、人間の詳細レビューでのみ発覚した。代表例は、パッケージ生成タスクにおける後方互換用の未指示フィールドと、未指示の API 追加。
- 標的レビューは「欠陥は依頼した箇所に住む」という注意配分の仮定に立つ。依頼外の混入型の失敗は定義上この配分の外に住む。したがって混入が1件発覚するたびに、欠陥が1件増えるのではなく、レビュー戦略そのものが無効化され、全差分レビューへの転落を強いる。
- 信頼崩壊とは欠陥の存在ではなく、人間の注意配分の無効化である。
- したがって信頼回復の条件も定まる。AI が誤らなくなることではない。「欠陥はどこに在りうるか」への機械裏付けのある答えが復活することである。監査装置が産出すべき商品は欠陥検出ではなく、正当化された注意配分である。

計量標準の語彙で言い換える。信頼できる測定とは誤差ゼロの測定ではなく、不確かさが定量化され値付けされた測定である。全数レビューという税は、不確かさが未定量であることの価格であって、欠陥率そのものの価格ではない。

## 2. 用語と定義

本報告の核心用語を、使う前に定義する。外部由来の概念は、持ち出す直前に最小限の説明を添える。

### 成果物、変更、意図

- **成果物**: AI が作業の結果として残すもの。コード、文書、計画、レビュー回答、調査結果、設定変更、生成ファイルを含む。
- **変更**: 作業前後で観測可能に変わった成果物の部分。追加された API、変更された差分の塊、生成された依存関係のロックファイル、修正された文書段落など。
- **意図**: その作業で満たすべき目的、要求、制約、禁止事項、受け入れ条件。ユーザー発言、task contract、仕様書、既存設計、常設許可規則（後述）に分散して存在しうる。

### 対応関係と認可

- **対応関係**: ある変更が、ある意図に由来する、またはある意図を満たす、という主張。対応関係には根拠が付く。
- **認可**: ある変更が何らかの意図によって許される、または必要とされる関係。認可されていない変更は、ただちに誤りとは限らない。仕様の書き漏らし、正当な派生、単なる発明のいずれかである。

### 判定と判定の区分

- **判定**: 列挙された各変更に対し、「どの意図がこれを許すか」を機械または人間が帰属させる行為。判定の結果は、次の5区分のいずれかになる。
  - **認可済**: ある意図条項がこの変更を許す。対応関係が存在し、根拠がある。
  - **派生**: 直接の条項ではなく、許可された操作（生成器、整形器、依存追加等）の帰結として生じた変更。
  - **孤立**: 許可する条項も、派生を説明する規則も見つからない。変更が当該作業単位に由来することは確定しているが、認可する根拠がない状態。
  - **保留**: 判定が曖昧、または人間の裁定が必要。
  - **帰属不明**: 変更が当該作業単位に由来するかどうか自体が、記録された観測にも規則にも説明がつかない。

- **判定被覆**: 列挙された全変更が、上記5区分のいずれかの判定を受けた状態。すべての変更に「認可済か、派生か、孤立か、保留か、帰属不明か」の札が付いていること。判定被覆は「すべての変更が認可されている」ことを意味しない。「すべての変更が分類済みである」ことを意味する。

### 根拠と監査

- **根拠**: 主張や対応関係を支える観測可能な材料。ファイルパスと引用、コマンド出力、公式仕様、既存コード、テスト結果など。
- **監査可能性**: 成果物の正当性を限定された範囲で確認できる性質。正しいことを意味しない。どこまで確認でき、どこが未確認かが見えることを意味する。
- **計器**: 確認を安くするための仕組み。resolver、fixtures、静的検査器（lint）、hook、テスト、試験入力による動作確認など。計器は成果物ではなく、成果物を測る側に属する。

### 監査装置

- **証明書規律**: 実装者が作成時に自発的に埋めた表（証明書）の行を、resolver が照合する監査装置。対応の健全性（根拠が本物であること）を検査する。証明書の各行は、主張、出典パス、範囲、引用を持つ。
- **Stop 再照合器**: turn 終了時に変更ファイルを列挙し、保護対象内の変更に証明書があるかを再検査する装置。左全域性の検査の一部を担う。
- **判定器**: 変更の判定を行う機構または人間。LLM を使う判定器、人間の裁定、機械照合など、複数の種類がある。

### 帰属と記録

- **帰属**: ある変更を「誰が、どの作業単位で、何のために行ったか」に結びつけること。帰属は書き込み時点にしか存在せず、作業後の差分だけからは復元できない（§5.1）。
- **状態差分**: 作業前後で作業空間に生じた変更の全体。版管理システムの差分がこれに当たる。
- **実行記録**: 書き込みの都度、実行者、作業単位、ツール呼び出し、変更前後の内容要約を記録したもの。
- **残差**: 状態差分から、実行記録で説明できた変更を除いた残り。残差があれば、観測されなかった書き込みがあったことを意味する。

### 台帳と複式突合

- **台帳**: 何が観測され、何が未説明で、どこに人間の判断が必要かを分ける記録体系。本報告は4種の監査台帳を固定する。台帳は複式突合の構造を持ち、監査の対象となる。
- **複式突合**: 状態差分と実行記録を突き合わせ、説明できない変更を残差として浮かべる操作。簿記の複式簿記と同型で、帳簿の完全性は証明できないが、帳簿の不備の存在は機械的に検出できる。

### 雑音と疲労

- **SN比（信号対雑音比）**: 監査が有意義に機能している部分（認可済・派生の判定）を信号、監査が曖昧な部分（孤立・保留・帰属不明の判定）を雑音とし、信号に対する雑音の比。SN比が低いほど、監査が曖昧な部分が多く、警報疲労のリスクが高い。仕様の不完全さと観測の限界が、この比として現れる。
- **警報疲労**: SN比が低すぎた結果、人間が札を機械的に承認し始め、札が信号でなくなる状態。本報告が扱う監査装置の真の故障モードは論理破綻ではなく、この警報疲労である。
- **還流経路**: 雑音を減らすための2本の経路。1つは、複数の作業で再発する雑音のクラスを常設許可規則へ収穫すること。もう1つは、単発の書き漏らしを人間の裁定で処理し、残余コストとして計上すること。
- **常設許可規則**: 複数の作業で再発する許可のパターンを、永続的な規則として登録したもの。還流経路の1つ。import 整理、フォーマッタの実行、依存関係のロックファイルの再生成等、許可された操作の帰結として毎回発生する定型変更をここへ収穫する。規則の具体的な形式、記録方法、運用は、監査の主体と変更の単位が確定した上で新設計書が扱う。

### 検証と適合性

- **条項**: 意図を具体化した文。task contract、Requirement contract、仕様書の各項目。意図は条項として具体化されて初めて検証の基準になる。
- **検証**: ある対象が基準を満たすことを確認する行為。本報告が扱う検証は、対象と基準の組合せで2種に分かれる。これらを「検証」という一語で圧縮すると、対象の違いが見えなくなる。
  - **成果物の検証**: 成果物が条項を満たすことの確認。
  - **条項の検証**: 条項が上位の意図を満たすことの確認。
- **検証の適合性**: 検証方法が、検証の対象と基準の関係を正しく反映していること。検証が「合格」を返しても、検証方法が対象と基準の関係に合っていなければ、検証は対応を支持しない。
- **検証行為の包絡**: ある検証対象と基準の組に対して、対象と基準の関係を正しく反映する検証方法の集合。実際の検証方法が包絡の外にあれば、検証の適合性は失われる。

### 較正と完備化禁止

- **較正負債**: 計器が建てられた瞬間から持つ、正しさの劣化リスク。時間経過、環境変化、モデル世代交代で増える。較正の切れた計器は、計器なしより悪い（誤った安心を生産するため）。
- **完備化禁止**: 仕様を完全にすることを禁止する方針。P1（意図の伝達不能性）により、仕様の完全化は原理的に不可能だから。仕様の不完全さは、正しさの前提条件ではなく、SN比の低下として扱う。完備化禁止は要件の精緻化を禁じない。主張が検証可能になる粒度までの精緻化は必須である。禁止するのはその先、消費されない条項の事前列挙による完全化の追求である。精緻化の停止規則は目的適合（§5.4）が与える。

## 3. 信頼崩壊の観察例（読み飛ばし可）

読者の直感のため、観察された失敗の構造を記す。以後の議論はこの構造に依存しないが、用語はここから拾われる。

あるパッケージ生成タスクで、次が混入した。

- 依頼にない後方互換用フィールド。いかにも妥協案に見える名前で、仕様にはない。
- 依頼にない API。取得系で、呼び出し先もない。
- OS 依存のコード。Windows と Linux の境界が仕様で決まっていないまま進み、検査器と生成器の両方に OS 固有の処理が残った。

これらは「品質の低い実装」というより、依頼外の要素が成果物に紛れた事例である。問題は、その要素が不要だったかだけではない。人間が「他にも勝手な変更が混ざっているかもしれない」と考えざるを得なくなる点にある。混入が1件見つかった時点で、標的レビューは成立しなくなり、全数レビューへ戻される。

検証方法の意図への不適合も観察された。パッケージ化スクリプトを単体で実行して成功を確認したが、ビルドシステム（CMake）から呼ばれる実際の経路では検証せず、想定環境で全滅した。検証の対象（スクリプトが動くこと）と基準（パッケージ化が対象環境で動くこと）の関係を、検証方法（単体実行）が正しく反映していなかった。これは依頼にない要素の混入でも、言われたことの漏れでもなく、検証方法そのものの不適合という別の失敗クラスである（§4）。

## 4. 構造: 変更と意図の対応関係

タスクの作業は、次の二部関係として書ける。

### 基準図: すべての変更と意図に対応が付いた状態

```text
   左側: 変更要素                右側: 意図要素
   ┌──────────────┐             ┌──────────────┐
   │ 変更 A       │──対応───────│ 意図 1       │
   │ 変更 B       │──対応───────│ 意図 2       │
   │ 変更 C       │──対応───────│ 意図 3       │
   └──────────────┘             └──────────────┘
```

各対応は「この変更はこの意図に由来する」という主張で、対応には根拠が付く。タスクが正しく完了したという主張は、この関係の3つの性質に分解される。

### 三つの性質

- **対応の健全性**: 存在する各対応の根拠が本物である。引用が実在し、逐語一致し、識別子が出現する。
- **左全域性**: すべての変更要素が少なくとも1本の対応を持つ。孤立した変更がない。「余計なことをしていない」。
- **右全域性**: すべての意図要素が少なくとも1本の対応を持つ。孤立した条項がない。「言われたことをやった」。

### 全域性の違反例

```text
   左側: 変更要素                右側: 意図要素
   ┌──────────────┐             ┌──────────────┐
   │ 変更 A       │──対応───────│ 意図 1       │
   │ 変更 B       │──対応───────│ 意図 2       │
   │ 変更 C       │             │ 意図 3       │
   └──────────────┘             └──────────────┘
        ↑                              ↑
   C に対応がない              意図 3 に対応がない

   左全域性違反: C は孤立（依頼外の混入）
   右全域性違反: 意図 3 は未実現（言われたことの漏れ）
```

依頼外のフィールド混入は左全域性違反。依頼されたが実装されなかった要件は右全域性違反。

### 三性質と監査装置の対応

- **対応の健全性** を検査する装置が証明書規律である。実装者が作成時に自発的に埋めた表の行を、resolver が照合する。
- **全域性** を検査する装置が、変更要素の全数を機械が列挙し、各要素に判定を強制する監査である。
- 両者は2つの装置ではなく、同じ表を、2つの主体（実装者と機械列挙）が、2つの時点で、2つの信頼水準で埋めている。証明書規律の Stop 再照合器は、保護対象に限定し、ファイル粒度で行う左全域性検査を既に内蔵しており、両装置の境界は設計上すでに越えられている。
- 全域性の自己申告（「全部確認した」）は P4（認識の自己申告不能性）により無意味である。全域性の供給源は機械列挙以外にない。

### 検証の適合性

三性質は変更と意図の二部関係の性質である。これと直交する第4の性質がある。

- **検証の適合性**: 各対応（変更↔意図）の検証方法が、検証の対象と基準の関係を正しく反映していること。

検証の適合性の違反は、三性質のいずれの違反としても現れない。対応は存在し、根拠は本物で、左全域性も右全域性も満たすが、検証方法が対象と基準の関係を反映していない。パッケージ化スクリプトの単体実行成功は、パッケージ化が対象環境で動くことの検証行為の包絡の外にあり、適合性を欠く（§3）。

検証の適合性は、変更↔意図の二部関係の外側にあり、検証方法という第3の要素を導入する。三性質が「対応の存在と根拠」を問うのに対し、検証の適合性は「対応の検証方法の正しさ」を問う。対応の健全性（根拠が本物である）と検証の適合性（検証方法が対象と基準の関係を反映している）は別の問いである。引用が実在しても（対応の健全性）、検証方法が実際の経路を含まなければ（検証の適合性）、対応は支持されない。

### 左右の非対称

一つ残る。左全域性は分母（変更要素）が機械製で強い。右全域性は分母（意図要素）が人間製の条項集合であり、検査力は契約の粒度に律速される。この天井は、監査装置だけでは上がらない。ただし、人間が契約の粒度を細かくする（仕様の不確かさを小さくする）ことで、右全域性の検査力は間接的に上がる。

意図面の知識配置では、Known Unknown にある事項は質問や決定によって Known Known へ移せる。Unknown Known にある事項は成果物や変種の提示に対する反応から外部化できる。Unknown Unknown にある事項は直接抽出できず、現実との衝突で一部が Known Unknown へ移って初めて条項化の候補になる（§5.4 の目的適合が停止規則を与える）。

契約が粗ければ、「どれかの変更が漠然と対応している」ことで右全域性は通ってしまい、実装漏れの感度は契約の書かれ方以上には上がらない。

## 5. 技術的な境界

### 5.1 帰属は事後に復元できない

状態差分が与えるのは「作業空間で何が変わったか」であり、「この作業単位が何を変えたか」ではない。同一の最終差分は複数の履歴から生成できるため、並列エージェント、人間の修正、自動生成物が混ざった作業空間では、2枚のスナップショットから帰属を復元することは原理的にできない。

帰属は書き込み時点にしか存在せず、次の2戦略でのみ確保できる。

- **記録型**: すべての書き込み経路に観測点を置き、実行者、作業単位、ツール呼び出し、変更前後の内容要約を記録する。帰属問題は観測被覆の問題へ転位する。代償は、観測点がすべての書き込み経路をカバーし続けることの保守。
- **分離型**: 作業単位ごとに作業空間を隔離する（worktree 等）。帰属は構成で得られる。帰属問題は統合の問題（隔離外の人間編集、複数作業の併合）へ転位する。

制約として、観測点の最小化と書き込みの完全観測は同時に取れない。記録型は観測点の保守を、分離型は統合コストを払う。どちらを選ぶかは費用比較であり、人間の決定事項である。どちらを選んでも最終状態との複式突合（§5.2）は必要である。

### 5.2 完全観測は証明できないが、残差は機械計算できる

「すべての書き込みが記録された」は全称主張であり、証明書規律が全称主張を運べないのと同じ限界が観測層で再帰する。しかし要求水準を変えられる。

- 実行記録は帰属を与えるが、完全性を保証しない。
- 状態差分は、リポジトリに物質化した変更の完全性を与えるが、帰属を与えない。
- 両者の複式突合、すなわち「状態差分が実行記録の合計で説明されるか」は機械計算でき、説明されない変更が残差として浮く。

これは複式簿記と同型である。帳簿の完全性は証明できないが、帳簿の不備の存在は決定論的に検出できる。買うべきは「完全観測の証明」ではなく「観測漏れの残差化」である。

境界の明示: この監査はリポジトリに物質化した変更についてだけ述べる。リポジトリ外の副作用（外部 API 呼び出し、環境変更、リポジトリ外ファイル）は境界外であり、別の手段で扱う。

### 5.3 機械が要求する完全性は分母と判定被覆だけである

決定論層が強制してよいのは次の2つである。

- **分母の完全性**: 変更要素の列挙に漏れがない。
- **判定被覆**: 列挙された全変更が、§2 の5区分（認可済、派生、孤立、保留、帰属不明）のいずれかの判定を受けた。

意図の完全性（すべての正当な変更が条項から導出できる）は P1 により原理的に不可能であり、機械はこれを要求しない。仕様の不完全さは機構を不健全にはしない。「孤立」の意味を曖昧にする——発明か、書き漏らしか——だけである。つまり仕様の不完全さは、正しさの前提条件としてではなく、SN比の低下として再出現する。

この装置の故障モードは論理破綻ではなく警報疲労である。孤立、保留、帰属不明が頻発すれば、人間は札を機械的に承認し始め、それは信頼崩壊の再演である。雑音への還流は §2 の2経路（常設許可規則への収穫、人間の裁定による残余コスト計上）に限る。タスクごとの契約細分化で雑音を埋めてはならない。それは完備化禁止に当たる。

### 5.4 仕様の不確かさは操作的に定義できる

計量標準（GUM / VIM）から移植できるのは連鎖構造ではなく体制である。計量標準の詳細は外部資料に譲る。ここでは、本報告が借用する概念を、測定方法、運用規準、借用除外の3群に分けて述べる。

測定方法:

- **仕様の不確かさ**: 条項と候補変更の組に対し、独立で有能な解釈者の集団が合理的に下しうる認可判定のばらつき。判定分布は統計量であり、自己申告ではない。そのため、P4 の自己申告バイアスが混入しない。保留は宣言から測定量に変わる。
- **統計的評価**: 脱相関した判定者に同一の組を独立判定させ、判定の割れ率を測る。
- **構造的評価**: 仕様の構造的な徴候（適用範囲の「含まないもの」の欠落、曖昧な語彙、量化子の不在）から、不確かさを見積もる。

運用規準:

- **定義の不確かさの床**: 測定量（「認可されている」）の定義自体の不完全さは消えない。これは P1 の床の対応物であり、扱いは「言明して止まる」。
- **目的適合**: 条項の精密化は、精密化によって上がるSN比が、精密化のコストを超えるときにだけ行う。契約の粒度は理想ではなく、監査が給送する意思決定の要求精度が決める。

借用除外:

- 複数の不確かさを代数的に合成する則（二乗和平方根による合成）。認可判定は範疇的であり、誤り率は連鎖に沿ってきれいに合成されない。

### 5.5 生成物は再現性で検査する

生成物、依存関係のロックファイル、整形器の出力に意味的な条項対応を直接要求すると壊れる。閉じられる問いは「記録された生成器、入力、版、コマンドから再生成でき、内容要約が一致するか」である。生成物への手書き混入は再生成不一致として機械検出できる。生成結果の意味的正しさは人間の裁定層に残る。

## 6. 四つの監査台帳

本節は解の骨格であり、解の一候補である（§0）。

台帳は、正しさを直接保証する仕組みではなく、何が観測され、何が未説明で、どこに人間の判断が必要かを分ける記録体系である。台帳は複式突合の構造を持ち、監査の対象となる。本節は、解の候補として、次の4台帳を提示する。4台帳は §4 の性質と観測基盤に対応する。変更台帳が分母を、認可台帳が全域性を、根拠台帳が対応の健全性を、計器台帳が装置自身を扱う。

### 6.1 変更台帳

- **問い**: 何が変わったか。誰の、どの作業単位による変更か。説明できない残差は何か。
- **独立させる基準**: 観測方法が異なる（状態差分と実行記録。§5.1、§5.2）。
- **核**: 状態差分、実行記録、残差の複式突合。残差の機械計算。
- **属性**:（独立分類にしない）
  - **観測境界**: 分母が状態差分由来か実行記録由来か。リポジトリ物質化限定（§5.2）。
  - **個体化**: 変更要素の粒度と識別の安定性の兼ね合い。差分の塊は差分の表現であって安定した同一性を持たず、隣接編集、rebase、整形で壊れる。ファイル粒度は安定だが粗い。粒度選択ごとの再判定率は測定項目である。

### 6.2 認可台帳

- **問い**: その変更は何によって許されるか。要求されたことは実現されたか。
- **独立させる基準**: 失敗時の処置が異なる。
  - 孤立の変更は、削除すべき発明か、仕様へ昇格すべき派生要件か、保留かに分岐
  - 帰属不明の変更は、人間編集の混入か、別セッションの成果か、観測漏れかに分岐
  - 未実現条項（意図要素のうち対応がないもの）は、実装漏れか、粒度の粗さか、保留かに分岐する。
- **核**: 明示された意図集合と観測された変更集合の対応表。仕様の完全性を前提にしない（§5.3）。
- **判定**: §2 の5区分（認可済、派生、孤立、保留、帰属不明）のいずれかを使う。判定に至る経路として、直接（条項の直接の実現）、生成（再現性の連鎖で検査、§5.5）、外部由来（作業単位の外から持ち込まれた、人間編集や別セッション等）の区別を記録に残すことがある。これらは判定結果ではなく、判定の発生理由である。
- **SN比と警報疲労**: 孤立の曖昧性は除去できない。誤判定は発生源別（契約の書き漏らし、人間編集に由来する帰属不明、個体化による再判定）に計数する。還流は §2 の2経路に限る。

### 6.3 根拠台帳

- **問い**: その主張、対応関係は何に支えられているか。
- **独立させる基準**: 検査方法が異なる。引用の実在、逐語一致、識別子出現は機械照合であり、引用が主張を支持するかは意味判断である。
- **核**: 証明書規律の照合項目（パス実在、引用一致、識別子出現、対象範囲束縛、書式妥当性）。証明書はこの台帳の表現形式であり、最上位概念にしない。
- 根拠なき主張は誤りと断定せず、既定で拒絶し「未証明」として見える場所に置く。

### 6.4 計器台帳

- **問い**: 上の3台帳を測る装置は、今も信用できるか。
- **独立させる基準**: 保守者と寿命が異なる。成果物側の雑音と観測系側の腐敗を混同すると、「監査レポートが静かになった」が、品質改善なのか、hook が発火しなくなったのか、fixtures が古くなったのか、判別できなくなる。
- **核**: すべての計器は建てた瞬間から較正負債を持つ。hook の観測網羅率は試験入力により harness の版ごとに、判定器の誤り率は人間の裁定との突合によりモデル世代ごとに測り直す。較正の切れた計器は、計器なしより悪い。誤った安心を生産するからである。
- 検証系は、自分自身の不確かさ収支（hook 観測網羅率、判定器誤り率、帰属不明率、較正切れ計器数）を測定し続ける。その結果、検証系が他者に課す回復条件を、検証系自身についても満たせるようになる。検証系全体の信頼言明は「証明された保証」ではなく「較正された継ぎ目の連鎖」である。

記録体系を1枚の表に載せる候補形式として、4台帳を統合する記録表がある。これは4台帳を統合する表現の1つであり、形式の名前を問題の名前にしてはならない。

## 7. P1〜P5 との関係

4台帳は P1〜P5 と競合する分類ではない。P1〜P5 は失敗の根源分類（なぜ失敗するか）、4台帳は監査の記録体系（何を数えるか）であり、直交する。

| 台帳     | P1〜P5 の露出点                                                                         |
| -------- | --------------------------------------------------------------------------------------- |
| 変更台帳 | P4 の回避（帰属と被覆を自己申告に依存させない）。残差は観測欠損であって意図問題ではない |
| 認可台帳 | P1 が雑音として露出する場所（孤立の曖昧性、保留、定義の不確かさの床）                   |
| 根拠台帳 | P2 への中心投資（発見の義務を行単位の照合へ変換する）                                   |
| 計器台帳 | P3 と同型の予算問題（較正負債）。P4 の再入場（判定器も LLM。除去ではなく脱相関と較正）  |

P5（蓄積の不在）への答えは4台帳の全域にわたる。常設許可規則、fixtures、較正記録が、モデル世代を越えて持ち越す資本である。

## 8. 分類規律と肥大化の抑制

新しい概念を独立分類にするには、必須条件と補助条件の両方を満たすこと。

必須条件: 既存の分類のままだと、誤検知、見逃し、警報疲労、保守腐敗のいずれかが現実に発生すること。「異なる」だけでは、既存と違うものは大抵どこかが異なるため条件を満たしてしまう。既存の分類では説明できない事象があることが必須である。

補助条件: 必須条件に加え、次のいずれかが成り立つこと。

1. 観測方法が既存分類と異なる。
2. 失敗時の処置が既存分類と異なる。
3. 責任者または保守者が既存分類と異なる。
4. 寿命または更新頻度が既存分類と異なる。

満たさない概念は、属性、状態、対応関係の区分、不確かさ、費用源、失敗モードとして既存分類へ畳む。本報告での適用実績: 観測境界と個体化は変更台帳の属性へ、判定の発生理由は認可台帳の記録項目へ畳んだ。計器台帳だけを独立させた（必須条件の保守腐敗と、補助条件3、4 を満たす）。

この規律の背景にはメタ問題がある。信頼回復のための制度が、それ自体として複雑化し、新たな検証対象・保守対象・認知負荷になる。監査装置の設計が、監査されるべき新しい成果物になる。抑制の原則:

- 分類と装置を増やしてはならない。まず統合する。
- 統合すると事故る場合に限り分ける。
- 分けたら、分けた後の観測方法、処置、所有者、寿命を明示する。

## 9. 未決事項

### 経験的な問い

装置導入の可否を決める測定項目である。理屈で先取りしてはならない。

- SN比: 信号（認可済・派生）に対する雑音（孤立・保留・帰属不明）の比の水準と推移。警報疲労の閾値以上に保たれるか。
- 常設許可規則の成長曲線: 有限で緩やかか、発散するか。
- 残差率: 人間編集由来の頻度と、人間の裁定コスト。
- 判定器の較正安定性: モデル世代交代をまたいで保つか。
- 個体化による再判定率: 粒度選択ごとの推移。
- 4台帳の枠自体も未実測である。証明書規律と同じく、実測の悪い結果は本報告の書き換え根拠として扱う。

### 設計分岐（人間の決定待ち）

- 帰属をどう買うか: 記録型（観測被覆を保守する）か、分離型（統合コストを払う）か（§5.1）。

## 10. 最終定義

AI 委任における信頼崩壊とは、AI の成果物に対して、変更の帰属、意図による認可、主張の根拠、監査装置の健全性が見えず、人間が全数詳細レビューに戻される状態である。

信頼回復とは、AI が誤らなくなることではない。人間が次を安く判断できる状態になることである。

```text
何が変わったか。
それは誰の、どの作業単位に属するか。
それは何によって許されるか。
その主張は何に支えられているか。
どの部分が未説明・未証明・未較正か。
どの計器はまだ信用できるか。
```

以後この問題を論じるときは、まず4台帳のどれを扱っているかを明示する。どの台帳にも属さない新分類は、§8 の条件充足が示されるまで追加してはならない。
